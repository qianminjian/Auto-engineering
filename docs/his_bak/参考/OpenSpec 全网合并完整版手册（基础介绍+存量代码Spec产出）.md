# OpenSpec 全网合并完整版手册（基础介绍\+存量代码Spec产出）

# 一、OpenSpec 基础完整介绍

## 1\.1 基础定义

OpenSpec 是 **Fission\-AI（YC孵化）** 开源、MIT协议、面向AI编程助手的轻量规范驱动开发（SDD, Spec\-Driven Development）CLI框架。

核心口号：**Align before code（先对齐规格，再写代码）**

核心定位：在人类开发者与AI编码工具之间建立一份可评审、可追溯、本地存储的需求契约，解决AI聊天式开发需求模糊、上下文丢失、实现跑偏、无法审计的痛点。

开源仓库：https://github\.com/Fission\-AI/OpenSpec

官网：openspec\.cn

## 1\.2 四大核心设计哲学

1. **fluid not rigid（流动而非僵化）**：无强制阶段门禁，可随时补充、修改规范，不卡死瀑布流程；

2. **iterative not waterfall（迭代而非瀑布）**：支持增量变更，不用一次性写完完整系统规范，适配持续迭代；

3. **easy not complex（简单而非复杂）**：仅用Markdown纯文本\+目录结构，零复杂配置、无需云端API Key；

4. **brownfield\-first（存量项目优先）**：主打已有老项目迭代，不强制从零搭建新项目，兼容现有代码库。

## 1\.3 核心架构：双目录模型

所有文件全部放在项目根目录 `openspec/` 下，完全纳入Git版本管理：

```shell
openspec/
├── specs/          # 主规范库：系统已落地功能的唯一真相源
│   ├── 功能模块名/
│   │   └── spec.md # 完整需求、接口、验收规则
└── changes/        # 增量变更提案池（并行开发互不冲突）
    └── 变更名称/
        ├── proposal.md  # 需求背景、业务目标（为什么做）
        ├── specs/       # Delta增量规范：ADDED/MODIFIED/REMOVED
        ├── design.md    # 技术方案、架构选型
        └── tasks.md     # 分步实现任务清单
```

### Delta Spec（增量规范核心规则）

每次新增功能/修改逻辑只写**变更差异**，不重写全量规范：

- ADDED：新增需求

- MODIFIED：修改原有规则

- REMOVED：废弃逻辑

开发完成归档后，工具自动将增量合并到顶层 `specs/`，永久沉淀系统基线。

## 1\.4 核心能力与特性

### 1）全主流AI编码工具原生集成

内置统一斜杠命令 `/opsx:xxx`，无需额外插件；兼容Claude Code、Cursor、GitHub Copilot、Codex、OpenCode、Trae等25\+工具。

常用内置命令：

- `/opsx:propose [需求名]`：一键生成完整变更提案目录

- `/opsx:reverse [文件路径]`：逆向存量代码生成模块规范

- `/opsx:explore`：需求调研、梳理代码边界、挖掘隐性业务逻辑

- `/opsx:apply`：AI按tasks\.md自动落地代码、测试、接口

- `/opsx:archive`：完成开发，合并增量到主specs，归档变更记录

### 2）本地离线、无云端依赖

- 不依赖第三方服务、不上传需求代码、不需要API Key；

- 全部文件本地Markdown，Git天然支持版本对比、回滚、代码评审；

- 完整变更审计链：每一次功能改动都留存提案、设计、任务、需求变更记录，满足企业稽核、金融合规追溯要求。

### 3）语言无关、全链路产出

一套规范可驱动 Python/Java/Go/TS/前端 等任意技术栈；AI可自动生成：业务代码、单元测试、API文档、CI配置、数据库表设计。

### 4）与 CLAUDE\.md 互补协作

- `CLAUDE.md`：项目全局静态约束（技术栈、编码规范、禁止逻辑）；

- `openspec/specs/`：每个功能动态业务契约（该模块必须实现什么、验收标准）；

AI启动时同时读取两者，兼顾全局约束与功能细节，减少AI幻觉。

## 1\.5 标准完整工作流（四步闭环）

1. **Propose 提案**：输入一句话需求，执行命令自动生成变更文件夹，填充业务目标与增量规范；

2. **Review 对齐评审**：人和AI共同审阅文档，修正歧义、补全边界条件，达成统一认知；

3. **Apply 落地实现**：AI严格按规范逐条编码、自测、生成配套文档，杜绝随意改动；

4. **Archive 归档沉淀**：功能验收通过后归档，增量规范合并进主specs库，留存历史迭代记录。

## 1\.6 解决行业核心痛点

1. AI聊天上下文易丢失：需求不再存在对话框，永久沉淀为结构化文档；

2. AI自由发挥、频繁返工：规范作为硬性契约，AI只能按定义实现；

3. 文档与代码长期脱节：规范是代码的唯一来源，变更必须先改spec再改代码；

4. 多人协作信息断层：团队共用一套spec库，不同AI工具输出逻辑统一；

5. 老项目迭代无完整文档：渐进式增量补全系统规范，不用一次性重构文档；

6. 变更无追溯、问题难定位：每次迭代留存完整决策与需求变更记录。

## 1\.7 同类SDD工具横向对比

|工具|适用场景|复杂度|侧重点|
|---|---|---|---|
|OpenSpec|存量项目1→n迭代、个人/中小团队|轻量极简|灵活增量、无强制流程、离线本地|
|SpecKit|新项目0→1完整系统搭建|重、流程严格|强门禁、完整瀑布式规范体系|
|BMAD\-METHOD|大型企业多团队协作|复杂、自动化强|多Agent编排、重型管控|

## 1\.8 安装初始化命令（前置环境：Node\.js 20\+）

```bash
# 全局CLI安装
npm install -g @fission-ai/openspec@latest
# 项目根目录初始化，生成openspec全套目录结构
openspec init
```

## 1\.9 适用 / 不适用场景

### ✅ 适合场景

- 日常使用Cursor/Claude/Copilot做业务开发迭代；

- 存量遗留系统长期迭代、缺少完整业务文档；

- 多人团队协作，需要需求评审、变更审计追溯；

- 金融、政企等对审计、合规有要求的项目；

- 多模块、跨端复杂业务，需要统一约束AI编码行为。

### ❌ 不适用场景

- 一次性快速原型、纯探索性实验脚本；

- 超小型单文件工具，无迭代与协作需求；

- 完全无固定需求、持续推翻重构的试错项目。

## 1\.10 一句话总结

OpenSpec 是一套轻量化、离线本地、适配绝大多数AI编程助手的规范先行开发工具，用增量Markdown规范把模糊的AI聊天开发，转变成可审计、可复用、低返工的标准化工程流程，**存量遗留系统长期迭代为核心优势场景**。

# 二、存量老项目/存量代码：Spec全套产出方案

OpenSpec 原生 Brownfield\-first（存量优先），**不改动现有业务代码、不要求一次性写完全系统规范**，提供三套并行产出方案，适配所有老旧代码项目。

## 2\.1 前置：存量项目标准初始化步骤

已有老代码库，第一步必做初始化，搭建规范骨架、注入IDE命令：

```bash
# 进入存量项目根目录
cd legacy-project
# 全局安装CLI
npm install -g @fission-ai/openspec@latest
# 初始化项目，适配存量棕地模式
openspec init
```

初始化自动产出内容：

1. `openspec/config.yaml`：标记存量项目模式，录入技术栈、数据库、中间件、项目目录信息；

2. `openspec/specs/`：空白基线规范库，后续归档沉淀存量业务标准；

3. `openspec/changes/`：存放逆向、迭代、改造所有增量变更包；

4. IDE自动注入斜杠快捷命令，无需手动配置插件。

## 2\.2 方式一：Reverse 代码逆向生成（最快批量产出）

### 核心原理

扫描存量源码、接口、SQL、注释、单元测试，反向推导代码**已落地真实业务逻辑**，自动生成标准化合规spec，解决老项目代码即文档、无业务说明痛点。

### 执行命令（双端可用）

```bash
# 终端命令：逆向单个文件
openspec reverse src/legacy/user/service.js
# 终端命令：逆向完整业务模块（推荐）
openspec reverse src/legacy/order --module order
```

IDE快捷命令（Cursor/Claude直接使用）

```Plain Text
/opsx:reverse src/legacy/pay
```

### 逆向自动产出物（变更包完整结构）

路径：`openspec/changes/reverse-模块名-时间戳/`

1. **proposal\.md**：逆向目的：还原存量模块现有代码逻辑基线；

2. **specs/spec\.md（核心）**：包含业务功能、接口出入参、字段校验、数据库结构、异常分支、错误码、验收标准；

3. **design\.md**：存量架构、依赖调用、第三方对接、历史技术债务记录；

4. **tasks\.md**：AI逆向偏差核对、人工校对任务清单。

### 必备流程：校对\+归档

1. 校对：补齐无注释隐性业务逻辑，标记废弃代码为REMOVED；

2. 归档：合并至全局基线，永久生效

```bash
# 终端归档
openspec archive reverse-order-20260623
# IDE归档
/opsx:archive reverse-order-20260623
```

## 2\.3 方式二：Explore 分模块探索补全（超大耦合单体专用）

### 适配场景

几万行耦合单体项目、一次性逆向卡顿超时、代码高度耦合无法整体扫描，分批轻量化产出spec，不阻塞开发。

### 完整工作流

1. IDE执行全域代码探索：`/opsx:explore 用户模块现有实现、数据库表、全部接口与流程逻辑`

2. AI输出模块边界、依赖链路、隐性规则、代码风险报告；

3. 创建逆向专项提案：`/opsx:propose reverse-user-base`

4. AI依托探索报告生成精准spec，人工校对后归档入库；

5. 拆分用户、订单、支付、库存模块，分批完成全项目补全。

## 2\.4 方式三：迭代Delta同步补齐（长期运维主流方案）

### 核心逻辑

不用停工专门写文档，**日常改存量代码、做需求迭代时，顺带补齐对应模块spec**，零额外工作量。依托Delta三段增量规则管控变更：

- ADDED：本次迭代新增业务逻辑

- MODIFIED：修改存量原有业务规则，覆盖基线内容

- REMOVED：下线废弃接口、过期业务分支

### 存量迭代标准化流程

1. 存量改造前创建变更：`/opsx:propose fix-legacy-order-refund`

2. AI校验模块基线：无存量spec则自动同步逆向补齐基线；

3. 编写delta增量：补齐存量现状\+本次迭代改动；

4. `/opsx:apply` 修改老代码，同步联动更新规范；

5. 验收归档，增量自动合并至全局spec基线；

6. 长期迭代后，所有改动模块自动补齐全套存量规范。

## 2\.5 三类存量Spec产出方式对比选型

|产出方式|适配场景|优点|短板|
|---|---|---|---|
|openspec reverse 逆向生成|边界清晰独立模块、微服务子系统|一键生成、速度最快|无注释代码易漏隐性逻辑，需人工校对|
|/opsx:explore 分块探索|耦合严重超大单体、逆向超时卡顿|分批轻量化、不卡顿IDE|流程更长，需要多一步代码调研|
|迭代Delta同步补全|长期运维、频繁小迭代老项目|开发文档同步做，无额外工作量|需长期迭代，短期无法拿到全量规范|

## 2\.6 存量项目最终落地目录结构

```shell
openspec/
├── specs/ # 存量代码全局基线规范（唯一真相源）
│   ├── user/
│   │   └── spec.md # 用户模块存量完整基线
│   ├── order/
│   │   └── spec.md # 订单模块存量完整基线
│   └── pay/
│       └── spec.md # 支付模块存量完整基线
├── changes/
    ├── archive/ # 所有逆向、迭代变更归档（合规审计）
    └── wip/ # 待校对、待归档中变更包
```

## 2\.7 存量项目落地注意事项

1. 所有spec文件全部纳入Git管理，禁止本地私有化存放，保障团队版本统一；

2. 逆向生成spec严禁直接归档，必须人工核对隐性业务规则、历史特殊兜底逻辑；

3. 存量代码修改**必须先改delta规范，再改动代码**，杜绝文档代码脱节；

4. 废弃历史接口、死代码，统一标记REMOVED，不直接删除spec内容，留存溯源记录；

5. 存量项目优先改造业务核心模块，边缘工具类模块可延后迭代补齐，降低落地压力。
