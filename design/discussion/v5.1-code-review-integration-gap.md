# v5.1 dev-loop 中 /code-review 集成的根本性差距分析

> 创建：2026-07-05 | 来源：用户指出当前 PR 创建实现是"空想"
> 对标：CLAUDE.md §v5.1 Agent Tool 执行协议 + GSD code-review 工作流

---

## 一、设计规范 vs 当前实现的差距

### 1.1 CLAUDE.md 规定的 Stage 3

```
Stage 3 — Critic (code-reviewer Agent）
  - Spawn a code-reviewer agent via Agent tool
  - Critic output: verdict (APPROVE/MAJOR), findings (file:line + severity + issue + suggested_fix)
  - MAJOR → back to Stage 2, fix issues
  - APPROVE + all gates PASSED → success
```

### 1.2 当前 Python Engine 的实际实现

```python
# orchestrator.py _step_2e_run_agent

if in_jsonl and current_stage == "critic":
    critic_response = await self._request_critic(...)  # JSONL stdin/stdout
    # → 构造 RoundResult, 写入 state.verdict
```

**根本性差异**：

| 维度 | CLAUDE.md 规范 | 当前实现 |
|------|---------------|---------|
| 审查方式 | Spawn code-reviewer agent（完整工具访问） | JSONL 一问一答（只能看传入的 context） |
| 产出 | 结构化 findings + 深度分析 | JSON dict（verdict/findings/feedback） |
| 修复能力 | Agent 可直接 fix + commit | 只能返回文字建议，developer 重新解读 |
| 集成度 | Agent 内部闭环 | Engine 外部调用，上下文受限 |

### 1.3 GSD code-review 工作流（业界参考）

GSD 的 `/gsd-code-review --fix` 工作流：

```
1. 确定审查范围（--files / SUMMARY.md / git diff）
2. Spawn gsd-code-reviewer subagent → 读取源文件 → 产出 REVIEW.md
3. REVIEW.md 含 severity-classified findings (P0/P1/P2)
4. --fix: 委托 code-review-fix.md
   → spawn gsd-code-fixer → 逐项修复 → 原子 commit
5. --auto: 迭代循环 (capped at 3) → 修复 → 再审查 → 直至通过
```

**核心能力**：
- 审查 agent 直接读取源文件（不受 Engine 上下文限制）
- 修复 agent 直接编辑文件 + commit（闭环）
- 迭代审查（修复后重新审查验证）

---

## 二、差距根因

当前 Engine 的 critic 是通过 JSONL 协议与 agent 通信的：

```
Engine (Python) → stdout JSONL → Agent 接收 → Agent 响应 → stdin JSONL → Engine
```

这个协议天然受限：
1. Engine 只能传有限上下文（files_changed + commit_hash + test_results + gate_results）
2. Agent 的响应只是一次 JSON，不是持续交互
3. Agent 不能直接读文件（只能看 Engine 传过来的数据）
4. Agent 不能修复文件（只能给建议）

而 CLAUDE.md 规范的 code-reviewer agent 是直接 spawn 的，有完整工具访问权限。

## 三、修正方案

### 3.1 架构决策：两种模式两条路径

| 模式 | 触发方式 | critic 实现 | code-review |
|------|---------|-----------|------------|
| **Agent Tool 模式**（生产） | Agent 直接执行 dev-loop protocol | Spawn code-reviewer agent | 同 critic，合并为一步 |
| **CLI 模式**（调试） | `ae dev-loop` | JSONL 协议（受限） | 输出信号，由调用方手动执行 |

### 3.2 Agent Tool 模式完整流程

```
/ae:dev-loop "需求"
  │
  ├─ Stage 1: Architect
  │   Spawn Plan agent → batch_plan + file_list + contracts
  │
  ├─ Stage 2: Developer (Agent 自己, TDD)
  │   for each task: RED → GREEN → REFACTOR → git commit
  │   全部 tasks 完成后: 并行跑 8 Gate
  │
  ├─ Stage 3: Critic = code-reviewer Agent
  │   Spawn code-reviewer agent (subagent_type="code-reviewer")
  │   给定: files_changed + test_results + gate_results + batch_plan
  │   Agent 产出: structured findings + verdict
  │
  ├─ if MAJOR:
  │   └→ 回到 Stage 2 (Self-Refine, max 3 轮)
  │
  ├─ if APPROVE:
  │   └→ Stage 4: Deep Code Review + Auto-fix
  │       Spawn code-reviewer agent with --fix
  │       → 深度审查所有变更文件
  │       → 自动修复 P1/P2 问题
  │       → 原子 commit 每个修复
  │       → 重新审查确认
  │
  └─ Stage 5: PR + Merge
      git push + gh pr create
      (Gate 结果 + REVIEW.md 摘要注入 PR body)
      CI pass → auto-merge
```

### 3.3 commands/dev-loop.md 需要改写

当前的 `dev-loop.md` 只有一个 Bash 块调用 `ae dev-loop`。需要改为 Agent Tool 直接执行模式：

```markdown
# /ae:dev-loop — Agent Tool 直接执行模式

## Stage 1: Architect
Spawn Plan agent via Agent tool...

## Stage 2: Developer
Agent 自己执行 TDD...

## Stage 3: Critic
Spawn code-reviewer agent via Agent tool...

## Stage 4: Deep Review + Auto-fix
Run /code-review --fix...

## Stage 5: PR
Create PR + merge...
```

### 3.4 CLI 模式保持兼容

`ae dev-loop` CLI 保留为调试入口，其 JSONL critic 路径保持不变。但生产推荐使用 Agent Tool 模式。

## 四、与 atdo Gate Code Review 的关系

atdo Step 7.5 Gate Code Review 规定：每个 gate phase 完成后必须执行 `/code review`。

dev-loop 中对应：
- Stage 3 critic（code-reviewer agent）= Gate Code Review
- Stage 4 deep review + auto-fix = 补充深度审查 + 自动修复

两者不冲突，Stage 3 是 gate check（阻断性），Stage 4 是深度清理（非阻断性自动修复）。
