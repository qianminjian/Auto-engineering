## P0-1: ContractGate 全量拼接 + 子串匹配

### 问题本质

`_check_contract_in_source` 做三件事：
1. 把所有源文件读入内存拼接成一个巨大字符串（O(total_file_size) 内存）
2. 用 `str(field_name) not in source_text` 做子串匹配
3. 检查 4 项 REST API 语义（path/status_code/request/response）

**三重缺陷叠加**：

| 缺陷 | 影响 |
|------|------|
| 全量拼接 | 中型项目 ~25MB，浪费内存和 GC |
| 子串匹配 | `"id"` 匹配到 `valid`、`middle`、`grid`——假阳性 100% |
| REST API 设计 | AE 是 CLI 工具，DEFAULT_GATES 中 ContractGate 永远返回 "skip: single agent mode" |

### 生产路径实际行为

```
DEFAULT_GATES → ContractGate()  # agent_count=1, contracts=None
↓
run() → contracts is None  # 从 run_round 传入, 默认 None
↓
return "skip: single agent mode"
```

**结论：4 项检查代码在生产中从不执行。** 只在显式传入 `contracts=dict` 且 `agent_count` 被覆盖时才会触发。

### 设计方案

**方案 A：从 DEFAULT_GATES 移除 ContractGate（推荐）**

理由：
- AE 项目没有 REST API，没有跨 Agent 契约需要检查
- DEFAULT_GATES 中的 ContractGate 永远 skip，浪费 round 时间
- 代码保留在代码库中供未来多 Agent REST API 项目使用，但不加入默认 Gate 列表

影响：
- `_build_default_gates()` 移除 ContractGate 行
- DEFAULT_GATES 从 9 道变为 8 道
- 测试需更新 Gate 数量断言

**方案 B：改为通用跨 Agent 契约检查**

设计：
- 检查项从 REST API（path/status_code/request/response）改为 Agent 契约（batch_plan schema、TaskOutcome schema、Gate 输入输出格式）
- 不再全量拼接源文件，改为逐文件扫描 + AST 级别匹配

影响：大重构，不适合现在。可以作为 v6.0 功能。

**方案 C：保持现状，标记 deprecated**

不做代码变更，只在文档中说明 "ContractGate 仅适用于多 Agent REST API 项目，AE CLI 工具不触发"。

## P0-2: default_registry() 沙箱 fail-CLOSED

### 当前状态

BEACON 38 已修复——`default_registry(project_root)` 现在接受可选 project_root 参数：

```python
def default_registry(project_root: Path | None = None) -> ToolRegistry:
    for tool in [
        ReadFileTool(project_root=project_root),  # 传 project_root
        WriteFileTool(project_root=project_root),
        ...
    ]:
        registry.register(tool)
```

当 `project_root=None`（默认），工具沙箱为 fail-CLOSED——拒绝所有文件操作。这是正确的安全默认。

### 剩余问题

1. 文档/注释不准确：函数注释说"默认工具列表供开发使用"，但开发使用时 project_root=None 会导致所有工具不可用
2. 无 project_root 时错误消息不友好：`"Sandbox disabled — call site should pass project_root explicitly"`

### 设计方案

**方案 A：修正文档 + 改进错误消息（推荐）**

- 更新函数注释说明 project_root 是生产必需的
- 无 project_root 时添加更友好的错误消息："工具注册表未绑定项目目录，文件操作已禁用。调用 default_registry(project_root=Path.cwd()) 启用。"

影响：极小，仅文档和错误消息

**方案 B：默认 project_root=Path.cwd()**

风险：测试环境中 cwd 可能指向测试目录，沙箱范围不可控

## P0-3: auto_gen 空 stage 乒乓循环

### 问题本质

`_step_2c_select_tasks` 的三层 fallback：

```
Layer 1: plan.get_tasks_by_stage(current_stage)  → 精确匹配
Layer 2: self.tasks 中 role 匹配                  → role 过滤
Layer 3: list(self.tasks)                         → 无差别返回所有 tasks
```

当所有 3 层都返回空 → `advance_decision = self._router.next(...)` 推到下一 stage → `advance = True`。

在 `_tick` 中：
```python
if not round_tasks and advance:
    return None, current_stage, "advance"  # 跳过 _after_tick
```

然后 while 循环继续，round_id++，再次进 _tick。如果新 stage 也没有 tasks，再次 advance。**结果：architect → developer → critic → architect → ... 循环到 max_iter。**

### 实际触发场景

1. 用户传入空 tasks 列表 + 无 JSONL/agent_runtime（纯 CLI 调试）
2. 所有 tasks role 与当前 stage 不匹配 + plan.get_tasks_by_stage 返回空

### 设计方案

**方案 A：advance 计数器 + 上限（推荐）**

在 `_tick` 中加一个 `consecutive_advances` 计数器。连续 advance ≥ 3 次（完整一轮 architect→developer→critic 无任何 task）→ 跳出循环。

```python
consecutive_advances = 0
# in _tick:
if action == "advance":
    consecutive_advances += 1
    if consecutive_advances >= 3:
        break
    continue
consecutive_advances = 0  # 有 task 执行时重置
```

影响：
- `_tick` 返回值从 `tuple` 扩展或 Orchestrator 加实例变量
- 测试需验证连续空 stage 停止行为

**方案 B：stage_router 跟踪已访问 stages**

在 StageRouter 中维护 `visited_stages` set。当所有 3 个 stage 都被访问过且都无 task → router 返回 should_stop=True。

影响：StageRouter 需要感知 task 存在性，增加耦合

**方案 C：从源头阻止——空 tasks 时提前退出**

在 `run()` step 1 初始化时检查：如果 `self.tasks` 为空且非 JSONL 且无 agent_runtime → 立即返回空 history。

影响：最简单，但可能影响 architect 自动生成 task 的路径

## P0-4: thread pool 耗尽（sync Anthropic SDK）

### 问题本质

当前 LLM 调用链：
```
BaseAgent.execute [async]
  → asyncio.to_thread(self.llm.create_message, ...)  [thread pool]
    → AnthropicProvider.create_message [sync]
      → self._client.messages.create(**kwargs)       [httpx sync]
      → time.sleep(backoff)                           [BLOCKS thread]
```

Python 默认 ThreadPoolExecutor max_workers = `min(32, os.cpu_count() + 4)` = 12（Apple Silicon）。

**并发场景分析**：

| 场景 | 并行 agent 数 | 线程占用 | 风险 |
|------|-------------|---------|------|
| 单 Agent（当前） | 1 | 1 thread | 无 |
| 3 Agent 并行（developer tasks） | 3 | 3 threads | 无 |
| 3 Agent + retry（1 次失败） | 3 | 3 threads + sleep | 低 |
| 3 Agent + SemanticEvaluator | 4 | 4 threads | 低 |
| 未来 5+ Agent | 5+ | 5+ threads | 中 |
| 极端：全部 retry 3 次 | 3 | 3×4s sleep | 低（线程数够） |

**实际风险评估**：当前使用场景（1-3 Agent 并行 + 1 evaluator），12 个线程足够。这是**未来的架构瓶颈**，不是当前的紧急问题。

### 设计方案

**方案 A：迁移到 AsyncAnthropic（推荐，v6.0）**

```python
class AnthropicProvider:
    def __init__(self, ...):
        self._client = anthropic.AsyncAnthropic(api_key=api_key)
    
    async def create_message(self, ...) -> LLMResponse:
        response = await self._client.messages.create(**kwargs)
        ...
```

影响范围：
- `AnthropicProvider.create_message` 改为 async
- `BaseAgent.execute` 移除 `asyncio.to_thread`，直接 `await`
- `ClaudeSemanticEvaluator._call_claude` 同上
- retry 逻辑中 `time.sleep` → `asyncio.sleep`
- 所有 mock 测试需更新为 AsyncMock

**方案 B：增加线程池大小（临时缓解）**

```python
# 创建专用线程池
_executor = concurrent.futures.ThreadPoolExecutor(max_workers=20)
asyncio.to_thread(func, executor=_executor)
```

影响：治标不治本。更大的线程池只是延迟问题。

**方案 C：当前不做变更，文档记录风险**

理由：当前并发度（1-3 Agent）不会触发线程耗尽。标记为 v6.0 技术债务。
