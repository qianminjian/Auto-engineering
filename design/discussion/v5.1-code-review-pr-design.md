# /code-review (PR 创建) 设计方案与价值评估

> 创建：2026-07-05 | 来源：用户需求 + 设计文档 §PE.5 on-pr.sh 未实现
> 评估方法：对照审计角色规范的 6 维约束 + 项目定位（5-20 人团队内部使用）

| 文件名 | 触发事件 | 时机 | 用途 |

```
用户: /dev-loop "需求"

  → Developer (TDD) → 代码 + commit
  → Critic (Audit) → APPROVE / MAJOR
```

## 二、价值评估

### 2.1 对照项目定位

| 判断标准 | 分析 |
|---------|------|
| 这个功能对 5-20 人团队本地使用是否关键？ | **是** — PR 是团队协作的标准机制。AI review 之后还需要 human review。没有 PR，critic APPROVE 后直接合并，缺少人类审核环节 |
| 这个功能需要服务端/集中化？ | **否** — PR 创建走 `gh pr create` CLI，不需要服务端 |
| 这个功能需要改 Engine 为服务端？ | **否** — 纯客户端操作 |

### 2.2 对照 audit-role 6 维约束

| 维度 | 评估 |
|------|------|
| 架构合理性 | PR 创建是 dev-loop 的**自然延伸**——critic APPROVE 后，产出应该进入人类审查管道。不改变现有架构，是追加步骤 |
| 代码质量 | 不引入新复杂度。本质上是"收集上下文 → 调用 gh CLI" |
| 工程化规范 | 现有设计文档已有规划（on-pr.sh），这是**补全设计而非新增设计** |
| 团队协作友好度 | **核心价值**：AI review + human review 双层防御。PR body 含 gate 结果 + critic findings，人类审查者一眼看到质量信号 |
| 代码逻辑虚化度 | 如果不实现，on-pr.sh 继续是设计文档中的"虚幻占位"——这正是 audit 反复发现的 P0 模式 |
| TDD/生产就绪 | 当前 dev-loop 产出是本地 commit，无法直接进入团队工作流。缺少 PR 环节意味着 dev-loop 不是完整的"开发→审查→合并"闭环 |

### 2.3 结论

**有价值，且价值高。** 理由：

1. **补全设计文档已有的规划**（on-pr.sh），消除"虚幻占位"
2. **补齐 dev-loop 缺失的最后一环**：本地开发 → AI review → PR 创建 → human review → merge
3. **对 5-20 人团队是关键功能**：没有 PR，团队无法协作审查 AI 生成的代码
4. **实现成本低**：本质上是"收集上下文 + `gh pr create`"
5. **不引入架构复杂度**：不改变现有 Engine 架构，是 dev-loop 流程的追加步骤

**但是有一个限制**：如果团队不使用 GitHub（使用 GitLab/Bitbucket），`gh pr create` 不可用。设计应允许可替换的 PR 创建方式。

## 三、设计方案

### 3.1 流程变更

```
当前:
  Critic APPROVE → 输出 JSON → 用户手动 git push + gh pr create

改为（自动）:
  Critic APPROVE → 输出 JSON → 自动 push + gh pr create → 输出 PR URL
                                    ↑
                         dev-loop.md Bash 块自动执行
```

**PR 创建是 dev-loop 成功的自动后续步骤，不需要用户手动触发第二个命令。**

`/ae:code-review` 作为独立命令保留，供以下场景使用：
- 自动创建失败后的手动重试
- 想为已有 commit 创建 PR（不经过 dev-loop）
- 使用 `--draft` 标志创建草稿 PR

### 3.2 实现方式

**Plugin 层：dev-loop.md Bash 块自动执行**

`commands/dev-loop.md` 的 Bash 块在 `ae dev-loop` 返回 `status=completed` 后，自动执行：

1. 检查 `gh` CLI 可用 → `gh auth status`
2. 检查有 unpushed commits
3. 收集上下文（diff stat、commits、rounds）
4. 构造 PR body（Gate 结果 + diff 摘要 + checklist）
5. `git push -u origin <branch>`
6. `gh pr create --title ... --body ... --base main`
7. 输出 PR URL

**Plugin 层：code-review.md 独立 fallback**

`commands/code-review.md` 保留为独立 slash command，用户可手动调用。

**Plugin 层：on-pr.sh hook**

`hooks/on-pr.sh` 在 `gh pr create` 后触发，自动追加 Gate 详情到 PR body。

### 3.3 为什么是 Plugin 层不是 Engine 层

- PR 创建本质是 GitHub/Git 操作，不是 Engine 的职责
- Plugin 层已有 Bash 委托模式，符合 D1 决策（Plugin = 薄壳, Engine = 控制流）
- 实现最简单，不修改 Engine 代码
- CLI 依赖（`gh`）在 Plugin 层处理更自然

### 3.4 PR Body 模板

```markdown
## AI Dev-Loop Summary

**Status**: APPROVED by Critic
**Rounds**: {rounds}
**Duration**: {duration_sec}s

### Gate Results

| Gate | Status | Message |
|------|--------|---------|
{safety} {lint} {type_check} {test} {tdd} {coverage} {build}

### Critic Findings

{findings_table}

### Changes

```
{diff_stat}
```

### Checklist for Human Reviewer

- [ ] 核心逻辑是否正确
- [ ] 是否引入了安全风险
- [ ] 测试覆盖是否充分
- [ ] 是否与现有架构一致

---
Generated by Auto-Engineering v5.1
```

### 3.5 与 critic 的分工

| | Critic (AI) | /code-review (Human) |
|---|---|---|
| 审查维度 | 代码质量、TDD 纪律、逻辑虚化度 | 架构合理性、业务逻辑正确性、安全风险 |
| 速度 | 秒级 | 分钟-小时级 |
| 可阻断 | MAJOR → 回到 developer | Request Changes → 回到 developer |
| 输出 | findings JSON | PR review comments |

**两者互补，不是替代关系。**

## 四、风险与缓解

| 风险 | 缓解 |
|------|------|
| `gh` CLI 未安装 | `/code-review` 第一步检查 `gh auth status`，未安装时给出安装指引 |
| 非 GitHub 仓库 | 设计为可替换：支持 `GIT_HOST=gitlab` 环境变量切换 `glab mr create` |
| PR body 过长 | 截断 gate_results 中的长 message（如 traceback），保留前 200 字符 |
| 与手动 PR 冲突 | 检测已有 open PR 指向同一分支，提示用户 |

## 五、是否立即实现？

**建议：是。** 理由：

1. 设计文档已规划（on-pr.sh），当前是"虚幻占位"——按 audit 规范这是 P0
2. 实现成本极低（一个 markdown command 文件，~50 行 Bash）
3. 补齐 dev-loop 的最后一块拼图，形成完整闭环
4. 对团队使用场景是关键功能
