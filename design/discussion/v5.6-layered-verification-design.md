## 一、讨论起点：SemanticEvaluator 与 critic 职责混淆

### 1.1 原始设计

v5.5 中 critic Agent 输出含 `requirement_satisfied` 字段，用于判断"需求是否被满足"。同时存在独立组件 SemanticEvaluator 做 LLM 语义评估。

### 1.2 发现的问题

用户质疑："critic 阶段是对本次修改的代码进行审计，还是全量审计？你这段实现的范围？"

核心矛盾：critic 是高频操作（每个 batch 后都跑），如果每次 critic 都要做"全量代码需求达成分析"，阻塞不可接受。

### 1.3 结论

- **critic 只做 diff 审查**，不判断需求是否验收。这是本 diff 的质量审计（文件:行号 + 严重度 + 修复建议），秒级操作。
- **SemanticEvaluator 彻底移除**，需求验收职责转移到 verifier 层。
- critic 的提示词和 verifier 的提示词是**完全不同的两件事**——前者审 diff，后者对设计文档。

## 二、颗粒度讨论：critic vs verifier vs deep_audit

### 2.1 用户核心观点

> "我说的颗粒度是流程颗粒度层面。如果每次 critic 都要做全量代码需求达成的分析，阻塞会无法接受。"

### 2.2 三个角色在流程中的位置

| 角色 | 触发频率 | 审查范围 | 深度 | 耗时目标 |
|------|---------|---------|------|---------|
| critic | 每个 batch 后 | 本次 diff | 代码质量 | 秒级 |
| verifier | 组件完成后 / 系统退出时 | 设计→代码覆盖 | 完整性检查 | 秒级~分钟级 |
| deep_audit | 板块完成后 / 系统退出时 | 全量代码 | 质量审计 | 分钟级 |

### 2.3 关键洞察

- verifier 和 deep_audit **不是替代关系**，是互补关系。
- verifier 回答"设计文档里的每一项代码实现了吗？"（完整性，确定性匹配）
- deep_audit 回答"代码写得好不好？跨组件交互是否正确？"（质量，需要深度推理）
- 两者在不同颗粒度层面都需要：组件级需要快速 verifier，板块级需要 deep_audit，系统级两者都需要。

## 三、用户关键纠正：非黑既白的思维模式

### 3.1 用户批评（直接引用）

> "你又是我说是啥就是啥，我什么时候说过设计文档模式就不需要细化设计文档了？我给你设计文档一定所有地方都是清晰完整的吗？可能其中一部分是粗的，实现过程中，需要再设计再细化，为什么不能是通过 Architect 进行 LLM 规划设计呢？你为什么总是非黑既白，不能整合解决思路吗？"
>
> "你动不动我说啥，然后跳到左，然后跳到右，这非常不好。"

### 3.2 正确理解

- **设计文档不是二元的"清晰/模糊"**——一个大型设计文档中，部分板块/组件可能完整清晰，部分可能只是概要（粗粒度）。
- **Architect 双模式**：
  - 模式 A（无设计文档）：从模糊需求推理出完整 batch_plan
  - 模式 B（有设计文档）：解析板块→组件→模块层次，对**模糊章节**做子设计细化，对**清晰章节**直接生成 batch_plan
- **plan_refine 回路**：verifier/deep_audit 发现缺失或问题时，回到 architect 修正计划（T9/T13/T17/T19），不是回到原始需求阶段，而是在现有设计文档基础上补充细化。

### 3.3 教训

- 不把用户的观点极端化（"你说 A → 我就全做 A"）
- 整合思路而非二选一
- 设计文档中的一个板块/组件 ≈ "实现登录功能"级别的需求单元

## 四、5 层验证架构的诞生

### 4.1 用户指导

> "不同层面，有的层面要 verifier，有的颗粒度要 deep_audit。你自己设计一下哪个情况下要 deep_audit，哪些层面要 verifier，你就用我们本项目的需求为输入（5.0 开始 + 5.6 不同阶段的来分析），请深度设计后给我反馈。"

### 4.2 以本项目设计文档为输入建模

本项目的 v5.6-Design-Loop.md 可自然分解为：

```
板块 A: 架构概述 (A.0-A.3)
板块 B: 组件接口规格 (B1-B8)
  ├─ 组件 B1: EngineState 数据模型
  ├─ 组件 B2: StageRouter T1-T22
  ├─ 组件 B3: GuardrailChain
  ├─ 组件 B4: ConvergenceJudge
  ├─ 组件 B5: 7+1 Gates
  ├─ 组件 B6: 8 Agent 规格
  ├─ 组件 B7: Orchestrator 主循环
  └─ 组件 B8: Sandbox
板块 C: Tick 协议规格 (C.1-C.13)
  ├─ 组件 C.1-C.2: 问题定义 + 架构总览
  ├─ 组件 C.3: 文件桥接协议
  ├─ 组件 C.4-C.5: CLI + TickOrchestrator
  └─ 组件 C.6-C.13: 流程图/边界/实施计划
```

### 4.3 分层验证设计

```
Layer 1: critic（每个 batch 后，秒级）
  └─ 只审本次 diff，不判断需求验收

Layer 2: component_verifier（组件所有 batch 完成后，秒级）
  └─ 检查该组件的设计项是否全部实现
  └─ 输入：组件级设计项清单 + 组件代码文件列表
  └─ 输出：coverage_map[{design_item, status, file, line, note}]
  └─ 模型：Haiku（设计→代码映射是确定性匹配，不需要深度推理）

Layer 3: plate_deep_audit（板块所有组件完成后，分钟级）
  └─ 检查板块内跨组件交互质量
  └─ 输入：cross_component_contracts（如 "TickOrchestrator→StageRouter.next() 参数匹配"）
  └─ 输出：audit_findings + severity
  └─ 模型：Sonnet（跨组件交互需要深度推理）

Layer 4: system_verifier（所有板块完成后，一次性，分钟级）
  └─ 遍历全部设计项，检查全量覆盖
  └─ 输出：full_coverage_map
  └─ 模型：Haiku

Layer 5: system_deep_audit（所有板块完成后，一次性，分钟级）
  └─ 全量代码 6 维质量审计
  └─ 输入含 system_verifier 产出的 coverage_map
  └─ 模型：Sonnet
```

### 4.4 核心原理：频率×范围矩阵

```
高频 × 窄范围 → 轻量 Agent，秒级（critic, component_verifier）
低频 × 全范围 → 重量 Agent，分钟级（system_verifier, system_deep_audit）
中频 × 中范围 → 中等 Agent（plate_deep_audit）
```

## 五、已解决的讨论问题（不再重新讨论）

### 5.1 SemanticEvaluator 是否应该存在

**结论：不应该。** SemanticEvaluator 已从 v5.6 设计文档彻底移除。需求验收由 verifier 层承担。不在后续讨论中重新提议引入 SemanticEvaluator 或类似语义评估组件。

### 5.2 critic 是否应该做需求验收

**结论：不应该。** critic 职责限定为 diff 审查（代码质量）。需求验收是 verifier 的职责。critic 提示词不包含需求描述或设计文档引用。

### 5.3 verifier 和 deep_audit 是否可以合并

**结论：不可以。** 两者回答不同问题：
- verifier："设计写了的功能做了没有？"（完整性，确定性）
- deep_audit："代码写得怎么样？跨组件交互有没有问题？"（质量，需要推理）

### 5.4 是否每次 critic 后都要跑 verifier

**结论：不。** verifier 在组件所有 batch 完成后触发（T7→component_verifier），不是每个 batch 后。system_verifier 只在所有板块完成后触发一次（T12→system_verifier）。

### 5.5 Architect 是否只在无设计文档时才需要

**结论：不。** Architect 双模式：
- 无设计文档 → 推理需求生成 batch_plan
- 有设计文档 → 解析层次，对清晰部分直接生成 batch_plan，对模糊部分做子设计细化
- plan_refine 回路中回到 architect 也是在现有设计文档基础上修正/补充

### 5.6 component_verifier 应该用什么模型

**结论：Haiku。** 组件级设计→代码映射是确定性操作（设计项 X → 代码文件 Y 行号 Z），不需要深度推理。Haiku 速度快、成本低，适合高频调用。

### 5.7 验证层的触发时机

**结论（已确定，不在后续讨论中重新设计）：**

| 触发 | 条件 | 转换 |
|------|------|------|
| component_verifier | 组件所有 batch 完成 + critic APPROVE | T7 |
| plate_deep_audit | 板块所有组件完成 + component_verifier 无缺失 | T11 |
| system_verifier | 所有板块完成 + plate_deep_audit 无问题 | T12 |
| system_deep_audit | system_verifier 无缺失无偏离 | T16 |

## 六、设计原则（本次讨论确立）

### 6.1 分层原则

- **每一层有明确的频率、范围、粒度。** 不混用。
- **高频窄范围用轻量 Agent，低频全范围用重量 Agent。**
- 决不允许"为了简单"把所有验证合并到一个 Agent——那样要么阻塞高频操作，要么牺牲全量深度。

### 6.2 互补原则

- verifier 和 deep_audit 互补，不是替代。
- 先 verifier（完整性）→ 后 deep_audit（质量），前者输出作为后者输入。
- 不能因为"deep_audit 已经做了全量审计"就跳过 verifier——verifier 的覆盖映射是 deep_audit 的结构化输入。

### 6.3 非降级原则

- 设计文档与代码不一致时，默认代码缺失，不是设计过时。
- 禁止以"简化流程"为由合并验证层或降低验证标准。

### 6.4 整合思维原则

- 用户说"设计文档模式"不意味着不需要设计细化——模糊章节仍需 Architect 子设计。
- 用户说"verifier"不意味着不需要 deep_audit——不同层面需要不同验证。
- 不把用户的观点推向极端，不要非黑既白。

## 七、后续设计参考（新增设计时先对照）

1. **新增 Agent 角色时**：对照 5 层验证的频率×范围矩阵，确认新角色是否已由现有 Agent 覆盖

## 九、Pre-flight Gap Analysis + ResearchAgent 分层知识源（2026-07-08 追加）

### 9.1 讨论起点：设计文档模糊的处理时机

用户追问："设计文档模糊的情况下的处理机制，当前设计中针对设计文档中模糊的情况是如何设计的？" 并进一步提出：

> "在 loop 启动的时候，是否可以结合设计文档的实际情况，进行 architect init 的初步判断……"

**核心洞察**：v5.6 原本只有 plan_refine 回路兜底模糊——但那是在主循环**中途**（verifier/audit 阶段）才暴露缺失，此时组件可能已开发，修正代价高（推倒重来）。应在主循环**之前**做一次预检。

### 9.2 结论：Phase 0 Pre-flight Gap Analysis

在 T1 architect 之前插入 Phase 0（仅 `--design-doc` 模式）：

```
gap_scan (Agent 扫描分级) → gap_review (用户介入) → research (分层检索) → architect
```

- 分级：architectural / component / module（architectural 阻塞，不允许全 defer，G6 Guardrail 强制）
- 4 条用户路径：Fill（直接补）/ Research（检索）/ Defer（留给 architect in-loop sub-design）/ Defer+Research（检索存档，细化留 architect）
- 与 plan_refine 互补：**pre-flight 消化可预见的模糊，plan_refine 兜底开发中才暴露的缺失**。二者不替代。

### 9.3 用户关键提案：ResearchAgent 分层知识源

用户原话：

> "考虑到存量历史项目的实施，可能通过反向工程将历史代码资产化，以及文档知识库化，这样的话，可以通过本项目的 claude.md 中明确指明这些方向，避免盲目搜寻现状，当然，外部的搜索也是必要，如果项目规则中没有明确告知，就按现在的方向去学习。"

**结论：采纳，四层知识源**（Tier0 CLAUDE.md 声明 → Tier1 参考代码 → Tier2 文档 KB → Tier3 web fallback）。

**AI 补充的两个约束（用户认可后写入设计）**：

1. **资产化边界（YAGNI）**：反向工程历史代码为知识库是**项目预置的前置条件**（用户在 CLAUDE.md 声明源路径 + 借鉴点），ResearchAgent 只**消费声明**，不在 loop 运行时建库。避免把索引管线塞进 loop 导致范围爆炸。
2. **内存安全（血泪红线）**：Tier1 读参考代码必须遵守三步法（grep 定位 → 50-200 行 Read → 丢弃），禁止批量/并行扫描——这是 CLAUDE.md ⚠️ 96GB 内存爆炸事故的硬约束，ResearchAgent 不得重造事故。

**治理规则**：CLAUDE.md 声明了该领域源 → 必须先走 Tier1-2，web 仅补充；无声明 → 直接 Tier3 但记录"未找到项目源"提示用户补声明。findings 标注来源 tier 以判可信度。

### 9.4 已解决的讨论问题（不再重复）

- **模糊处理是否只靠 plan_refine**：否。plan_refine 是中途兜底，pre-flight 是启动前预检，二者互补。
- **ResearchAgent 是否盲搜 web**：否。优先项目策展源（CLAUDE.md 声明），web 是 fallback。
- **是否在 loop 运行时反向工程建知识库**：否（YAGNI）。资产化是项目预置前置，ResearchAgent 只消费声明。
- **architectural gap 能否 defer**：否。跨组件契约模糊会级联，组件设计无依据——必须 Fill/Research 先解（G6 Guardrail）。

### 9.5 后续设计参考

1. 新增知识源类型时：对照 Tier0-3 层级，确认是否已被现有层覆盖，不新增 loop 运行时建库逻辑
2. 修改 ResearchAgent 时：保留三步法内存约束 + 来源 tier 标注
3. 讨论"模糊处理"时：先确认是 pre-flight（启动前）还是 plan_refine（开发中）——两者时机、代价、决策者不同

## 十、借鉴 Superpowers 提示词技术加固 Agent 行为层（B11，2026-07-09 追加）

### 10.1 讨论起点

用户："Superpowers 这个工具链的实现是很成熟的实践，我们本产品是否可以借鉴其实现的功能，请分析一下给我反馈一下，你的思考。"

### 10.2 分析方法（避免空想）

遵守 CLAUDE.md 96GB 内存约束的三步法读 Superpowers 源，不批量扫描：README.md（哲学）+ CLAUDE.md（贡献规则，暴露 eval 哲学 + "94% PR 拒绝率"）+ writing-skills/SKILL.md（CSO / 合理化破解 / red-flags 技术）+ executing-plans/SKILL.md。全库 9 个 SKILL.md、零依赖。

### 10.3 两种架构哲学的根本差异（核心洞察）

- **Superpowers = Agent 自调节**：通过有说服力的提示词（CSO description / Iron Law / Red Flags / 合理化破解表）让 Agent 自愿遵守规范。
- **本项目 = Python 结构化门控**：Guardrail / Gate / StageRouter / ConvergenceJudge 强制 Agent 必须遵守，Agent 无法绕过。
- **关系：互补，非替代**。Superpowers 说服"应该做"，我方门控强制"必须做"。可借提示词技术加固 Agent 行为层，但绝不放弃 Python 门控作为底线。

### 10.4 借鉴项（固化到 B11）

6 项提示词行为塑形技术：CSO description 纪律（description = 触发条件，非工作流摘要）/ Iron Law / Red Flags 清单 / 合理化破解表（developer/critic/architect/verifier 四角色成品文本）/ Letter-vs-Spirit / 渐进披露。开发时直接从 B11 粘贴。

### 10.5 明确不借鉴（3 项 + 原因）——关键边界

1. **Agent 自调节执行模型** —— = v5.0 灭亡根因（Agent 可跳过 markdown 规范，无强制力）。保留 Python 门控。
2. **压力测试评估法（eval 框架）** —— 独立项目范畴，不在 Loop Engineering 范围（YAGNI）。
3. **subagent-per-task 编排** —— 与 Tick-Based 离散调用协议冲突（我方 Agent 反复调 `--tick` 驱动循环，非派生 subagent）。

### 10.6 已解决的讨论问题（不再重复）

- **Superpowers 能否整体照搬**：否——其 Agent 自调节正是 v5.0 失败根因。
- **借鉴的是提示词还是架构**：只借提示词行为塑形（B11），不借编排架构。
- **借鉴会否削弱 Python 门控**：否——提示词层加固与门控层强制互补，门控是底线，不因提示词"说服力强"而放松。

### 10.7 后续设计参考

1. 新增/修改 Agent prompt 时：从 B11 取成品文本（Iron Law / Red Flags / 合理化表），按 B11.8 映射先读对应 Superpowers 源复用。
2. 讨论"是否借鉴外部框架 X"时：先区分借的是提示词技术还是执行架构——后者需对照"是否重蹈 Agent 自调节覆辙"。

## 十一、中央提示词管理 Prompt Registry（B12，2026-07-09 追加）

### 11.1 讨论起点

用户："本项目所有环节涉及到的提示词……请深度分析设计文档和代码，当前哪些环节需要提示词、当前已设计的提示词如何。是否可以单独抽象出一个管理目录进行配置，显式主动地单独优化，运行时加载使用……先把提示词清单以表格列出，说明目的、要求和现状。"

### 11.2 清单盘点发现的核心问题：三层三版本漂移

提示词散落且版本不一致：

- `agents/prompts.py`（v5.5，architect/developer/critic 三常量，已含 Superpowers 集成）
- `.claude-plugin/commands/*.md`（v5.1，Agent Tool 直执行模式）
- `skills/auto-engineering/SKILL.md`（v5.0，7 命令，无 Tick 协议约束）

设计文档已是 v5.6，代码 prompt 落后 1-3 版本 → 严重漂移；且 6 个 v5.6 验证层 Agent 的 prompt 尚未落地。这是"集中管理"的实际驱动，非空想抽象。

### 11.3 三类提示词

- **A 类**：Agent system prompts（architect/developer/critic + 6 v5.6 验证层）
- **B 类**：共享行为片段（B11 的 Iron Law / Red Flags / 合理化表）
- **C 类**：命令 `.md`（用户可见触发入口）

### 11.4 结论：Prompt Registry（B12）

- A/B 类集中到 `prompts/`（roles/ + fragments/ + schema/），frontmatter 声明片段组合。
- init 一次性加载 + sha256 hash 锁入 checkpoint（**可复现性**——审计时能确认某次运行用的确切 prompt 版本）。
- C 类命令 `.md` **不移位**（Claude Code 命令发现机制要求固定在 `.claude-plugin/commands/`），共享片段由 `sync-prompts.py` 注入。

### 11.5 关键边界决策

- **为什么 C 类不集中**：Claude Code 命令发现机制的结构约束，`.md` 必须在固定位置——强行移位会破坏命令加载。故保留原位 + 同步机制。
- **不做**（YAGNI，团队内部 5-20 人工具）：模板引擎 / 运行时热重载 / A-B 测试框架。init 加载 + hash 锁已满足可复现需求。

### 11.6 附带修复：dev-loop.md 虚构引用

盘点中发现 `.claude-plugin/commands/dev-loop.md` 引用"BEACON 决策 47"，但 BEACON 实际最高 #44——同 2026-07-08 事故的虚构引用类（design-governance §3 / design-document-inviolability §4）。已修复为真实 #39 + 过时横幅。

### 11.7 已解决的讨论问题（不再重复）

- **提示词要不要模板引擎**：否，YAGNI。
- **运行时热重载**：否，init 一次性加载 + hash 锁，兼顾可复现。
- **C 类命令能否也集中到 prompts/**：否，Claude Code 结构约束。
- **集中管理会不会过度设计**：以"消除三层漂移 + 可复现审计"为实际驱动，非空想抽象。

### 11.8 后续设计参考

1. 新增 Agent 时：在 `prompts/roles/` 加 `.md`（frontmatter 声明 fragments），不在代码里硬编码 prompt。
2. 修改共享行为规则时：改 `prompts/fragments/`，C 类命令由 `sync-prompts.py` 重新注入，不手改各 `.md`。
3. 讨论"是否集中某资源"时：先确认有无结构约束（如 Claude Code 命令发现机制）——有则保留原位 + 同步机制，无则可集中。

## 十二、Commit → PR Review → CI/CD Pipeline 专题（B13，2026-07-09 追加）

### 12.1 讨论起点

用户："讨论一个专题：Commit → PR Review → CI/CD，当前 loop 工程设计方案中，针对这个专题的实现方式是如何的，在哪些点上做了什么。" 随后逐层追问四个问题（见 12.4-12.7）。

### 12.2 现状分析发现（问题清单）

深度分析 design + code 后发现专题散落无统一归属（PE.2 commands / PE.3 hooks / B5 Gates / B6 developer commit / 独立 code-review.md），且存在多处 v5.1 漂移与缺口：

| 缺口 | 级别 |
|------|------|
| 无 PR 触发的远程 CI（缺 ci.yml / .gitlab-ci.yml） | P0 |
| `release.yml` 含未解决 git merge 冲突（行 11-25） | P0 |
| `code-review.md` 是 v5.1 时代，前置 `VERDICT=="APPROVE"` ≠ v5.6 终态 `GOAL_ACHIEVED` | P1 |
| `code-review.md:204` 虚构引用 `v5.1-code-review-pr-design.md`（不存在） | P1 |
| `GitCommitTool` 用 `git add -A`（应精确 stage target_files） | P1 |
| commit message 格式无 Gate 强制 / 环内 test gate 默认全量偏慢 | P2 |

### 12.3 颗粒度金字塔与动作挂载（用户："这里比较复杂，我需要理解"）

产出直观设计说明：颗粒度金字塔 `task→batch→component→plate→system→PR`，6 动作挂载点（commit=task / 本地Gate=stage / AI review=batch→system 递进 / push=branch / PR=loop / merge=PR），一次 FULL dev-loop 生命周期时间轴，动作主表（时间点/颗粒度/自动人工/触发者），自动/人工「环界线」。**核心心智模型**：Loop 环内全自动到 `done`，环外由人触发推 PR 交人类 review + merge。（完整图表见设计文档 B13.1-B13.4）

### 12.4 用户追问一：PR=plate 切分是否中断自动化？

**用户敏锐指出**上一轮"plate_deep_audit 通过 = PR 切分点"的措辞会导致 loop 中途出 PR 等人 → 中断自动化。

**结论**：会中断——但**根源不是"按 plate 切分"，而是"把人工 merge 闸门插进了 loop 中间"**。核心解耦：PR「切分粒度」≠ PR「人工闸门时机」。人工闸门恒锚 `done` 之后（环界线外）。

**四方案对比**：A(一loop一PR,不中断) / B(done后延迟切分plate,不中断,中复杂度) / C(每板块停,**中断**,高复杂度) / **D(输入端控粒度,不中断,零复杂度)★推荐**。

**推荐 D**——与 §五 D13（LEAF/PLATE/FULL 自动裁剪）一脉相承：不在 loop 内造切分机制，"想要板块级 PR 就喂板块级需求"，选择权在输入端。方案 C 仅强监管场景作**显式开关**（中断是特性非缺陷）。

**修正措辞**：plate_deep_audit 是**环内质量门，不触发 PR**。

### 12.5 用户追问二：CI 是否支持 GitHub Actions + GitLab 双平台？如何实现？

**结论**：支持，且架构上零逻辑重复。**核心原则**：CI 真实逻辑收敛到平台无关单一入口（`make ci` / `ae gate-check --all`），两平台 YAML 仅为"checkout→装依赖→调入口"的薄壳（DRY，呼应两层架构——commands 是 Bash 薄壳委托 `ae`，CI YAML 是薄壳委托 `make ci`）。

- **远程 CI 跑 Gate（质量门），不跑 dev-loop**（后者需 LLM + 人在 Claude Code 触发）→ **CI 不需要 `ANTHROPIC_API_KEY`**（纯静态检查）。
- ci.yml（`on: pull_request`）+ .gitlab-ci.yml（`rules: merge_request`）骨架见 B13.6。
- coverage 在 CI 放开（本地因 16G skip）。
- **YAGNI**：只落实际使用的平台，薄壳设计"需要哪个加哪个"。

### 12.6 用户追问三：环内 Gate 与远程 CI"同一套"有必要吗？会不会很慢？

**用户质疑正确，我上轮"同一套 Gate 实现"措辞过度。**

**先澄清核心误解**：环内 Gate 与远程 CI **永不互相调用**，是独立进程（环内 Python subprocess / 远程 runner），**不存在"环内调远程"**——用户担心的"调远程变慢"不存在。

**共享分三层次**：①标准/配置(`pyproject.toml`,**必须共享**,零成本) / ②命令入口(可选) / ③运行时实现(**不该共享,有害**——会把远程重活塞进高频路径拖慢 loop)。**只共享 ① 标准，不共享 ③ 运行时。**

**频率×范围矩阵**（呼应 §四 D9/BEACON #40）：环内=**高频秒级增量快子集**（lint 改文件 + type_check + 增量 test + skip coverage）；远程=**低频分钟级全量权威**（pytest + coverage≥90% + build）。

**"绿"关系**：共享 `pyproject.toml` 同一标准 ⇒ 本地绿≈远程绿。环内**红**⇒远程**必红**；环内**绿**≠远程一定绿（远程更严）——高频挡明显问题，低频兜全量底。

**附带发现**：现设计 B5.1 环内 test gate 超时 180s，若全量则偏慢，应改增量测试。

### 12.7 已解决的讨论问题（不再重复）

- **PR=plate 是否必然中断自动化**：否。中断源于"人工闸门插进 loop 中间"，非切分本身。人工闸门恒在环外。
- **PR 颗粒度是否要在 loop 内造切分机制**：否（YAGNI）。由输入端需求层次决定（方案 D），呼应 D13。
- **环内 Gate 是否要跑与远程完全相同的运行时**：否，有害。只共享 `pyproject.toml` 标准，执行深度按频率分层。
- **CI 是否需要 ANTHROPIC_API_KEY**：否。远程 CI 跑 Gate 非 dev-loop，纯静态检查。
- **是否两个平台都要立即建 YAML**：否（YAGNI）。只落实际使用的平台。
- **merge 能否自动化**：否。红线 #6/#7，人类决策，AI 绝不触发 merge。

### 12.8 后续设计参考

1. 讨论 PR/分支策略时：先确认「切分粒度」（输入端控制）与「人工闸门时机」（恒在环外）是否被混为一谈。
2. 讨论 CI 检查时：先确认是环内（高频增量快子集）还是远程（低频全量权威）——两者共享 `pyproject.toml` 标准，不共享运行时。
3. 新增质量检查时：判定挂载颗粒度（task/stage/batch/PR），高频挂环内 Gate（保持秒级），全量挂远程 CI。
4. 修改 CI 逻辑时：只改单一入口（`make ci`/`ae gate-check`），不在各平台 YAML 里复制逻辑（DRY）。
5. 触及 merge/自动化边界时：环界线不可移——`done` 之前全自动，之后人工，AI 绝不自动 merge。

## 十三、外部 Skill/Agent 依赖审计与内化判断（2026-07-09 追加）

### 13.1 讨论起点

用户："全面分析一下当前代码实现，是否有直接调用外部 skill 或插件的情况，列出清单，确认必要性或者是否需要复制代码或组件进行内化。"

### 13.2 分析方法

全项目排查：75 个 .py + 3 命令 .md + SKILL.md + 5 hooks。扫描维度：Agent spawn（`subagent_type=`）/ 外部 CLI（`gh`/`npm`）/ MCP 调用 / 外部框架运行时引用（gsd/superpowers/claude-mem）/ 外部 skill 命令（`/code-review`）/ PyPI 依赖。

### 13.3 核心发现：唯一外部依赖集中在 dev-loop.md v5.1

**运行时对外部 skill/agent 的依赖只集中在 1 个文件：`dev-loop.md`（v5.1 遗留）**。其余要么已内化（Superpowers 技术复制进 prompts.py），要么是必要系统依赖（gh/uv/PyPI 库）。

**4 个外部 agent 依赖**（dev-loop.md v5.1）：

4. **`gsd-code-fixer`**（GSD 框架专属 agent）— **最危险**，跨框架耦合，别人环境未必装 GSD

**已正确内化的范本 — Superpowers**：`prompts.py` 中 architec/developer/critic 三 prompt 已把 Superpowers 技术复制进自有 prompt，注释仅作溯源，无运行时链接。这正是内化的标准做法。

### 13.4 必要性判断

| 外部依赖 | 必要性 | 原因 |
|---------|:------:|------|
| Plan agent | ❌ | v5.6 ArchitectAgent 自有 role 替代 |
| code-reviewer agent | ❌ | v5.6 5 层验证（critic + verifier + deep_audit）替代 |
| `/code-review --fix --auto` | ❌ | v5.6 plan_refine 回路 + developer redo 替代 auto-fix |

| `gh` CLI | ✅ 必要但需抽象 | GitHub-only，与 CI 双平台（B13.6）冲突。建议抽象为可替换后端，非内化 |
| `uv` / PyPI 库 | ✅ 必要 | 标准系统依赖，内化是反模式（成熟库优先） |

### 13.5 关键洞察

- **附带发现 = 代码漂移**（非外部依赖，不入本次范围）：`semantic_evaluator.py` 仍存在（v5.6 D6 已移除）；`cli/agent.py`+`anthropic_provider.py` 中 Python 直接调 LLM（v5.6 Tick 协议 Python 永不调 LLM）。

### 13.6 已解决的讨论问题（不再重复）

- **是否需要"内化" Claude Code 的 Plan agent**：不可行。Plan agent 是 Claude Code 内置功能，无法复制。正确做法是定义自有 architect role + prompt（v5.6 已完成）。
- **Superpowers 是否要"内化"**：已内化。复制技术到自有 prompt，无运行时依赖。
- **gh 能不能内化**：不可。服务端客户端无法内化，应抽象外层。不属"skill/agent"范畴。
- **PyPI 依赖（anthropic 等）要不要 vendor**：不。成熟库优先，内化是反模式。

### 13.7 后续设计参考

## 十四、借鉴 Superpowers 验证方法论加固审计与验证层（B15，2026-07-09）

### 14.1 讨论起点

两个连续请求合并为一条借鉴线：
1. "充分分析 test-driven-development、verification-before-completion、requesting-code-review 三个工具的借鉴吸收，从中提升本项目设计" → 产出 6 项借鉴矩阵。
2. "本机的 `/audit` 命令是否可以也借鉴分析一下" → 产出 `/audit` 三层现状 + 5 项缺口。

两组分析主题同源（Superpowers 验证方法论 + 本项目审计/验证能力），且存在重叠（RegressionGate、push-back 在两组都出现），因此**去重合并为统一借鉴清单**（B15），而非拆成两章。用户指令"固化"确认持久化。

### 14.2 三个 Superpowers 工具的方法论内核

| 工具 | Iron Law | 可门控化的确定性动作 |
|------|----------|---------------------|
| test-driven-development | NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST | RED 阶段必须先见测试失败（且失败原因正确）|
| verification-before-completion | NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE | 证据必须新鲜；回归测试 revert 后必须 FAIL |
| requesting-code-review | 审查用 crafted context（非 session history）；Critical/Important/Minor + push-back | 给 LLM 针对性上下文；发现分级 + 反驳协议 |

**核心洞察**：Superpowers 用 prompt **说服** Agent"应该验证"，本项目 B11 已注入这层说服；但真正防止 Agent 偷懒的是把三条 Iron Law 转成 **Python 门控的确定性检查**（REDGuard/FreshGate/RegressionGate）——这正是 D16"说服 vs 强制"哲学的落地。

### 14.3 `/audit` 命令三层实现的现状与缺口

- 命令入口 `audit.md`：委托通用 `/audit`（Superpowers 运行时依赖，违反 B14）。
- 确定性 Gate `audit.py`（476 行）：6 个正则 pattern，**只有语法检查无语义**。
- LLM 审计 Gate `deep_audit.py`（123 行）：**骨架**，`deep_audit.py:47` 自述"当前 Phase 1 为骨架实现，不实际 spawn agent"，真正的 LLM 审计从未运行。

**关键结论**：`/audit` 命令与 v5.6 5 层验证**不冗余、是互补**——5 层验证是环内自动闸门（tick 驱动、通过/阻塞），`/audit` 是环外人手动探索性深度分析（频率/范围/触发均不同）。AuditGate 与 system_deep_audit 需澄清分层：前者环内高频确定性、后者退出闸门低频全量（承接 D9 矩阵）。

### 14.4 合并去重后的 9 项借鉴矩阵（B15.3）

去重要点：
- 6 项 Superpowers 矩阵中的"RegressionGate"与 `/audit` 分析中的"审计规则回归测试"是**同一技术不同应用目标**（feature 代码回归 vs 审计正则自测）→ 合并为 B15 #5（一个 Gate，两处应用）。
- 6 项矩阵的"push-back 协议"与 `/audit` 的"finding 生命周期"同源 requesting-code-review → 合并为 B15 #9。

最终 9 项：DeepAuditGate 实际化(P0) / `/audit` 内化(P0) / REDGuard(P0) / FreshGate(P1) / RegressionGate(P1) / AuditGate 语义层(P1) / AuditGate↔system_deep_audit 澄清(P1) / crafted context(P2) / finding 生命周期+push-back(P2)。

### 14.5 用户关键决定

- **"固化"** → 合并两组分析为 B15 + D20 + Phase 6（T27-T31），不拆分。
- 借鉴项一律实现为 **Python 确定性门控**，不是 Agent 自觉——避免重蹈 v5.0 自调节执行灭亡覆辙。

### 14.6 不借鉴项（承接 D16）

- Agent 自调节执行（v5.0 灭亡根因）。
- 压力测试评估法（独立项目）。
- 审计发现阈值自学习（BEACON #38 提及但 YAGNI，团队内部工具不需自适应）。

### 14.7 已解决的讨论问题（不再重复）

- **`/audit` 与 5 层验证是否重复**：不重复，环内自动 vs 环外手动，互补。
- **AuditGate 与 system_deep_audit 谁保留**：都保留，频率×深度分层（AuditGate 环内高频确定性 / system_deep_audit 退出闸门低频 LLM）。
- **6 项矩阵与 /audit 5 项是否各写一章**：合并去重为一章 B15，避免 RegressionGate/push-back 重复登记。
- **借鉴要不要让 Agent 自己验证**：不。三条 Iron Law 全部转 Python 门控。

### 14.8 后续设计参考

1. 实现 REDGuard/FreshGate/RegressionGate 时挂载现有 GuardrailChain / DEFAULT_GATES，不新造框架（承接 B5）。
2. DeepAuditGate 实际化时复用 system_deep_audit 的 3-agent 编排设计，不另起炉灶。
3. `/audit` 内化（T28）与 B14 T10（dev-loop.md 移除外部依赖）是同一内化原则的两个落点。
