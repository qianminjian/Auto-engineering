## 二、架构合理性(总分 B+)

### 2.1 ✅ 优秀

| 设计 | 评价 | 依据 |
|------|------|------|
| **三层架构**(LoopEngine/StageGraph/AgentRuntime) | 职责清晰,无循环依赖 | LangGraph pregel/runtime 同构 |
| **TDD 模式执行** | 165 测试覆盖,Red→Green→Refactor 全程 | audit 已 commit Phase 0-3 |
| **Agent Protocol 抽象** | `@runtime_checkable` Protocol 兼容 mock + 真实 Agent | AutoGen register_factory 模式 |
| **工具子进程隔离** | subprocess + timeout + capture_output | 安全 + 可测试 |
| **Checkpoint 持久化** | SQLite WAL + foreign_keys + context manager | LangGraph SqliteSaver 简化版 |
| **失败检测 hook** | conftest 自动 skip 持续失败 ≥3 次 | 本会话新增(commit b68aace) |

### 2.2 ⚠️ 待改进(架构)

#### ⚠️ A1: Channel 类型系统过度简化
- **问题**:LoopState 是单一 dataclass(`engine/state.py`),所有 field 平铺
- **影响**:v2.0 多 Agent 并发需要区分 LastValue/Topic/Barrier 时,需重构整个 state
- **依据**:LangGraph channel 类型系统(参考 LangGraph `channels.py`)
- **建议**:v2.0 引入 Channel 抽象(向后兼容 — dataclass 仍可用)

#### ⚠️ A2: 没真正实现 PUSH(动态路由)
- **问题**:`_pending_sends: list = field(default_factory=list)` 在 LoopState,但 runtime.execute 没消费
- **影响**:Send dataclass 存在但 dead code,Architect 无法动态分派 Agent
- **依据**:LangGraph `_algo.py:392-468` (Send PUSH)
- **建议**:v2.0 实现 runtime.execute 消费 pending_sends(动态生成 Stage)

#### ⚠️ A3: Agent 系统 prompt 与 LangGraph/AutoGen 字段不对齐
- **问题**:system_prompt 是裸字符串,没有 role/goal/backstory/tools 字段
- **影响**:难做 system_prompt 模板化(每 Agent 写一份)
- **依据**:CrewAI `task.py:147-185` (role/goal/backstory 字段)
- **建议**:Phase 5 加 Agent dataclass 字段(role/goal/backstory/tools_config),system_prompt 从字段渲染

#### ⚠️ A4: 工具调用权限未控制
- **问题**:RunBashTool / WriteFileTool 可执行任意命令 / 写任意路径
- **影响**:dev-loop 真接 LLM 时,LLM 误调可能 rm -rf / 改 /etc/passwd
- **依据**:CrewAI 工具沙箱 + Anthropic prompt injection 防护
- **建议**:加 path 白名单(只允许项目根内)+ bash 黑名单(rm -rf / dd / mkfs / chmod 777)+ 文件写入确认

### 3.2 ⚠️ 待改进(代码)

#### ⚠️ C1: cli.py 中 TokenTracker 实际是 dead code(本会话引入的)
- **问题**:本会话 commit 08d56a0/773c384 中,BaseAgent 接受 token_tracker,cli.py 创建 TokenTracker 但**传 None 给 _execute_with_progress**(见 cli.py:323)
- **影响**:用户设 `--max-tokens 10000` 不会真生效,audit 报告曾标 P0
- **依据**:TokenTracker 设计(应有 token 累加 + 超阈值抛 BUDGET_EXCEEDED)
- **建议**:把 cli.py 中 `token_tracker=None` 改为传实际 token_tracker(dev_loop 命令创建的)

#### ⚠️ C2: BaseAgent.execute 内部 LLM 错误处理简化
- **问题**:try/except 仅捕获 ToolResult 失败,LLM 调用错误直接传播
- **影响**:LLM_TIMEOUT / NetworkError 等无分类,无法 cli 错误归类
- **依据**:LangGraph TimeoutError/NetworkError 异常体系
- **建议**:BaseAgent 捕获 anthropic.APIError / Timeout,转 AEError(LLM_TIMEOUT / LLM_NETWORK_ERROR)

#### ⚠️ C3: Tool 错误是字符串而非 AEError
- **问题**:ToolResult.error: str | None = None,不是 AEError
- **影响**:Tool 错误难以被 cli 错误归类(USER_ERROR / BUSINESS_ERROR)
- **依据**:gates/gates.py 已有 GateResult 借鉴 → AEError
- **建议**:ToolResult 可选 error_code: ErrorCode 字段

#### ⚠️ C4: 工具调用结果 type 不严格
- **问题**:`tool_input` 通过 `**tool_input` 展开,LLM 实际给非预期字段会变 kwarg 错误
- **影响**:LLM 错误参数时 BaseAgent 抛 TypeError 而不是 INVALID_AGENT_OUTPUT
- **建议**:BaseTool.execute 接 `**kwargs` 时做参数校验(根据 parameters dict)
