> 创建：2026-07-08 | 来源：用户需求 — 系统-板块-组件-模块多层次开发进度跟踪

## 一、需求澄清

### 1.1 用户需求

1. 维护 system → plate → component → module 多层次进度视图
2. 每个层次记录：设计状态、实现完成比例（task 粒度）、测试覆盖、验证状态
3. 阶段性自动展示（system_verifier / system_deep_audit 之后），不等待用户确认
4. 用户可随时通过 `ae progress` 命令查询
5. **关键约束：动态演进。** 设计文档在开发过程中会更新补充，ProgressTree 必须同步调整——这不是一次性静态快照

### 1.2 用户确认

- 进度百分比：**task 粒度**
- 层次深度：**展开到 Module 层**
- 查询命令：`ae progress [--plate X] [--depth N] [--json]`

## 二、数据模型

### 2.1 ProgressNode

```python
@dataclass
class ProgressNode:
    """层次进度节点 — 动态演进，非一次性快照"""

    # 标识
    id: str                          # "sys/B/component/B6/module/B6.1"
    name: str                        # "ArchitectAgent"
    level: str                       # "system"|"plate"|"component"|"module"
    parent_id: str | None            # "sys/B/component/B6"
    sort_order: int
    design_section_ref: str          # 对应的设计文档章节，如 "§B6.1"

    # 生命周期 (architect 管理)
    design_status: str               # "locked"|"stable"|"fuzzy"|"pending"|"removed"
    version: int = 1                 # 每次 plan_refine 修改该节点时 +1
    design_updated_at: str = ""

    # 实现进度 (developer 更新)
    total_tasks: int = 0
    done_tasks: int = 0
    current_task: str | None = None  # 当前正在执行的 task 描述

    # 验证状态 (component_verifier 更新)
    verifier_status: str = "pending" # "pending"|"pass"|"failed"|"skipped"
    verifier_missing: int = 0        # 设计项未实现的条目数
    verifier_diverged: int = 0       # 实现偏离设计的条目数

    # 质量审计 (plate_deep_audit / system_deep_audit 更新)
    deep_audit_status: str = "pending"  # "pending"|"pass"|"failed"|"skipped"
    deep_audit_p0: int = 0
    deep_audit_p1: int = 0
    deep_audit_p2: int = 0

    # 测试 (CoverageGate 更新)
    test_coverage_pct: float | None = None

    # Gate 汇总
    gate_pass_count: int = 0
    gate_run_count: int = 0

    # 时间
    created_at: str = ""
    updated_at: str = ""

    @property
    def completion_pct(self) -> float:
        if self.total_tasks == 0:
            return 0.0
        return self.done_tasks / self.total_tasks * 100

    @property
    def gate_pass_rate(self) -> float:
        if self.gate_run_count == 0:
            return 0.0
        return self.gate_pass_count / self.gate_run_count * 100
```

### 2.2 ProgressTree

```python
@dataclass
class ProgressTree:
    """系统级进度树 — 动态演进的核心数据结构"""

    system_id: str
    system_name: str
    design_doc_path: str | None      # 关联的设计文档路径
    design_doc_version: int = 1      # 设计文档变更次数，用于检测不同步
    nodes: dict[str, ProgressNode] = field(default_factory=dict)
    # 父子关系索引（加速查询）
    _children_cache: dict[str, list[str]] = field(default_factory=dict, repr=False)

    updated_at: str = ""
    last_displayed_tick: int = 0     # 上次展示时的 tick，避免重复展示

    # ─── 工厂方法 ───

    @classmethod
    def from_design_doc(cls, doc: DesignDoc) -> ProgressTree:
        """从设计文档层次结构构建初始树（Architect init 时调用）"""

    @classmethod
    def from_batch_plan(cls, batch_plan: list[dict], requirement: str) -> ProgressTree:
        """从 batch_plan 反向构建层次（无设计文档模式）"""

    # ─── 动态更新方法 ───

    def sync_from_design_doc(self, doc: DesignDoc) -> SyncResult:
        """
        与设计文档同步（plan_refine / Stage 4 doc sync 时调用）。
        返回 SyncResult 说明新增/修改/删除的节点。
        """

    def sync_from_batch_plan(self, batch_plan: list[dict]) -> SyncResult:
        """
        与 batch_plan 同步（无设计文档模式下的 plan_refine）。
        """

    def upsert_node(self, id: str, **kwargs) -> ProgressNode:
        """插入或更新节点，version 自增"""

    def mark_removed(self, id: str):
        """标记节点为 removed（不删除，保留审计轨迹）"""

    def recalculate_parents(self, node_id: str):
        """子节点变化后，向上递归重算父节点的聚合指标"""

    # ─── 查询方法 ───

    def completion_pct(self, node_id: str = "sys") -> float:
        """递归计算节点及其子树的完成百分比（按 task 加权）"""

    def children(self, node_id: str) -> list[ProgressNode]:
        """获取子节点列表（按 sort_order 排序）"""

    def find_by_design_section(self, section_ref: str) -> ProgressNode | None:
        """通过设计文档章节引用查找节点"""

    # ─── 展示方法 ───

    def display(self, max_depth: int = 3, plate_filter: str | None = None) -> str:
        """生成层次化进度表格（ASCII 文本）"""

    def summary(self) -> dict:
        """JSON 格式摘要（供机器消费）"""

    # ─── 序列化 ───

    def to_dict(self) -> dict:
        """序列化为 JSON（写入 EngineState.progress_tree_json）"""

    @classmethod
    def from_dict(cls, d: dict) -> ProgressTree:
        """从 JSON 反序列化（从 checkpoint 恢复）"""

@dataclass
class SyncResult:
    """设计文档同步结果"""
    added: list[str]        # 新增节点 ID
    modified: list[str]     # 修改节点 ID（design_status 变化、task 数变化等）
    removed: list[str]      # 标记为 removed 的节点 ID
    unchanged: list[str]    # 无变化节点 ID
    conflicts: list[str]    # 有冲突需要人工解决的节点描述
```

## 三、动态同步协议（核心设计）

### 3.1 同步触发时机

```
Timeline:
  init → architect (首次) → developer → critic → ...
    │          │
    │          └─ ProgressTree.from_design_doc()  一次性构建
    │
  ... → component_verifier → plate_deep_audit → system_verifier → ...
    │                            │
    │                            └─ [自动展示 ProgressTree]
    │
  ... → plan_refine → architect → ...
              │            │
              │            └─ ProgressTree.sync_from_design_doc()
              │               或 sync_from_batch_plan()
              │               检测差异 → 新增/修改/标记删除节点
              │
  ... → Stage 4 Doc Sync → 更新设计文档 → ...
              │
              └─ ProgressTree.sync_from_design_doc()
                  design_doc_version += 1
```

### 3.2 plan_refine 同步流程

当 verifier 或 deep_audit 发现缺口，回到 architect 时：

```
1. Architect 收到 plan_refine feedback（含 coverage_map / audit findings）
2. Architect 分析后产出新的 batch_plan / 更新的设计文档
3. TickOrchestrator 在 _after_architect() 中调用:
   a. 如果设计文档被 Architect 更新 → ProgressTree.sync_from_design_doc()
   b. 如果只有 batch_plan 更新 → ProgressTree.sync_from_batch_plan()
4. sync 方法:
   a. 遍历新层次结构，对比现有节点
   b. 新发现的设计条目 → 添加 ProgressNode(version=1)
   c. 已有的条目但 task 数变化 → 更新 total_tasks，保留 done_tasks
   d. 不再出现的条目 → 标记 design_status="removed"（不删除，保留 audit trail）
   e. 向上递归重算父节点聚合指标
5. SyncResult 写入 round_history，供后续 audit 追溯
```

### 3.3 父节点聚合规则

非叶子节点的指标由其子节点递归聚合：

```python
def recalculate_parents(self, node_id: str):
    node = self.nodes[node_id]
    children = self.children(node_id)
    if not children:
        return  # 叶子节点无需聚合

    # task 进度 = 所有子节点 task 的加权和
    node.total_tasks = sum(c.total_tasks for c in children)
    node.done_tasks = sum(c.done_tasks for c in children)

    # verifier 状态 = 任一 failed → failed; 全部 pass → pass; 其余 → pending
    statuses = {c.verifier_status for c in children}
    if "failed" in statuses:
        node.verifier_status = "failed"
    elif statuses == {"pass"} or statuses == {"pass", "skipped"}:
        node.verifier_status = "pass"
    else:
        node.verifier_status = "pending"

    # deep_audit P0/P1 = 子节点之和
    node.deep_audit_p0 = sum(c.deep_audit_p0 for c in children)
    node.deep_audit_p1 = sum(c.deep_audit_p1 for c in children)
    node.deep_audit_p2 = sum(c.deep_audit_p2 for c in children)

    # 测试覆盖率 = 子节点加权平均
    covered_children = [c for c in children if c.test_coverage_pct is not None]
    if covered_children:
        total_tasks = sum(c.total_tasks for c in covered_children)
        if total_tasks > 0:
            node.test_coverage_pct = sum(
                c.test_coverage_pct * c.total_tasks for c in covered_children
            ) / total_tasks

    # 继续向上递归
    if node.parent_id and node.parent_id in self.nodes:
        self.recalculate_parents(node.parent_id)
```

### 3.4 removed 节点处理

```
- 设计文档中不再出现的节点 → 不删除
- 标记 design_status = "removed"
- 在展示中不显示（默认过滤）
- 在 round_history 中保留 SyncResult.removed 记录
- 如果后续 plan_refine 又把该节点加回来 → design_status 恢复，version 继续累加
```

## 五、自动展示触发规则

| 触发条件 | 展示内容 | 是否阻塞 |
|---------|---------|---------|
| system_verifier 完成 | 全量进度树，depth=3 | 否，展示后继续 |
| system_deep_audit 完成 | 全量进度树，depth=3，含底部 Gate/Verifier/DeepAudit 汇总 | 否 |
| GOAL_ACHIEVED | 最终进度树，depth=3 | 否（已是终点） |
| plan_refine 触发（T9/T13/T17/T19） | 简要同步摘要："ProgressTree: +2 nodes added, 1 modified, 0 removed" | 否 |

去重：`last_displayed_tick` 防止同一 tick 重复展示。

## 九、已解决问题（2026-07-08 确认）

### Q1: Module 层的 task 归属

**决议：architect 在 batch_plan 中强制声明 `module_ref` 字段。**

task 定义增加 `module_ref`（如 `"§B6.1"`），ProgressTree 直接使用。不依赖推断——跨 module 的 task 容易出现推断错误。详见 C.3.1 architect action JSON schema 变更。

### Q2: plan_refine 中设计文档大幅修改时的节点对应

**决议：按 `design_section_ref` 做模糊匹配，匹配不上的标记 NEED_MANUAL。**

- 旧节点 `§B6.1` → 新节点 `§D.1`：名称或设计章节引用相似 → 自动迁移进度数据（done_tasks、verifier_status 等）
- 匹配不上的 removed 节点 → `SyncResult.conflicts` 中标记 `[NEED MANUAL]`，不阻塞流程
- 人工可通过 `ae progress --show-removed` 查看 removed 节点列表，手动处理

### Q3: 大 tick 数时的可读性

**决议：默认折叠已完成板块，活跃板块完整展开。**

- 已完成板块（所有组件 `completion_pct == 100` 且 `verifier_status == "pass"`）→ 折叠为一行
- 活跃板块 → 完整展开
- `ae progress --all` → 展开全部（含已完成板块）
- `ae progress --active` → 仅展示活跃板块（默认行为）
