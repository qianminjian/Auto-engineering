## 0. 一句话

把 v5.6 的**循环引擎（TickOrchestrator）**收敛为**唯一真相源**，在其"谁执行 LLM"的接缝上挂**两个驱动**：
- **Driver A（现状）**：Claude Code Agent 在 tick 之间做 LLM 工作，文件桥接。
- **Driver B（v7.0 新增）**：独立进程内 AgentRuntime **自带 key** 调 LLM 填 result，回喂同一 tick 循环。

**内部编排机制完全一致**，只换执行后端。这是 ports & adapters（六边形）模式，用来**替代** v5.5/v6 双引擎分叉，并给 v5.5 orchestrator 一个干净的退役出口。

## 1. 背景与动机

### 1.1 现状问题（两套分叉引擎的维护税）

| | v5.5 Orchestrator（`orchestrator.py`） | v6 TickOrchestrator（`tick_orchestrator.py`） |
|--|--|--|
| 控制流 | 进程内**连续** while 循环 | **离散** tick（init → tick(result) → action） |
| 谁调 LLM | **Python 进程自己调** anthropic | **Python 永不调 LLM**（BEACON #39/#40），外部 Agent 做 |
| 流水线 | architect→developer→critic + deep_audit | + component/plate/system 验证 + gap 分析 + plan_refine（更丰富） |
| 前置 | 需进程内 `ANTHROPIC_API_KEY` | Python 侧零 key |
| CLI 入口 | `ae dev-loop "需求"`（裸参数） | `ae dev-loop --init/--tick/--status/--resume` |

**税的证据**：`orchestrator.py:580-609`（T9 逻辑）为在 v6 的 DS-8 双预算 `next()` 签名迁移后仍能跑，专门加了 `10**9` 旁路 shim。每次改 gates/state/router 都要对两套路径验证 → 分叉即税。

**v5.5 唯一护城河**：脱离 Claude Code 的**独立/headless 全自动跑**（`ae dev-loop "需求"` + 自带 key）。但它的流水线停在三阶段，落后 v6。且在项目主场景（Claude Code Plugin）下**完全失效**（2026-07-04 生产反馈：子进程拿不到 agent 的 AUTH_TOKEN）。

### 1.2 双驱动如何解决

- **单引擎**：v5.6 TickOrchestrator 成为唯一流水线真相源，加新功能只加一次。
- **Driver B 跑完整 v5.6 流水线**：headless 场景不再被迫用落后的 v5.5，而是用统一引擎。
- **subsume v5.5**：v5.5 的存在理由（独立跑）由 Driver B 在统一引擎上提供 → v5.5 orchestrator 可干净退役（换薄驱动，不留 fork）。

## 2. 目标架构

```
                    ┌─────────────────────────────────────────┐
                    │  TickOrchestrator (v5.6 循环引擎, 唯一)   │
                    │  stages / gap分析 / 5层验证 / guardrail   │
                    │  / gates / convergence / plan_refine     │
                    │                                          │
                    │  核心接缝 (driver-agnostic):              │
                    │    tick_dict(result: dict) -> action: dict│  ← 纯函数核 (已存在 _after_tick)
                    └───────────────┬──────────────────────────┘
                                    │ action/result 版本化契约 (schema SSOT)
              ┌─────────────────────┴─────────────────────┐
              │                                            │
   ┌──────────▼───────────┐                    ┌───────────▼──────────────┐
   │ Driver A: AgentDriver│                    │ Driver B: StandaloneDriver│
   │ (现状)               │                    │ (v7.0 新增)               │
   │ Claude Code Agent 填 │                    │ 进程内 AgentRuntime 自带  │
   │ result, 文件桥接      │                    │ key 调 anthropic 填 result│
   └──────────────────────┘                    └───────────────────────────┘
```

### 2.1 核心接缝：action/result 契约

两驱动的**唯一耦合点**是 action JSON（引擎→驱动：该做什么）与 stage-result JSON（驱动→引擎：做完的结果）。当前这两个形状散落在代码里（`_build_action` / `_read_and_validate` / `_apply_result_to_state`），**没有 schema**。

**v7.0 前提**：把它固化为**版本化 schema SSOT**（类比 `init-manifest.schema.json`），配消费者驱动契约测试。→ 这是 **Phase 10 P0 必须任务**（当前阶段做）。

### 2.2 两驱动对比

| 维度 | Driver A（AgentDriver, 现状） | Driver B（StandaloneDriver, v7.0） |
|------|------------------------------|-----------------------------------|
| LLM 执行者 | Claude Code Agent（外部） | 进程内 `AgentRuntime`（复用 `run_round`/`BaseAgent` tool-loop/sandbox） |
| key 来源 | Claude Code OAuth 注入 | 显式 BYO `ANTHROPIC_API_KEY` |
| 触发方式 | `ae dev-loop --init/--tick` + agent | `ae dev-loop --standalone "需求"`（v7.0 新增 flag，示意） |
| 适用 | 交互式、Plugin 主场景 | headless/CI/cron、无 Claude Code 环境、引擎回归测试 |
| 执行保真度 | 高（agent 工具/上下文丰富） | 子集（进程内 runtime 能力受限） |

### 2.3 复用点（不是新造）

Driver B 的执行动作 —— `action → 调 LLM → 产出符合 schema 的 result` —— **正是 v5.5 `_step_2e_run_agent`（run_round→AgentRuntime→anthropic）已实现且测过的代码**。Driver B = 「TickOrchestrator 循环骨架」+「v5.5 的执行栈作为 tick 填充器」。

## 3. BEACON 原则精确化（扩展，非翻转）

原则 **"Python 永不调 LLM"**（BEACON #39/#40）精确化为：

> **循环引擎（TickOrchestrator）永不调 LLM；驱动（Driver）可以。**

- Driver A 遵守"进程不调 LLM"（agent 调）。
- Driver B **显式 opt-in** 打破该约束（进程内调，需 BYO key）。
- 引擎本身保持纯 Python、可确定性测试 —— 这才是原则要保护的东西（可测试、可在 Claude Code 内跑）。

这是**扩展**，不是决策翻转（#39/#40 status 不变）。已记入 BEACON 决策 #54。

## 6. 风险 / caveat

| 风险 | 缓解 |
|------|------|
| **"完全一致"仅限编排层**，执行保真度 Standalone < Agent（进程内 runtime 是 Claude Code agent 子集） | 预期设为"循环逻辑一致、执行后端不同"，V7-8 基准量化，不承诺质量对齐 |
| BYO key 成本 + sandbox 安全 | 用户付费 + 复用 v5.5 已有 sandbox（保留即可，见 T33b） |
| v5.5 退役撞决策翻转红线 | V7-7 独立成任务，退役时走 AskUserQuestion 审批（design-governance §2） |
| action/result 契约漂移 | T33a 版本化 schema + 契约测试（当前阶段前置） |

## 7. 不做（YAGNI 边界）

- **当前不实现** StandaloneDriver、不设计其 CLL flag、不加其 config —— 当前只做 Phase 10 两项 P0 预留（保接缝）。
- **不为多驱动预留插件框架/注册表** —— 两个驱动够了，不做 N 驱动抽象。
- **不动 v5.6 现有流水线逻辑** —— 双驱动是加执行适配器，不改编排。
