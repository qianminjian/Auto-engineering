## 二、依赖图与推荐执行顺序

```
Plan A (bugfixes)             ← 必须先做: D4 interrupt_after 阻塞后续设计验证
   ↓
Plan B (config-cli)           ← Settings + CLI 是 Phase 4 基础
   ↓
Plan C (templates-borrowing)  ← Stage 字段扩展,影响 Agent 输出 schema
   ↓
Plan D1 (runtime-guardrail)   ← Runtime 是 Agent 的执行环境
   ↓
Plan D2 (agent-tools)         ← 真实 LLM 调用,需要 ANTHROPIC_API_KEY
   ↓
Plan D3 (cli-observability)   ← 端到端 CLI + 进度/成本追踪
```

**为什么 A 先于 B/C**：
- A 修了 D1-D6 设计 bug（特别是 D4 interrupt_after / D5 §2.1 vs §7.4 互指），后续 phase 设计验证依赖 A 后的状态
- A 修了 B1-B9 Phase 1 隐藏 bug（特别是 B1 interrupt_after 不 break 循环），Runtime 测试依赖 B1 修复

**为什么 B 在 C 之前**：
- B 输出 Settings 完整实现，Stage 字段扩展（C）依赖 Settings 中的 `tools`/`output_schema` 等配置

**为什么 D1 → D2 → D3**：
- D1 完成 Runtime + Guardrail 后，D2 的 Agent 才能在真实 Runtime 中测试
- D2 完成 Agent 后，D3 的 CLI 才能跑端到端

## 三、并行可行性

| 组合 | 可并行？ | 原因 |
|------|:--------:|------|
| A + B | ❌ | B 引用 A 修复后的 Stage dataclass |
| A + C | ❌ | C 修改 Stage，A 修 Stage 字段 |
| A + D1 | ❌ | D1 实现 Runtime 调用 Stage，A 修了 Stage 行为 |
| B + C | ⚠️ | C 改 Stage，B 用 Stage 输出 CLI 字段——可并行但要小心冲突 |
| D1 + D2 | ❌ | D2 调 Runtime 接口，D1 先定接口 |
| D2 + D3 | ⚠️ | D3 调 Agent，D2 定 Agent 接口——理论上 D3 可在 MockAgent 下先开发 |

**推荐**：严格串行 A → B → C → D1 → D2 → D3（避免 merge 冲突 + 测试复杂）

## 十、执行命令示例

```bash
# 启动 Plan A（bugfixes）
atdo design/v1.1-PLAN-A-bugfixes.json

# Plan A 完成后启动 Plan B
atdo design/v1.1-PLAN-B-config-cli.json

# 一次性串行（用 ||; 当 Plan A 失败时停）
atdo design/v1.1-PLAN-A-bugfixes.json && \
atdo design/v1.1-PLAN-B-config-cli.json && \
atdo design/v1.1-PLAN-C-templates-borrowing.json && \
atdo design/v1.1-PLAN-D1-runtime-guardrail.json && \
atdo design/v1.1-PLAN-D2-agent-tools.json && \
atdo design/v1.1-PLAN-D3-cli-observability.json

# 跳过已通过的 Plan（用 --from N）
atdo design/v1.1-PLAN-D2-agent-tools.json --from 2

# Dry-run 验证 Plan 解析
atdo design/v1.1-PLAN-A-bugfixes.json --dry-run
```
