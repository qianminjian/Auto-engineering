# Driver A vs Driver B 保真度对比

> 创建：2026-07-15 | 来源：S-2 保真度对比任务
> 关系：验证 `v7.0-Plan-DualDriver.md` §2.2 的"执行保真度 Standalone < Agent"预期

## 1. 对比方法

三层面验证：
1. **循环逻辑层**：tick() vs tick_dict() 是否产出一致 action 序列（契约测试）
2. **验证入口层**：_read_and_validate vs _validate_result_dict 是否等价
3. **执行能力层**：进程内 AgentRuntime vs 外部 Claude Code Agent 的能力差异

## 2. 循环逻辑一致性：PASS

12 个契约测试全部通过（`tests/test_tick_orchestrator.py` S-2 段）：

| 测试类 | 验证点 | 结果 |
|--------|--------|------|
| TestValidationConsistency (5 tests) | dict 验证 vs 文件验证 — 有效/不匹配/类型错误/空batch/解析失败 | PASS |
| TestTickVsTickDictIdenticalActions (3 tests) | 同一 state + 同一 result → tick() 与 tick_dict() 产出完全一致的下一 action | PASS |
| TestDualDriverContract (4 tests) | init schema / _build_action 确定性 / _tick_process_result 共享路径 / checkpoint 一致性 | PASS |

**结论**：TickOrchestrator 的 action/result 接缝是驱动无关的。两个驱动在循环逻辑层完全一致——这是设计的预期结果（`_tick_process_result` 是公共代码路径）。

唯一的合理差异：Driver A 的 `_read_and_validate` 可返回 `RESULT_PARSE_ERROR`（文件解析失败），Driver B 的 `_validate_result_dict` 不存在该场景（dict 已在内存中）。

## 3. 执行能力差异（已知限制）

循环逻辑一致，但执行后端不同 —— 这是设计文档 §2.2 明确预期的：

| 维度 | Driver A (Claude Code Agent) | Driver B (StandaloneDriver) | 差距 |
|------|---------------------------|----------------------------|------|
| LLM 能力 | Claude Opus/Sonnet（完整能力） | 取决于 BYO key 的模型（DeepSeek 等受限） | **显著** |
| 工具生态 | Claude Code 全工具集（Bash/Read/Write/Edit/Agent/WebSearch 等） | 进程内 10 工具子集 | **中等** |
| 上下文窗口 | 完整 Claude Code 上下文管理 | AgentRuntime 单次 tool_use loop | **显著** |
| 代码质量 | 高质量（已验证） | 可变（DeepSeek 产出质量不稳定） | **显著** |
| 自主决策 | Agent 可自主判断何时读文件、何时修改 | 受限于 prompt 引导 + tool 限制 | **中等** |
| 流水线丰富度 | 10+ stage（gap_scan → system_deep_audit） | 相同（共享 TickOrchestrator） | **无** |
| Checkpoint/Resume | 支持（tick 级 restore） | 支持（同机制） | **无** |
| Gate 执行 | 真实 lint/test/type_check | 真实（同 gate_runner） | **无** |

## 4. 适用场景建议

| 场景 | 推荐驱动 | 原因 |
|------|---------|------|
| Claude Code Plugin 交互式开发 | Driver A | 利用 Claude Code Agent 完整能力 |
| headless/CI 自动化 | Driver B | 无需 Claude Code 环境，自带 key |
| 引擎回归测试 | Driver B | 纯 Python 可确定性测试，不依赖外部 Agent |
| 高质量代码生成 | Driver A | 代码质量显著更高 |
| 快速原型/ scaffold | Driver B | 速度优先，质量可接受 |
