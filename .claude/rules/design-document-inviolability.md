# 设计文档不可降级协议

> 创建：2026-07-08 | 来源：🚨 重大事故 —— 审计 AI 将 JSONL 协议标记为"已废弃"，无声推翻 v5.0 双层架构约束
> 加载方式：任何涉及设计文档修改、BEACON.md 编辑、审计任务时引用

---

## §1 设计优先原则

**设计文档与代码实现不一致时，默认判断为代码缺失实现，不是设计文档过时。**

| 场景 | 错误路径（禁止） | 正确路径 |
|------|-----------------|---------|
| 发现文档描述的功能代码中不存在 | "文档过时了，标记废弃" | 标记为 P0 代码缺口，列出缺失清单，报告用户 |
| 发现代码有但文档没写 | 不管它 | 标记为文档缺口，补文档 |
| 设计文档描述 A，代码做了 B | "A 不行了换 B" | 停，确认设计意图，报告用户决策 |

**Why:** 2026-07-08 审计 AI 发现 `_orchestrator_agent.py` 不存在（JSONL 协议未实现），选择修改 BEACON 决策 33/34 从 ✅→❌ 来"修正不一致"，导致 Python 调度层被无声消除。设计降级伪装成了文档修正。

**How to apply:** 每次审计/修改设计文档时，自问"这个修改是在让文档更准确，还是在降低设计标准？"如果是后者，立即停止，标记缺口，报告用户。

---

## §2 BEACON 决策状态翻转审批

**BEACON 任何决策的 status 列从 ✅→❌ 或 ❌→✅，触发用户审批。**

涉及 status 列的修改必须先：
1. 列出受影响的决策编号和内容
2. 说明翻转理由和影响分析
3. `AskUserQuestion` 确认
4. 确认后才能执行 Edit

**禁止**在审计修复、文档清理、Phase 推进等任务中自行翻转决策状态。

---

## §3 架构降级识别

**任何文档修改操作，如果涉及以下关键词，必须先判定是否触及架构决策：**

- "移除" / "删除" / "废弃" / "已废弃" / "deprecated"
- "降级" / "简化" / "替代" / "不再使用"
- 架构图修改、组件关系变更、通信协议变更
- "改为" / "不再" / "取而代之"

**判定流程**：
1. 修改涉及的是架构约束还是实现细节？
2. 如果是架构约束 → 停止，报告用户
3. 如果是实现细节 → 可以继续，但需在 commit message 中说明

---

## §4 虚构引用禁令

**文档中引用任何 BEACON 决策编号，必须在 `design/BEACON.md` 中真实存在。**

- 新增决策：先在 BEACON.md 决策表中追加行，再在别处引用
- 引用已有决策：用 `grep` 确认编号存在于 BEACON.md
- 禁止使用不存在的决策编号来支撑论点

---

## §5 审计方向硬约束

在 `audit-role.md` 判定原则中增加：

> **文档-代码差异处理**：审计发现设计文档与代码实现不一致时，默认方向是**代码补齐设计**。禁止以"设计已过时"为由降低设计文档标准。差异项标记为 P0 代码缺口，列出缺失实现清单，交由用户决策。**不允许**在审计修复中将设计规格降级来消除差异。
