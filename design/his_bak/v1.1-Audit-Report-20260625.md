## 三、P0 阻断性问题

### P0.1：Agent Tools 完全未连接（dev-loop 无法真运行）

**严重度：P0（阻断）**

**问题描述：**
BaseAgent.execute 使用 `self.tools`（永远空列表），而非 `task.tools`（Stage 定义的字符串工具名列表）。这导致：
1. `tool_map = {}`（空字典）
2. 所有工具调用都走 `tool not found` 分支
3. LLM 收到工具 schema 但无法执行
4. dev-loop 实际运行走 mock 数据路径

**证据链：**
```python
# agents/architect.py:45
kwargs.setdefault("tools", [])  # 永远空列表

# agents/developer.py:51
kwargs.setdefault("tools", [])  # 永远空列表

# agents/base.py:86-87
messages: list[dict] = [{"role": "user", "content": task.description}]
tool_map = {t.name: t for t in self.tools}  # t 是字符串，无 .name 属性

# runtime/runtime.py:171
task = Task(..., tools=list(stage.tools), ...)  # task.tools = list[str]

# 但 base.py execute() 从未使用 task.tools！
```

**影响：**
- `ae dev-loop "实现一个函数"` 实际无法调用任何工具
- Architect 无法读取文件了解项目现状
- Developer 无法写文件
- 运行结果等价于 ScriptedMockRuntime 的 mock 数据

**设计文档对比：**
- v1.0-Design-Loop.md §Layer 3：AgentRuntime 职责包括"工具分发"
- v1.0-Design-Loop.md §三.Agent：工具由 runtime.execute 传给 BaseAgent
- 实现与设计的根本偏差：runtime 传了 task.tools（字符串列表），但 BaseAgent 用 self.tools（空列表）

**修复建议：**
在 `AgentRuntime.execute()` 中，从 `task.tools`（字符串列表）解析为 `BaseTool` 实例：
```python
# runtime/runtime.py
from auto_engineering.tools.registry import default_registry
registry = default_registry()  # 或从参数传入

# 在 execute() 中
tool_instances = [registry.get(name) for name in task.tools if registry.get(name)]
task_with_tools = Task(...tools=tool_instances, ...)
```

或在 `BaseAgent.execute()` 中优先使用 `task.tools`：
```python
# base.py
effective_tools = task.tools if task.tools else self.tools
tool_map = {t.name: t for t in effective_tools if hasattr(t, 'name')}
```

### P0.2：SearchCodeTool 路径遍历漏洞

**严重度：P0（安全）**

**问题描述：**
`SearchCodeTool.execute()` 直接使用 `Path(path)`，没有调用 `_is_path_safe()` 检查。相比之下，`WriteFileTool` 和 `EditFileTool` 都调用了安全检查。

```python
# tools/file_tools.py:139
path = Path(kwargs.get("path", "."))  # 无安全检查
# ...
for file in path.rglob(file_pattern or "*"):  # 可遍历任意目录
```

**攻击场景：**
```
search_code(pattern="password", path="../../")  # 遍历项目外目录
```

**修复建议：**
在 `SearchCodeTool.execute()` 开头加路径安全检查（复用 `BaseTool._is_path_safe`）。

## 四、P1 重要问题

### P1.1：Agent 初始化时未注入 project_root

**问题：** `default_registry()` 创建工具实例时未传 `project_root`：
```python
# tools/registry.py:63
WriteFileTool()  # 无 project_root，白名单沙箱无效
```
**修复：** `default_registry(project_root=...)` 并在 `_build_runtime` 时传入当前 project_root。

---

### P1.2：cli.py 进度输出 tokens 永远为 0

**问题：** `_emit_stage_done(stage, elapsed, tokens=0)` 硬编码 0：
```python
# cli.py:209
_emit_stage_done(stage, elapsed, tokens=0)  # TokenTracker 累加了但未传入
```
**修复：** TokenTracker 累加后，从 `engine/loop.py` 的 `on_stage_end` 回调传回 cli。

---

### P1.3：AEError 被 bash_tools 转换为 ToolResult

**问题：** `bash_tools.py` 的 `except Exception` 捕获所有异常包括 `AEError`：
```python
except Exception as exc:
    return ToolResult(success=False, content="", error=str(exc))
```
**影响：** `AEError`（业务错误，如沙箱拦截）被当作工具失败处理，丢失错误分类。
**修复：** 明确捕获 `AEError` 并在 `ToolResult.error_code` 中传递。

---

### P1.4：init A1 增量模式实现与设计不符

**问题：** 设计文档（v1.0-Design-Init.md §1.3.10）要求"增量 MERGE 逐文件合并"，实际 `_phase_merge` 实现是"已存在则跳过"：
```python
# scaffold.py:177
if dst_file.exists():
    skipped.append(dst_file)  # 完全跳过，不做内容合并
```
**影响：** 模板更新时存量项目的已有文件无法合并新字段（如新增字段）。
**修复：** 实现逐字段合并，而非整文件跳过。

---

### P1.5：RunTestsTool._detect_runner 默认 pytest

**问题：** 无 cwd 时默认 pytest，JS 项目会误用：
```python
# test_tools.py:88
return "pytest"  # fallback
```
**修复：** 在 `cwd` 为 None 时返回错误而非 fallback，或在 Agent 初始化时强制指定 runner。

---

### P1.6：cli.py dev-loop dry_run 参数未使用

**问题：** `_run_loop_engine` 签名有 `dry_run: bool` 但代码未使用：
```python
# cli.py:282
dry_run: bool,  # 未在任何地方使用
```
**修复：** 实现 dry_run 逻辑（只跑 architect 后立即返回）。

## 五、P2 建议问题

### P2.1：CheckpointStore 无清理机制

`.ae-checkpoints/` 下 `.db` 文件只增不清，`list_all()` 返回全部历史无过期删除。

### P2.2：ListDirTool 无路径安全检查

`ListDirTool` 直接遍历 `Path(path)`，没有调用 `_is_path_safe`。

### P2.3：Agent.system_prompt 无法动态更新

system_prompt 是硬编码字符串，project_root 等动态信息无法注入。

### P2.4：RunTestsTool._detect_runner 依赖 cwd 检测

若 cwd 参数错误传入，会误判 runner。

### P2.5：3 Agent tools 列表与 system_prompt 不完全匹配

Developer system prompt 说 `edit_file`，但 graph 定义是 `edit_file`（一致）。Architect system prompt 列举工具与 Stage.tools 一致（但因 P0.1 都无效）。

### P2.6：max_tool_calls 全用默认值

Architect 只读场景用 max_tool_calls=10 浪费 LLM 调用配额。

## 七、总结

**P0 阻断：2 项**
- P0.1：Agent Tools 完全未连接（dev-loop 无法真运行，最严重）
- P0.2：SearchCodeTool 路径遍历漏洞（安全）

**P1 重要：6 项**
- P1.1：project_root 未注入（沙箱在 registry 层无效）
- P1.2：tokens 显示永远为 0
- P1.3：AEError 被 ToolResult 吞没
- P1.4：init A1 增量模式设计不符
- P1.5：RunTestsTool 默认值不安全
- P1.6：dry_run 参数未实现

**P2 建议：6 项**

**核心问题根因：** Phase 0 的实现"形式上完整，实质上断裂"：
- 工具 schema 传给 LLM ✓
- 但工具未注册到 Agent（task.tools vs self.tools 断裂）
- 结果：LLM 知道有什么工具，但无法调用 → 走 mock 数据路径

**v1.1 待办清单另见 `v1.1-Plan-Dev.md` §一（P0/P1/P2 分级表）。**
