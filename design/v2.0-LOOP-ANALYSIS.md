## 一、当前 LOOP 设计诊断（v1.0-LOOP.md）

### 1.1 架构骨架正确但维度单一

当前设计的核心是 `DevLoop` 的 `while True` 循环内嵌三个 Stage（Architect → Developer → Critic）。这个结构解决的是「一条需求 → 一套代码」的单轨场景，能工作，但缺少一个维度：「多需求并行时怎么办」。

```
当前设计（单轨）：
  ae dev-loop "实现用户登录"

  ARCHITECT → DEVELOPER → CRITIC → END
                ↑            │
                └── MAJOR ───┘

缺失维度（多轨）：
  需求池: [登录, 商品列表, 支付流程, ...]
    │
    ├─ 谁来拆分需求？
    ├─ 哪些可以并行？
    ├─ 如何保证多轨产出可集成？
    └─ 如何判定"整个项目"到了什么状态？
```

这不是单 Agent loop 的 bug——单轨是正确的基础设施。问题是它被当作了**唯一的执行模型**。

### 1.2 engine / crew / runtime 三层职责重合

| 层 | 来源 | 核心机制 | 当前设计状态 |
|------|------|---------|------------|
| `engine/` | LangGraph | while True + tick() + StateGraph + checkpoint | stub |
| `crew/` | CrewAI | Task + SequentialProcess + Crew.kickoff() | stub |
| `runtime/` | AutoGen | asyncio.Queue + message dispatch + agent registry | stub |

三个层各自包含一个"主循环"概念：

- `engine/loop.py` — `DevLoop.tick()` while True
- `crew/process.py` — `SequentialProcess.execute()` for loop
- `runtime/runtime.py` — `AgentRuntime.run_loop()` while True

**问题**：三个循环谁是驱动者？它们什么关系？

v1.0 设计没有回答这个问题。从 SHARED.md 的代码片段推断，应该是 `DevLoop.run()` 在最外层，内部调用 `SequentialProcess.execute()`，后者通过 `AgentRuntime` 来 dispatch。但这个三层嵌套的调用链在单 Agent 场景下是过度抽象的。

### 1.3 其他具体问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | `DevLoopState` vs `Checkpoint` 边界模糊 | state.py 和 checkpoint.py 各管什么不清楚 |
| 2 | Gate 系统只有 4 个闸门，且是硬编码的 | 历史项目的 7 道门模型更完整 |
| 3 | 缺少收敛判定机制 | 没有 hard limit / quality threshold / stagnation detection |
| 4 | Agent 的 LLM 调用循环在 BaseAgent 内部 | 缺少与 Loop 层的状态同步（Agent 在工具调用时 Loop 不知道） |
| 5 | `develop --multi` 是未来的 flag | 但架构没有预留多轨并行的接口 |
| 6 | 没有 Round 或迭代范围概念 | 一次 `ae dev-loop` 就跑到底，中间没有人工确认点 |
| 7 | `agents/llm.py` 文件不存在 | 设计 §五 定义了 LLMProvider 但文件未创建 |

## 二、三个框架的模式评估

### 2.1 LangGraph — 状态 Channel 系统是最强资产

LangGraph 的核心竞争力不是"图执行"，而是 **Channel 系统**——每个状态 key 是一个 typed channel：

| Channel 类型 | 语义 | LOOP 中可应用场景 |
|-------------|------|-----------------|
| `LastValue` | 单写者，最后写入覆盖 | Plan 状态、Review 结论 |
| `Topic` (accumulate=True) | 多写者，追加 | 任务队列、发现列表 |
| `Topic` (accumulate=False) | 单步广播 | Round 内通知 |
| `BinaryOperatorAggregate` | 多写者，汇聚 | 覆盖率累计、token 用量汇总 |
| `EphemeralValue` | 一次消费 | 路由指令、Signal |
| `NamedBarrierValue` | 等待全部到达 | 多 Agent 完成同步 |

**应该借用的**：
- StateGraph 的 node/edge/conditional_edge + compile 模式
- Channel 系统作为共享状态的基础抽象
- 版本触发的任务调度（channel version 变化 → 触发对应 node）
- DeltaChannel 用于高频状态（对 Agent 的 step 级事件日志）

**不应该借用的**：
- `PregelLoop` 的 `tick()` 过于通用，假设任何 node 都可能并发执行。LOOP 场景的任务有明确的前置依赖和角色分配，应该用更简单的 DAG 拓扑排序。
- `Runnable` 封装层——我们不需要 LangChain 的 Runnable 协议

### 2.2 CrewAI — Task 模型 + Flow 系统

**应该借用的**：
- `Task` 作为一等模型（description / expected_output / agent / context / guardrails / structured output）
- `Flow` 的 `@start` / `@listen` / `@router` 装饰器模式——比 LangGraph 的 `add_node` / `add_edge` 更声明式
- `Guardrail` 机制：输出验证失败 → 重新执行（不是简单报错）
- `ExecutionContext` + ContextVar 快照用于 checkpoint/resume
- Pydantic 验证器链：构造时校验，而不是运行时才发现配置错误

**不应该借用的**：
- `Crew.kickoff()` 的单一入口——我们需要 Round 循环
- 字符串拼接的 context passing（`"\n\n----------\n\n".join(...)`）——应该用结构化 context
- `Process.sequential` / `Process.hierarchical` 的二分——LOOP 需要的模式更多样（并行、条件、Fan-out/Fan-in）

### 2.3 AutoGen — Pub/Sub 消息路由 + 干预机制

**应该借用的**：
- `TopicId` + `Subscription` 模式：Agent 声明订阅的 topic，运行时负责路由
- `InterventionHandler` 作为 Gate 机制：`on_send` / `on_publish` / `on_response` 三个切入点
- `@rpc` / `@event` 装饰器 + 类型驱动路由：代理方法自动路由到消息类型
- `CancellationToken` 链接 Future 生命周期
- `AgentRuntime` Protocol 解耦实现

**不应该借用的**：
- `SingleThreadedAgentRuntime` 的单一队列——多 Agent 并发应该在 OS 进程级隔离（Worktree），不需要进程内消息队列
- `asyncio.Queue` 作为唯一通信通道——LOOP 场景中 Agent 在独立 Worktree 中运行，文件系统是更合适的通信通道

## 三、历史项目五层架构的关键概念

历史项目 `project-engineering-init` 定义了一套完整的工程化体系，LOOP 子系统需要整合其中的核心概念：

### 3.1 三层 Loop 模型

```
L3 Meta Loop:  项目级治理
  ASSESS → REORGANIZE → CONTINUE/COMPLETE
  └─ 产出：新 Task、架构调整、退役决策

L2 Outer Loop:  Round 多 Agent 协调
  PLAN → DISPATCH → EXECUTE_ALL → CONVERGE → CLOSE
  └─ Facilitator 发布计划，Agent 并行拉取

L1 Inner Loop:  单 Agent 任务执行
  PROPOSE → EXECUTE → VERIFY → REFINE
  └─ 每个 Agent 在独立 asyncio Task 中运行，LLM 调用是 I/O bound，天然适合异步并发
```

### 3.2 核心概念映射

| 历史项目概念 | LOOP 系统中的角色 | 实现位置 |
|------------|-----------------|---------|
| Facilitator 模式 | Round 协调者（发布计划/收集结果/收敛判定） | `loop/orchestrator.py` |
| Round 机制 | 多 Agent 并行执行的原子单元 | `loop/round.py` |
| L1 Inner Loop | 单个 Agent 完成单个 Task 的迭代 | `agents/agent.py` |
| 收敛判定 | 每轮结束后的退出条件检查 | `loop/convergence.py` |
| 7 道质量门 | Python 代码级确定性检查 | `gates/` |
| 任务四段式指令 | Task 的统一输入格式 | `loop/task.py` |
| 文件隔离 | 确保并行 Agent 文件集无交集 | `loop/plan.py` 拆分五原则 #1 |
| 退化检测 | 循环质量监控 | `loop/degradation.py` |
| Token 预算 | Agent 调用成本控制 | `loop/token_budget.py` |

### 3.3 Facilitator vs Manager 对比

历史项目明确选择了 **Facilitator 模式**而非 Manager 模式。在 LOOP 系统中，Facilitator 只是 Orchestrator 的一个执行阶段——它拆分任务、构建 DAG、分配到 Agent 池，但不参与 Agent 运行时的 LLM 调用。

| 维度 | Manager（CrewAI Hierarchical） | LOOP 采用的 Facilitator |
|------|------------------------------|------------------------|
| Agent 关系 | 上下级 | 平级，Facilitator 只是规划角色 |
| 任务分配 | Manager 运行时推送 | 规划时静态分配，Agent 拉取 |
| 调度时机 | 运行时实时 | Round 边界 |
| 并发模型 | ThreadPoolExecutor 进程内 | asyncio.gather 进程内 |
| 人的位置 | 嵌入为 "human agent" | 治理层关口（Round Close） |

选择 Facilitator 的原因：三个框架全部用进程内异步并发，LLM 调用是 I/O bound。Round 边界是自然的确认点——Agent 在 Round 内全自动执行，人在 Round 结束做治理决策。

## 四、优化后的 LOOP 架构设计

### 4.1 架构总揽

```
┌─────────────────────────────────────────────────────────────────┐
│                     LOOP 子系统（单一 Python 进程）                │
│                                                                  │
│  ┌──────────────────────────────────────────────────────┐       │
│  │            L2: Orchestrator                          │       │
│  │  PlanBuilder → TaskSplitter → ConvergenceJudge       │       │
│  │  产出: plan_output.json → asyncio.gather(agents...)  │       │
│  └────────────┬─────────────────────────────────────────┘       │
│               │ 构建 Task DAG，分配到 Agent 池                   │
│               ▼                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │         L1: Agent 并行执行（asyncio）                       │  │
│  │                                                            │  │
│  │  asyncio.gather(                                            │  │
│  │    agent_A.execute(task_1),   # LLM API call (I/O bound)   │  │
│  │    agent_B.execute(task_2),   # LLM API call (I/O bound)   │  │
│  │    agent_C.execute(task_3),   # LLM API call (I/O bound)   │  │
│  │  )                                                          │  │
│  │                                                            │  │
│  │  每个 Agent 的 L1 Inner Loop:                               │  │
│  │  PROPOSE → EXECUTE → VERIFY(7 gates) → REFINE              │  │
│  │                                                            │  │
│  │  共享状态通过 Channel 系统协调:                              │  │
│  │  - LastValueChannel: Plan 状态、Review 结论                │  │
│  │  - AccumulatingChannel: Task 完成列表、Gate 结果汇总       │  │
│  │  - BarrierChannel: 等待所有 Agent 完成当前阶段              │  │
│  └────────────────────────────────────────────────────────────┘  │
│                         │                                        │
│                         ▼                                        │
│  ┌──────────────────────────────────────────────────────┐       │
│  │              L2: Round Close                          │       │
│  │  收集产出 → 交叉审查 → 收敛判定 → 产出决策             │       │
│  │  写入 verdict.md + 更新 LoopState                     │       │
│  └──────────────────────────────────────────────────────┘       │
│                                                                  │
│  ┌──────────────────────────────────────────────────────┐       │
│  │              L3: Meta Loop（项目级）                   │       │
│  │  审计 → 架构评估 → 技能生命周期 → 目标再校准            │       │
│  └──────────────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────────┘
```

**并发模型的三个框架验证**：

| 能力 | LangGraph | CrewAI | AutoGen | LOOP 采用 |
|------|-----------|--------|---------|----------|
| 并发原语 | `asyncio` + `ThreadPoolExecutor` | `asyncio.create_task` + `ThreadPoolExecutor` | `asyncio.create_task` | **`asyncio.gather`** |
| 状态协调 | Channel 系统（LastValue/Topic/Barrier） | ContextVar + LockedProxy | Pub/Sub Topic | **Channel 系统（来自 LangGraph）** |
| 文件冲突处理 | 不适用（非代码编辑场景） | Task 顺序执行 | 不适用（非代码编辑场景） | **任务拆分文件隔离** |
| Checkpoint | SQLite/Postgres fork | 事件溯源 + JSON/SQLite | Agent state save/load | **SQLite（来自 LangGraph checkpointer）** |

多 Agent 并发的核心约束：**任务拆分原则 #1（文件隔离）**。Orchestrator 在构建 Task DAG 时确保任意两个并行 Agent 的 `target_files` 集合无交集。这是配置阶段的确定性检查——文件集有交集就拆开或串行，不依赖运行时隔离。

### 4.2 核心设计决策：两层 Loop 取代三层引擎

**决策**：删除 `runtime/` 层，合并 `engine/` + `crew/` 为统一的 `loop/` 模块。

```
旧设计（三层引擎）：
  engine/loop.py      ← while True 循环
  crew/process.py     ← 任务顺序执行
  runtime/runtime.py  ← 消息队列 dispatch

新设计（两层 Loop）：
  loop/
  ├── orchestrator.py   ← L2: 任务拆分 + Plan 构建 + asyncio 调度
  ├── round.py          ← L2: Round 生命周期 + 收敛判定
  ├── plan.py           ← L2: Task DAG + 拓扑排序 + 文件隔离检查
  ├── state.py          ← Channel 系统 + LoopState
  ├── checkpoint.py     ← SQLite Checkpoint 持久化
  ├── convergence.py    ← L2: 退出条件检查（硬上限/质量/停滞/语义）
  ├── degradation.py    ← L2: 退化检测
  └── token_budget.py   ← L2: Token 预算监控

  agents/
  ├── agent.py          ← L1: Agent 基类 + L1 Inner Loop
  ├── llm.py            ← L1: LLMProvider 抽象 + Anthropic/OpenAI 实现
  └── tools.py          ← L1: 工具系统（原 tools/ 合并）
```

**理由**：
1. engine/crew/runtime 三层的"三个循环"在单 Agent 场景完全冗余
2. 多 Agent 并发用 asyncio——LLM API 调用是 I/O bound，三个框架全部这么用
3. Agent 之间的协调通过共享 Channel 状态（来自 LangGraph），不需要进程内消息队列
4. AutoGen 的 Pub/Sub 对实时 Agent 通信有价值，但代码编辑场景的"通信"发生在 Round 边界（Task 分配/结果收集），不是 LLM 调用过程中

### 4.3 并发安全：文件隔离替代文件系统隔离

AutoGen/CrewAI 的 Agent 不直接编辑代码库——它们处理的是消息、文档、工具调用结果。LOOP 系统面对的核心挑战是：**多个 Agent 同时编辑同一个代码库的文件**。

解决方案不是 Worktree 隔离，而是 **Orchestrator 规划阶段的文件冲突检测**：

```python
# loop/plan.py
def check_file_isolation(tasks: list[Task]) -> list[str]:
    """检查并行执行的 Task 是否有文件冲突。返回冲突列表。"""
    parallel_groups = topological_levels(task_dag)
    conflicts = []
    for group in parallel_groups:
        seen_files: dict[str, str] = {}  # file → task_id
        for task in group:
            for f in task.target_files:
                if f in seen_files:
                    conflicts.append(
                        f"文件冲突: {f} 同时被 {seen_files[f]} 和 {task.id} 修改"
                    )
                seen_files[f] = task.id
    return conflicts
    # 有冲突 → Orchestrator 重新拆分或串行化
```

这是确定性检查（Python 代码），不依赖 Agent 自觉。文件隔离是 Task DAG 构建的硬约束——和 Gate 一样，代码级保证。

Git 作为安全网：当文件隔离因意外情况被打破时，Agent 的 git commit 是原子工作单元。git merge 解决冲突比 Worktree 的"物理隔离 + 人工合并"更符合现有开发流程。

### 4.4 状态管理：Channel 系统（借鉴 LangGraph）

```
loop/
├── state.py           ← 共享状态定义 + Channel 系统
│   ├── LoopState      ← Pydantic BaseModel，所有 Channel 的容器
│   ├── Channel[T]     ← 泛型 Channel 基类
│   ├── LastValueChannel[T]
│   ├── AccumulatingChannel[T]
│   └── BarrierChannel
└── checkpoint.py      ← Checkpoint 持久化（SQLite）
    ├── CheckpointManager
    ├── save_state(state, step, round)
    └── restore_state(checkpoint_id) → LoopState
```

`.ae-answers.yml` 是 init 和 dev-loop 的共享契约（项目环境信息）。
`LoopState` 是 Loop 内部的运行时状态（任务进度、Agent 状态、质量门结果）。
两者不重叠——前者是静态配置，后者是动态执行状态。

### 4.5 多 Agent 并发：asyncio + 文件隔离

```
ae dev-loop "实现电商平台用户系统"
  │
  │  Orchestrator 开始:
  │  1. 分析需求 → 拆分为 Task DAG
  │  2. 文件隔离检查（check_file_isolation）
  │  3. 拓扑排序 → 识别可并行的 Task 组
  │
  ▼
Round 1 执行:
  ┌────────────────────────────────────────────┐
  │  asyncio.gather(                           │
  │    developer_A.execute(task_auth_api),     │  # target: src/auth/api.py
  │    developer_B.execute(task_product_page), │  # target: src/product/page.tsx
  │    developer_C.execute(task_user_model),   │  # target: src/user/model.py
  │  )                                         │
  │                                            │
  │  三个 Agent 文件集无交集 → 安全并行          │
  │  共享 LoopState 跟踪各自进度               │
  │  LLM API 调用并发 → I/O 复用               │
  └────────────┬───────────────────────────────┘
               │ asyncio.gather 全部完成
               ▼
  ┌────────────────────────────────────────────┐
  │  Orchestrator Round Close:                  │
  │  1. 收集每个 Agent 的 TaskResult            │
  │  2. Critic 审查所有产出（可并行）            │
  │  3. 运行 7 道质量门                          │
  │  4. 收敛判定 → CONTINUE / DONE              │
  │  5. 保存 Checkpoint                         │
  └────────────────────────────────────────────┘
```

### 4.6 L1 Inner Loop：单 Agent 执行模型

```
Agent.execute(task) 在 asyncio Task 中运行：

1. 读取 Task 四段式指令: 目标/边界/验收标准/禁止项

2. PROPOSE: 分析任务，提出方案假设
   └─ LoopState 记录: task_id → PROPOSE 摘要

3. EXECUTE: 最小改动，只做方案中承诺的事
   ├─ 通过 Anthropic SDK 调用 LLM（I/O bound）
   ├─ LLM 使用 tools（ReadFile/WriteFile/EditFile/Bash）
   └─ LoopState 记录: task_id → EXECUTE 变更文件列表

4. VERIFY: 确定性工具验证
   ├─ Gate 0: Safety (Semgrep/Gitleaks)
   ├─ Gate 1: Lint
   ├─ Gate 2: Type Check
   ├─ Gate 3: Contract (跨 Agent 契约)
   ├─ Gate 4: Test
   ├─ Gate 5: Coverage (≥80%)
   └─ Gate 6: Build
   └─ LoopState 更新 gate_results

5. REFINE: 打磨优化（仅当 VERIFY ✅ 后）
   └─ 改进结构/命名/注释，不改行为
   └─ 再次 VERIFY 确认

6. 产出 TaskResult
   └─ 写入: task_id, status, files_changed, gate_results, commit_hash
```

L1 Inner Loop 的关键约束：
- VERIFY 步骤必须基于确定性工具输出（exit code, lint 报告, test 结果），禁止 LLM 自我审视
- 最多迭代 3 次（max_iterations），超过则标记为 blocked 并停止
- 每次 EXECUTE 后立即 VERIFY，不做批量修改后才验证
```

### 4.7 收敛判定（4 级）

```
check_convergence(round_state) → CONTINUE | GOAL_ACHIEVED | STAGNANT | MAX_ROUNDS

1. 硬上限 🛑
   ├─ 当前 Round >= max_rounds → MAX_ROUNDS
   └─ 单次会话超过 max_duration → MAX_ROUNDS

2. 质量阈值 ✅
   ├─ 所有 Task P0 验收标准通过
   ├─ 所有 Gate 0-6 绿
   └─ goal.yaml 所有 criteria.verified = true → GOAL_ACHIEVED

3. 进展检测 ⚠️
   ├─ 连续 stagnation_threshold（默认 2）轮:
   │   ├─ 无新增 P0/P1 → counter++
   │   └─ 无新增 Task → 确认停滞 → STAGNANT
   └─ 改动量趋势检测: 连续 2 轮递减 → 正常收敛中

4. 语义收敛
   └─ Orchestrator 自问: "本轮产出与上轮相比，
       是否还有实质性的改进空间？"
```

### 4.8 任务拆分五原则

从历史项目继承，应用于 Orchestrator 的 Plan 构建阶段：

| # | 原则 | LOOP 落地方式 |
|---|------|-------------|
| 1 | **文件隔离** | 不同 Agent 的 task.target_files 无交集。有交集 → 串行或再拆分 |
| 2 | **契约优先** | 跨 Agent 协作的 task 先产契约文件，契约定稿后才分配实现 task |
| 3 | **依赖最小化** | 每个 task 的 deps ≤ 2 |
| 4 | **粒度均衡** | 每个 task 预估 20-45 分钟 |
| 5 | **闭环** | 每个 task 产出可独立验证 |

### 4.11 CLI 命令设计

```bash
# ── 单 Agent 开发循环（单轨，一个 ae 进程） ──

$ ae checkpoint list                        # 列出所有 checkpoint
$ ae checkpoint show <id>                   # 查看 checkpoint 详情

单入口设计：`ae dev-loop` 自动判断需求复杂度——简单需求直接单 Agent L1 Loop，复杂需求 Orchestrator 拆分 + 多 Agent 并行。用户不需要手动选择模式。

### 4.12 关键数据结构

```python
# ── Task（借鉴 CrewAI） ──
@dataclass
class Task:
    id: str
    title: str
    description: str               # 四段式指令
    expected_output: str
    role: str                      # developer | critic | reviewer
    deps: list[str]                # 前置 Task ID
    target_files: list[str]        # 涉及的文件
    context_files: list[str]       # 上下文文件
    validation: TaskValidation     # 验证方式
    estimated_minutes: int
    status: TaskStatus             # pending → in_progress → completed | blocked
    output: TaskOutput | None

# ── Round Plan（借鉴 plan_output.json） ──
@dataclass
class RoundPlan:
    round_number: int
    goal: Goal
    tasks: list[Task]
    task_dag: dict[str, list[str]]   # task_id → [dependent_task_ids]
    parallel_groups: list[list[str]] # 拓扑排序后的并行组
    created_at: datetime

# ── LoopState（借鉴 LangGraph Channel 系统） ──
class LoopState(BaseModel):
    round: int
    step: int
    tasks: dict[str, Task]           # LastValue channel
    task_results: dict[str, TaskOutput]  # Accumulating channel
    gate_results: dict[str, GateResult]  # LastValue channel
    signals: list[Signal]            # Topic channel (accumulate=False)
    metrics: MetricsSnapshot         # BinaryOperatorAggregate channel
    status: LoopStatus

# ── Checkpoint ──
@dataclass
class Checkpoint:
    id: str
    round: int
    step: int
    state: LoopState
    parent_id: str | None
    created_at: datetime
```

## 六、关键设计决策

| # | 决策 | 理由 | 日期 |
|---|------|------|------|
| 1 | 删除 engine/crew/runtime 三层，统一为 loop/ 模块 | 三层"循环"在单Agent场景冗余，多Agent并发用asyncio（三个框架全部验证） | 2026-06-24 |
| 2 | 多 Agent 并发使用 asyncio.gather，不是 Worktree 隔离 | LLM API 调用是 I/O bound；文件冲突通过 Task DAG 构建时的 check_file_isolation 确定性检查防止 | 2026-06-24 |
| 3 | Channel 系统（借鉴 LangGraph）作为并行 Agent 共享状态基础 | 不同状态 key 有不同并发语义（单写/多写/等待全部），Channel 抽象完美匹配 | 2026-06-24 |
| 4 | v1.0 先实现单 Agent L1 Loop + 7 道 Gate，Phase 3 增量到多 Agent | 单 Agent 验证核心循环正确，多 Agent 在已验证基础上增量 | 2026-06-24 |
| 5 | 7 道质量门全由 Python 代码执行 | 确定性保证，不依赖 LLM 自觉 | 2026-06-24 |
| 6 | Gate 3（Contract）为多 Agent 场景预留 | 单 Agent 时跳过，多 Agent 时检查跨角色契约一致性 | 2026-06-24 |
| 7 | 收敛判定使用 4 级检查（硬上限→质量→停滞→语义） | 防止无限循环，历史上"无限重试同一任务"是最常见失败模式 | 2026-06-24 |
| 8 | Agent LLM 调用通过 Anthropic SDK（进程内），不是 CLI 子进程 | 这是 Python 应用，控制流在 Python 代码中 | 2026-06-24 |
| 9 | `ae dev-loop` 单入口，自动判断是否需要 Orchestrator 拆分 | 用户不需要手动区分"简单任务"和"复杂项目" | 2026-06-24 |
| 10 | 不引入 Worktree 隔离作为并发机制 | 三个参考框架全部使用进程内异步并发；Worktree 是 CLI 进程隔离的遗留思路，与本项目 Python 应用架构不匹配 | 2026-06-24 |

## 七、风险评估

| 风险 | 概率 | 影响 | 缓解 |
|------|:---:|------|------|
| Agent L1 Loop 中的LLM工具调用失控（无限循环） | 中 | 高 | max_iterations硬上限 + token_budget预算拦截 + CancellationToken超时 |

| Checkpoint恢复后状态不一致 | 低 | 高 | SQLite事务保证 + LoopState schema版本号 |
| 7道门执行超时（大项目） | 中 | 低 | 并行执行独立Gate，可配置timeout |
| LLMProvider切换行为不一致 | 低 | 中 | 统一LLMResponse格式 + Anthropic/OpenAI各自单元测试 |
