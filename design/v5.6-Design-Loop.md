# PART A: 架构概述

## A.0 产品定位

**本项目专注 Loop Engineering**——提供 Claude Code Plugin 形态的 Stage-sequenced Agent loop 调度脚手架。用户在 Claude Code 会话中输入 `/dev-loop "实现登录功能"`，插件通过 Tick-Based 离散调用协议调度 Python Loop Engine 在每次 tick 中做确定性决策（StageRouter → Guardrail → Gate → ConvergenceJudge → Checkpoint），Agent 在 tick 之间执行 LLM 推理工作。

**Init Engineering 是独立项目，不在本项目范围内**。Init 项目负责项目脚手架初始化，loop 项目只消费 Init 产出的结构化产物（通过 Init-Loop 接口契约 IL.1-IL.6）。

**不是 standalone CLI 工具**。CLI (`ae dev-loop`) 是 Engine 的调试入口。正式使用形态是 Plugin——安装在 Claude Code 中，通过 slash command 触发。

| 维度 | 说明 |
|------|------|
| **安装方式** | 复制 `.claude-plugin/` 到项目根，Claude Code 自动识别 plugin.json |
| **交互方式** | Claude Code slash commands (`/dev-loop`, `/status`, `/checkpoint list` 等) |
| **Engine 运行** | 离散 CLI 调用 — `ae dev-loop --init` + `ae dev-loop --tick --result`，每次调用独立进程 |
| **数据持久化** | SQLite (`.ae-state/checkpoints.db`)，跨 tick / 跨 Claude Code 会话保留 |
| **进程模型** | 每次 `--tick` 启动一个 Engine 进程 → 验证结果 → 运行 Guardrail/Gate/Judge → 保存 checkpoint → 输出下一个 action JSON → 退出。循环由 Claude Code Agent 反复调用 `--tick` 驱动 |

**业界参照**: Claude Code plugin convention × VS Code extension manifest × LangGraph PregelLoop (tick/after_tick 控制流在 Python) × CrewAI (Guardrail post-filter) × AutoGen (消息交换机)

## A.1 v5.6 双层架构：Tick-Based Discrete Invocation

```
┌──────────────────────────────────────────────────────────────┐
│ Claude Code Agent (宿主进程)                                   │
│                                                                │
│  /dev-loop "需求" ──→ commands/dev-loop.md                    │
│                        Bash while 循环:                        │
│                          ae dev-loop --init → action JSON      │
│                          loop:                                 │
│                            Agent 按 action 执行 LLM 任务        │
│                            结果写入 stage-result.json          │
│                            ae dev-loop --tick --result → action│
│                          until action == "done"                │
│                              │                                 │
│  hooks/ ── 事件驱动 ──→ pre-tool / post-edit / stop /         │
│                        session-start / on-pr                   │
│                              │                                 │
│  skills/auto-engineering/SKILL.md                             │
│    定义 Agent 如何使用 ae 命令（含 Tick 协议硬约束）            │
└──────────────────────────┬───────────────────────────────────┘
                           │ 文件桥接 (action/result JSON)
                           │ 每次 tick: Python 启动 → 读状态 →
                           │ 验证结果 → Guardrail → Gate →      │
                           │ Convergence → Checkpoint → 输出    │
                           │ action → 退出
┌──────────────────────────▼───────────────────────────────────┐
│ Python Loop Engine (离散 CLI 调用, `uv run ae dev-loop ...`)   │
│                                                                │
│  cli/dev_loop.py ──→ TickOrchestrator                        │
│                        │                                       │
│          ┌─────────────┼─────────────┐                        │
│          ▼             ▼             ▼                        │
│     StageRouter   GuardrailChain  ConvergenceJudge            │
│     (T1-T9)       (9 Guardrail)   (4 级判定)                  │
│          │             │             │                        │
│          ▼             ▼             ▼                        │
│     7+1 Gates (asyncio.gather 并行)                           │
│          │                                                    │
│          ▼                                                    │
│     SQLite Checkpoint (每 tick 保存)                           │
│                                                                │
│  Python 职责: 状态机 · Guardrail · Gate · 收敛 · Checkpoint   │
│  Python 不做: LLM 调用 (不需要 API Key)                        │
│  Agent 职责: Architect(设计) · Developer(TDD编码) ·            │
│             Critic(审查) · DeepAudit(深度审计)                  │
└──────────────────────────────────────────────────────────────┘
```

**两层之间的契约**：Plugin 层做 Bash `while` 循环（反复调用 `--tick`）+ 每个 stage 的 LLM 任务执行。**所有控制流和状态判定在 Python Engine 内**。Plugin 不维护跨 tick 状态。每次 Python 调用是独立进程，从 SQLite 恢复状态，做完决策后写回 SQLite，输出 action JSON 到 stdout，退出。

**核心约束 D1（从未变更）**："Plugin = 薄壳, Engine = 控制流 — Plugin 只做 Bash 委托, 所有 loop 逻辑在 Python"

**为什么是 Tick-Based（离散）而不是连续 while**：
- Claude Code Bash 块是 fire-and-forget——启动命令，等它退出，拿输出。不支持子进程运行时交互 I/O
- 离散 tick 与 LangGraph BSP/CrewAI task 循环/AutoGen 消息队列模式一致——三个参考项目都采用离散步骤 + 状态持久化模型
- 每次 tick 是独立 Python 进程——启动 <0.5s，在 LLM 调用（数秒到数十秒）面前微不足道

## A.2 核心概念

| 概念 | 定义 | 所在层 | 类比 |
|------|------|--------|------|
| **Plugin** | Claude Code 安装单元 — plugin.json + commands/ + hooks/ + skills/ | Plugin 层 | VS Code Extension |
| **Command** | 一个 slash command 的 Markdown 定义，含 Bash 委托块 | Plugin 层 | VS Code Command |
| **Hook** | Claude Code 事件触发器，调 ae 命令或执行 shell 检查 | Plugin 层 | VS Code Activation Event |
| **Tick** | 一次 `ae dev-loop --tick` 调用 = 读状态 + 验证 + Guardrail + Gate + Judge + Checkpoint + 输出 action | Engine 层 | LangGraph tick() |
| **Stage** | 五阶段流水线的一步 (architect/developer/critic/deep_audit/design_sync) | Engine 层 | LangGraph Node |
| **Round** | 一个 Stage 的完整周期 = Agent 执行 + Python 验证推进 | Engine 层 | LangGraph Superstep |
| **Action** | Python 输出给 Agent 的执行指令 (JSON) | 桥接 | CrewAI Task |
| **Result** | Agent 提交给 Python 的执行产出 (JSON 文件) | 桥接 | CrewAI Task Output |
| **Task** | 工作单元, role/description/expected_output/depends_on/file_targets | Engine 层 | AutoGen Task |
| **Guardrail** | 进程内确定性校验, 4 态 (pass/block/drop/retry), 按 timing+stage 过滤 | Engine 层 | CrewAI GuardrailResult |
| **Gate** | subprocess 质量检查, pass/fail, 7 道并行 | Engine 层 | CI Pipeline Stage |
| **StageRouter** | 决定下一 Stage, 综合 stage+verdict+MAJOR+audit_findings | Engine 层 | LangGraph conditional edge |
| **Gap** | Pre-flight 扫描到的设计文档模糊项, 分级 architectural/component/module | Engine 层 | — |
| **Supplement** | gap 的细化产出（用户填写 / research / architect sub-design），注入 DesignDoc | Engine 层 | — |

## A.3 关键设计决策

| # | 决策 | 理由 | 业界参照 |
|---|------|------|---------|
| D1 | Plugin = 薄壳, Engine = 控制流 | Plugin 只做 Bash 委托 + Agent 任务执行, 所有 loop 逻辑在 Python | LangGraph PregelLoop |
| D2 | Tick-Based 离散调用替代连续 while | Bash 块不支持子进程运行时交互 I/O；与 LangGraph BSP 一致 | LangGraph tick/after_tick |
| D3 | 文件桥接替代 JSONL stdin/stdout | 文件协议更健壮、可调试、与 Bash 块机制兼容 | — |
| D4 | Python 是 Gatekeeper，Agent 是 Worker | Agent 不能自行推进状态；所有状态转换必须经 Python 验证 | CrewAI GuardrailResult, AutoGen AgentRuntime |
| D5 | Stage-sequenced, Agent→LoopState 隔离 | Agent 不写共享状态, Orchestrator 控制写入 | LangGraph apply_writes |
| D6 | Task 来源双模式 (外部 YAML / ArchitectAgent 生成) | 灵活适配有/无预定义 Task 的场景 | Devin plan-act-observe |
| D7 | StageRouter + MAJOR 计数分离 | 回路控制独立于状态机 | LangGraph recursion_limit |
| D8 | Gate 全并行 | 7 Gate 互不依赖, 串行浪费 | CI 并行 job |
| D9 | Checkpoint 在每个 tick 后保存 | 避免 resume 重复执行已完成 tick | LangGraph after_tick |
| D10 | Plugin 安装 = 文件复制, 无全局注册 | 项目级隔离, 不同项目可用不同版本 | VS Code workspace extension |
| D11 | 保留 AgentRuntime 路径为调试入口 | 开发者在有 ANTHROPIC_API_KEY 的环境中仍需完整 CLI 调试能力 | — |
| D12 | 合并两套 dev-loop.md 为单一 Tick 协议文件 | 双文件（生产/调试）导致混淆；单一权威源减少 AI 解释偏差 | — |

# Init-Loop 接口契约

> v5.6 扩展（D21 / discussion §十五）：引入版本化 Schema SSOT（A）+ `design_root`/`ci_platform` 字段（B）+ monorepo 限制澄清（C）+ 消费者驱动契约测试（D）。契约**架构选型不变**（单向 / 文件桥接 / 只读 / forward-compat），仅补缺口。

## IL.1 契约总览

| 方向 | 数据 | 位置 | 说明 |
|------|------|------|------|
| **Init → Loop** | `.ae-state/init-manifest.json` | 项目根 | Init 完成时写入, Loop 启动时读取（只读）|
| **Loop → Init** | (无) | — | Loop 不反向调用 Init (单向依赖) |
| **契约 SSOT** | `init-manifest.schema.json` | 双仓库共享 | 版本化 JSON Schema，Init 依它生成 / Loop 依它校验 |

**契约面边界（v5.6 澄清，解 spec 债 — 原缺口 #4）**：`.ae-state/checkpoints.db` 是 **Loop 私有运行时状态，非 Init-Loop 契约面**。当前 Init 不读该 DB。原 IL.1 将其列为"共享契约"却无 schema/版本 = 声明未定义的 spec 债，现移除。未来若 Init 需读，须**另立**版本化契约，不复用 manifest schema。

## IL.2 契约 SSOT — `init-manifest.schema.json`（A，核心）

**问题**：Init 是独立仓库。此前契约由"本设计文档表格 + Loop 侧 `init_contract.validate_*` 手写函数"两处定义，Init 靠读文档表对齐 → 跨仓库漂移必然。

**方案**：抽出版本化 JSON Schema 文件 `init-manifest.schema.json` 作为**双方唯一权威源**：

- **Init 侧**：生成 manifest 后依 schema 自校验（产出即合约）。
- **Loop 侧**：`validate_init_manifest` 改为**对照 schema 校验**（`jsonschema` 库），替代手写字段检查（保留业务级补充校验如版本窗口）。
- **版本联动**：schema 文件含 `$id` + `version`，与 manifest 的 `schema_version` 字段联动；Loop 保留兼容窗口（min 1.0 硬下限 / max 9.9 软上限 WARN）。
- **发布/内化（遵守 B14）**：schema 文件在**本仓库为权威副本**；Init 仓库以**复制 + 版本 pin** 引用（复制内化，非运行时链接）。schema 变更 = 契约变更，需双仓库同步 + bump version。

JSON Schema 骨架（权威定义，实现时抽为独立文件）：

```jsonc
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://auto-engineering/init-manifest.schema.json",
  "version": "1.1",                         // v5.6 引入 ci_platform/design_root → 1.1
  "type": "object",
  "required": ["schema_version", "project_type", "language", "conventions", "structure"],
  "additionalProperties": true,             // IL-AC-03 forward-compat: 未知字段忽略
  "properties": {
    "schema_version": { "type": "string" },
    "project_type": { "enum": ["app-service","library","cli-tool","skill","hook","mcp-server","spec-doc","monorepo"] },
    "language": { "enum": ["python","typescript","go","rust","bash"] },
    "conventions": {
      "type": "object",
      "required": ["linter", "type_checker", "test_runner"],
      "properties": {
        "linter": { "type": "string" },
        "type_checker": { "type": "string" },
        "test_runner": { "type": "string" },
        "build_cmd": { "type": "string" },
        "ci_platform": { "enum": ["github","gitlab","none"], "default": "none" }
      }
    },
    "structure": {
      "type": "object",
      "required": ["source_root", "test_root"],
      "properties": {
        "source_root": { "type": "string" },
        "test_root": { "type": "string" },
        "design_root": { "type": "string" }
      }
    }
  }
}
```

## IL.3 完整字段表（含 v5.6 新增）

| 字段 | 类型 | 必需 | Loop 用途 | 版本 |
|------|------|------|----------|------|
| `schema_version` | string | 是 | 兼容性检测 (min 1.0 / max 9.9) | v5.0 |
| `project_type` | enum(8) | 是 | 类型校验（见下 monorepo 限制）| v5.0 |
| `language` | enum(5) | 是 | 决定 linter/type_checker/test_runner | v5.0 |
| `conventions.linter` | string | 是 | 配置 lint Gate | v5.0 |
| `conventions.type_checker` | string | 是 | 配置 type_check Gate | v5.0 |
| `conventions.test_runner` | string | 是 | 配置 test Gate | v5.0 |
| `conventions.build_cmd` | string | 否 | 配置 build Gate | v5.0 |
| **`conventions.ci_platform`** | **enum(github/gitlab/none)** | **否（默认 none）** | **B13 远程 CI 薄壳选型；Init 声明脚手架了哪个平台** | **v5.6 (B1)** |
| `structure.source_root` | string | 是 | 文件沙箱根 | v5.0 |
| `structure.test_root` | string | 是 | 测试目录 | v5.0 |
| **`structure.design_root`** | **string** | **否** | **设计文档目录约定；Architect design-doc 模式 / Pre-flight Gap 默认检索根** | **v5.6 (B2)** |

**判断决策（我建议、用户已授权，可否决）：**

- **B1 · ci_platform 入 manifest**：Init 知道自己脚手架了 `.github/` 还是 `.gitlab-ci.yml`，由它声明最权威。Loop B13 远程 CI 薄壳据此选型，避免运行时探测目录的脆弱推断。
- **B2 · design 文档内容不入 manifest**：设计文档**内容**是人工/外部输入（`ae dev-loop --design-doc <path>`，T9 已有），manifest 只声明**目录约定** `structure.design_root`。理由：manifest 载 structure/conventions（结构约定），不载 content（设计意图）；Init 不生产设计意图。
- **C · monorepo 已知限制（非承诺）**：v5.6 沙箱 + BatchState 设计基于**单包布局**。`monorepo` 枚举值**保留**（不删，避免破坏 Init 已有输出 = 契约降级），但 Loop 将 `structure.source_root`/`test_root` 视为**主包**根；project_type=monorepo 时输出 WARN 提示单包降级；**多包沙箱隔离推迟**（YAGNI，团队内部工具当前不需要）。

### B1.1a BatchState — 组件/板块进度游标（DS-4，权威定义）

文件: `auto_engineering/engine/batch_state.py` | 写入者: 仅 Orchestrator | 机器视角的路由状态（与 ProgressTree 人视角互补，见 B9）

> 填补原 G4：C 章 orchestrator sketch 用了 ~14 个 BatchState 方法但无集中定义。本节是唯一权威定义。`Plate/Component` 见 B10.4a；`batch`（dict）见 B6.1a。

**层级模型**：`BatchState` 维护 **plate → component → batch** 三级游标。两种模式统一为同一视图：
- **design-doc 模式**：`plates` = `DesignDoc.plates`（真实板块层次）。
- **batch_plan 模式**（模糊需求，无 design_doc）：合成**单一 plate** 包裹 batch_plan 中出现的 distinct components（`_determine_verification_layers` 恒 `total_plates=1`，即 LEAF/PLATE，永不 FULL）。

**字段：**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `plates` | `list[Plate]` | ≥1 | design-doc 模式=真实；batch_plan 模式=单一合成 plate |
| `batch_plan` | `list[dict]` | 0..20 | architect 产出的权威 batch 列表（B6.1a）|
| `current_plate_idx` | `int` | ≥0 | 当前 plate 索引 |
| `current_component_idx` | `int` | ≥0 | **相对当前 plate** 的 component 索引 |
| `current_batch_idx` | `int` | ≥0 | **相对当前 component** 的 batch 索引（**非全局** batch_plan 索引 — 解决原 sketch line 2818/2824 二义）|
| `total_batches` | `int` | =len(batch_plan) | 派生统计，供进度显示 |

**构造（双模式，均在 `_after_architect` batch_plan 就绪后调用）：**

```python
@classmethod
def from_design_doc(cls, doc: DesignDoc, batch_plan: list[dict]) -> "BatchState":
    return cls(plates=doc.plates, batch_plan=batch_plan, total_batches=len(batch_plan))

@classmethod
def from_batch_plan(cls, batch_plan: list[dict]) -> "BatchState":
    # 按出现顺序提取 distinct component → 单一合成 plate
    names = list(dict.fromkeys(b["component"] for b in batch_plan))
    comps = [Component(name=n, design_section="", design_items=[], source_marker="batch_plan") for n in names]
    plate = Plate(name="(single)", design_section="", components=comps, cross_component_contracts_raw=[])
    return cls(plates=[plate], batch_plan=batch_plan, total_batches=len(batch_plan))
```

**访问方法（确定性，无副作用）：**

| 方法 | 返回 | 定义 |
|------|------|------|
| `current_plate()` | `Plate` | `plates[current_plate_idx]` |
| `current_component()` | `Component` | `current_plate().components[current_component_idx]` |
| `batches_for(comp)` | `list[dict]` | `[b for b in batch_plan if b["component"] == comp.name]`（保持 batch_plan 顺序）|
| `current_batch()` | `dict` | `batches_for(current_component())[current_batch_idx]` |
| `current_component_name()` | `str` | `current_component().name` |
| `current_batch_id()` | `str` | `current_batch()["batch_id"]` |
| `current_design_section()` | `str` | `current_component().design_section` |
| `current_batch_tasks(plan)` | `list[Task]` | plan 中属于 `current_batch_id()` 的 developer Task |

**推进方法（有副作用，仅 Orchestrator 调用）：**

| 方法 | 效果 |
|------|------|
| `advance_batch()` | `current_batch_idx += 1` |
| `advance_component()` | `current_component_idx += 1; current_batch_idx = 0` |
| `advance_plate()` | `current_plate_idx += 1; current_component_idx = 0; current_batch_idx = 0` |

**三个完成判定（确定性算法，供 T7/T11/T12 路由）：**

| 判定 | 方法 | 算法 |
|------|------|------|
| **组件所有 batch 完成** | `is_component_complete()` | `current_batch_idx >= len(batches_for(current_component()))` |
| **板块所有组件完成** | `is_plate_complete()` | `current_component_idx >= len(current_plate().components)` |
| **所有板块完成** | `is_all_complete()` | `current_plate_idx >= len(plates)` |
| （辅助）还有 batch | `has_more_batches_for(comp)` | `current_batch_idx < len(batches_for(comp))` |
| （辅助）plate 内还有组件 | `has_more_components_in_plate()` | `current_component_idx < len(current_plate().components)` |
| （辅助）还有 plate | `has_more_plates()` | `current_plate_idx < len(plates)` |

**推进流（与 StageRouter T3/T6/T7/T11/T12 对齐；critic 每 batch 跑，对齐 B6.3 + D18「review=batch」）**：developer 完成**一个 batch** → `critic`（审本 batch 的 diff）→ APPROVE 后 `advance_batch()` → `is_component_complete()`? **否**→下一 batch 回 developer（T6）；**是**→`component_verifier` → 通过后 `advance_component()` → `is_plate_complete()`? 否→下一组件（回 developer 起新组件首 batch）；是→`plate_deep_audit` → `advance_plate()` → `is_all_complete()`? 否→下一板块；是→`system_verifier`。
> 关键：**critic 锚在 batch 边界（每 batch 必跑）**，component_verifier 锚在组件边界（组件所有 batch 的 critic 均 APPROVE 后一次）。原推进流"组件全 batch 完成才 critic"是错的，已纠正。

**边界守卫（DS-4 P2）**：`advance_*` 把游标推出界后、下一 advance 复位前，若 orchestrator 构 action 调 `current_batch()`/`current_component()`/`current_batch_id()`/`current_component_name()` 会 IndexError。**调用约定**：这些访问方法**仅在对应 `is_*_complete()` 为 False 时调用**（推进流已保证：advance 后先判 `is_component_complete()`/`is_plate_complete()`/`is_all_complete()` 决定路由，complete=True 分支不再调越界访问方法而是转下一层 stage）。实现侧访问方法加断言 `assert not is_component_complete()` 兜底，越界即抛明确错误而非静默 IndexError。

**构造期一致性校验（DS-4 P2）**：`from_design_doc` 时校验 `batch_plan` 的 distinct components ⊆ `design_doc` 各 plate 的 components 并集：
- **孤儿 batch**（component 不在任何 plate）→ 构造抛错，触发 G2 retry（architect 重出 batch_plan）；否则该 batch 永不被游标遍历 = 静默漏实现。

### 验证层自动裁剪（Layer-Dependent Routing）

T11/T12 的转换条件根据设计层次自动裁剪——不引入手动模式切换，层次结构本身就决定了验证深度。

```
VerificationLayers = LEAF | PLATE | FULL

判定 (_determine_verification_layers):
  total_components == 1               → LEAF   (单组件，无跨组件交互)
  total_plates == 1 && components > 1 → PLATE  (单板块多组件，有跨组件交互)
  total_plates > 1                    → FULL   (完整验证)

T11 裁剪:
  LEAF:  component_verifier 无缺失 → system_deep_audit    (跳过 plate_deep_audit)
  PLATE: component_verifier 无缺失 + 板块完成 → plate_deep_audit  (不变)
  FULL:  component_verifier 无缺失 + 板块完成 → plate_deep_audit  (不变)

T12 裁剪:
  LEAF:  不触发 (T11 已直接到 system_deep_audit)
  PLATE: plate_deep_audit 无问题 + 板块完成 → system_deep_audit  (跳过 system_verifier)
  FULL:  plate_deep_audit 无问题 + 所有板块完成 → system_verifier (不变)
```

三种模式下的 Agent 数：

| 模式 | Agent 数 | 跳过 | 判定条件 |
|------|---------|------|---------|
| FULL | 7 | 无 | `total_plates > 1` |
| PLATE | 6 | system_verifier | `total_plates == 1 and total_components > 1` |
| LEAF | 5 | plate_deep_audit + system_verifier | `total_components == 1` |

### B3.1 REDGuard 检查逻辑（G7，TDD RED commit-time 校验）

> **borrow**：Superpowers `test-driven-development`「说服」Agent 先写失败测试；REDGuard「强制」——Python commit-time 确定性校验。D16（说服 vs 强制）。

**前置数据**：developer result 增加 `red_evidence: list[{task_id, test_id, red_commit, failure_excerpt}]`（每 TDD task 一条，记录 RED 阶段的测试 commit + 当时失败输出摘要）。

**git 操作序列**（无副作用，只读 git log/history）：

```python
def redguard_check(state, project_root) -> GuardrailResult:
    impl_commit = state.commit_hash                       # 本轮 developer 实现 commit
    for task in state.batch_state.current_batch_tasks(state._plan):
        test_files = [f for f in task.file_targets if _is_test_file(f)]  # tests/ 或 *_test / test_*
        if not test_files:                                # 纯配置/文档 task 豁免
            continue
        # 1. 定位测试 commit: git log --follow test_files, 取早于 impl_commit 的最近一个
        test_commit = _git_log_first_touching(test_files, before=impl_commit, cwd=project_root)
        if test_commit is None:
            return GuardrailResult("retry", f"task {task.id}: 无先于实现的测试 commit — 违反 TDD RED")
        # 2. 祖先校验: test_commit 必须是 impl_commit 的祖先 (git merge-base --is-ancestor)
        if not _git_is_ancestor(test_commit, impl_commit, cwd=project_root):
            return GuardrailResult("retry", f"task {task.id}: 测试 commit 非实现 commit 祖先")
        # 3. 确认当时 FAIL:
        ev = _find_evidence(state.red_evidence, task.id)
        if ev and ev["red_commit"] == test_commit:
            continue                                      # 信任 developer 记录的 RED 证据
        if _STRICT_RED:                                   # 严格模式(opt-in): checkout 重跑确认 FAIL
            if _run_test_at_commit(test_commit, ev["test_id"], project_root) != "FAIL":
                return GuardrailResult("retry", f"task {task.id}: 测试在 red_commit 未 FAIL — 非真 RED")
        else:
            return GuardrailResult("retry", f"task {task.id}: 缺 red_evidence — 补充后重试")
    return GuardrailResult("pass", "")
```

**与 G3/G4 关系**：G3(GitDiffExists)/G4(TestsPass) 确认"有改动 + 测试绿"；REDGuard 补充"测试**先于**实现且**曾红**"——三者共同构成 TDD 闭环校验，不重叠。默认非严格模式（信任 red_evidence，避免 checkout 重跑成本）；`_STRICT_RED` opt-in。

### B3.2 FreshGate 检查逻辑（G8，Gate 证据新鲜度锁定）

> **borrow**：Superpowers `verification-before-completion` 反对"用过期证据声称通过"。

**前置数据**：`gate_results`（EngineState #23）每项扩展 `{passed, message, files_snapshot_sha, ran_at}`——`files_snapshot_sha` = 该 Gate 运行时 `files_changed` 内容的聚合 sha256。

```python
def freshgate_check(state, project_root) -> GuardrailResult:
    current_sha = _aggregate_sha(state.files_changed, project_root)   # 当前工作树聚合哈希
    for gate_name, r in state.gate_results.items():
        if r.get("files_snapshot_sha") and r["files_snapshot_sha"] != current_sha:
            return GuardrailResult("retry", f"Gate {gate_name} 证据陈旧(代码在其后又变更) — 强制重跑 Gate")
    return GuardrailResult("pass", "")
```

**动作语义**：retry → 触发本 stage Gate 重跑（清除陈旧 gate_results），确保收敛判定读到的是**当前代码**的 Gate 结果，杜绝"绿灯锁旧码"。

### B3.3 RegressionGate 检查逻辑（G9，revert-red-restore）

> **borrow**：`verification-before-completion` 的"证明测试真的能失败"。**同一技术复用到审计规则自测**（audit.py 每正则 pattern 附正例/反例断言，随 T30）。DS-3 定案为 Guardrail（有状态改工作树，见 B5.5）。

**触发条件**：仅当前 task 标记为回归修复（batch task 扩展可选字段 `kind: "regression_fix"`；无标记 → pass N/A）。

**序列（改工作树，故串行 Guardrail 非并行 Gate）**：post-developer 时工作树已 clean（G5 已保证全部 committed），故**不能用 `git stash`**（clean tree 上 stash 是 no-op → MUST FAIL 步恒 false-positive → 恒 block）。改用 **checkout 父版本**：把实现文件回退到 `impl_commit` 的父提交（保留 HEAD 的测试文件），跑测试证明回归被捕捉，finally 用 `git checkout HEAD` 恢复。

```python
def regressiongate_check(state, project_root) -> GuardrailResult:
    task = _current_task(state)
    if getattr(task, "kind", None) != "regression_fix":
        return GuardrailResult("pass", "N/A")             # 非回归修复 task 豁免
    test_id = task.regression_test_id                     # 回归修复必带其复现测试 id
    impl_commit = state.commit_hash                       # 本轮 developer 实现 commit
    impl_files = [f for f in task.file_targets if not _is_test_file(f)]  # 只回退实现, 保留测试
    if not impl_files:
        return GuardrailResult("block", f"回归修复 task {task.id} 无实现文件 — 仅改测试无法验证回归")
    # 前置: 当前实现下测试 PASS (已由 G4 确认)
    try:
        # 1. 把实现文件回退到 impl_commit 的父版本(=修复前的 buggy 代码); 测试文件保持 HEAD 不动
        #    clean tree 上此操作使工作树 dirty, 但仅限 impl_files, finally 恢复
        _git_checkout_paths(f"{impl_commit}^", impl_files, project_root)  # git checkout <impl>^ -- <impl_files>
        if _run_test(test_id, project_root) != "FAIL":          # 2. MUST FAIL (证明测试能捕捉回归)
            return GuardrailResult("block", f"回归测试 {test_id} 在回退实现后仍 PASS — 测试无效, 未真正捕捉回归")
    finally:
        _git_checkout_paths("HEAD", impl_files, project_root)   # 3. Restore 实现 (git checkout HEAD -- <impl_files>)
    if _run_test(test_id, project_root) != "PASS":              # 4. 确认恢复
        return GuardrailResult("block", f"回归测试 {test_id} 恢复实现后未 PASS — 工作树状态异常")
    return GuardrailResult("pass", "")
```

**为何不用 stash**：post-developer 工作树 clean（G5 保证），`git stash` 无可暂存内容 → revert 步为空操作 → 测试仍 PASS → MUST FAIL 步永远 false-positive → 恒 block。改用 `git checkout <impl_commit>^ -- <impl_files>` 从 git 历史显式取父版本写入工作树，不依赖未提交改动，clean tree 可用。

**边界**：若某 impl 文件在 `impl_commit` 中系新建（父版本不存在），`git checkout <impl>^ -- <file>` 报 pathspec 错 → 该文件在 finally 前用 `git rm` 移除以模拟"修复前不存在"；回归修复通常改既有文件，此为少见分支，实现时按 checkout 返回码分流。

**block 而非 retry 理由**：测试无法捕捉回归 = 测试本身无效，须**重写测试**（非重试实现）；block 停下交开发者/critic 处理。`finally` 保证任何路径都 `git checkout HEAD` 恢复实现，杜绝工作树残留（呼应破坏性操作审慎原则）。

**代码缺口（Phase 1/6 补齐）**：现 `GuardrailChain.default()` 精确返回 **5 条**（G1-G5，测试 `test_default_factory_returns_5_guardrails` 佐证）。**G6 NoDeferredBlockingGap 亦为 v5.6 新增**（gap_review Guardrail，代码尚不存在，随 gap_review 环节 Phase 落地）——B3 表头"G1-G6 为基线"应理解为"G6 与 G7-G9 同为 v5.6 新增门控，G1-G5 才是既有基线"。G7-G9 随 Phase 6（T29 REDGuard/FreshGate、T30 RegressionGate）加入；需扩展 developer result 契约（`red_evidence` EngineState #33）+ batch task 可选字段（`kind`/`regression_test_id`）。归入 Phase 6。

## B4. ConvergenceJudge — 4 级收敛判定

```
ConvergenceJudge.evaluate(history, design_coverage_ok, system_deep_audit_ok)

  1. len(history) >= max_iterations → stop(4, HARD_LIMIT)
  2. system_verifier MISSING 或 DIVERGED → stop(3, QUALITY)

**停滞检测**: 仅比较同 Stage 连续轮次的产出变化。跨 Stage 自动重置计数器。连续 N 轮 diff_ratio < 5% → STAGNANT。

| `component` | `str` | 是 | 非空 | `_determine_verification_layers`（组件计数）、BatchState 分组、ProgressTree Component 层 |
| `tasks` | `list[Task]` | 是 | ≥1 | 本 batch 的有序 TDD 任务 |
| `depends_on` | `list[str]` | 否 | 默认 `[]`，元素为其它 `batch_id` | batch 级 DAG（round.py 拓扑分层） |

**Task dict（`Batch.tasks[]` 元素）：**

| 字段 | 类型 | 必需 | 约束 | 用途 |
|------|------|------|------|------|
| `id` | `str` | 是 | 单次 run 内唯一，如 `"T1"` | ProgressTree Module→Task 追踪、developer `tasks_completed` 回报 |
| `description` | `str` | 是 | 1..2048 | TDD 单元目标 |
| `module_ref` | `str \| list[str]` | 是 | 如 `"§B6.1"`，跨模块可列多个 | ProgressTree Module 层节点构建（呼应原 line 2006 约束） |
| `file_targets` | `list[str]` | 是 | ≥1，相对 source_root | 精确 git stage（替代 `git add -A`，见 B13/git_tools 修复）+ 文件沙箱白名单 |

**约束（确定性规则）：**

1. **≤5 files/batch**：`len(dedup(⋃ task.file_targets for task in batch.tasks)) ≤ 5`。architect 产出违反 → **G2 PlanExists Guardrail** retry（post-architect，见 B3）。
2. **≤20 batches**：`len(batch_plan) ≤ 20`（EngineState #6）。
3. **batch DAG 无环**：所有 `depends_on` 引用的 `batch_id` 必须在同一 batch_plan 内存在且不构成环；round.py 拓扑排序时环 → block。
4. **component 一致性**：同一 `component` 可跨多个 batch（一个组件拆多 batch 开发）；`_determine_verification_layers` 按 **distinct component 数**判定 LEAF/PLATE。

**规范示例：**

```json
{
  "batch_id": "batch-SR-1",
  "design_section": "B2",
  "component": "StageRouter",
  "depends_on": [],
  "tasks": [
    {"id": "T1", "description": "StageDecision dataclass + next() 骨架",
     "module_ref": "§B2", "file_targets": ["auto_engineering/loop/stage_router.py", "tests/test_stage_router.py"]},
    {"id": "T2", "description": "T1-T22 转换表填充",
     "module_ref": "§B2", "file_targets": ["auto_engineering/loop/stage_router.py"]}
  ]
}
```

**batch Task → developer action task 字段映射（B6.1a Task ≠ developer action task）**：architect 产出的 batch `Task{id, description, module_ref, file_targets}` 经 `_build_action()`（C.3）转为 developer 的 action task `{id, description, expected_output, file_targets, depends_on}`：
- `id`/`description`/`file_targets` 直传；
- `module_ref` **丢弃**（仅供 ProgressTree Module 层构建，不进 developer action）；
- `expected_output` **新增**——由 architect 的 `description` + design_section 派生（"实现并测试通过 <description>"）；
- `depends_on` **新增**——task 级默认 `[]`（batch 内 task 顺序即隐式依赖；跨 batch 依赖由 batch 级 `depends_on` 表达，不下放到 task）。

**代码缺口（Phase 1 补齐，非设计降级）**：现 `task_factory.tasks_from_batch_plan` 消费**扁平** v5.0 结构 `{id, description, files, depends_on}`（batch=单 task），缺 `component`/嵌套 `tasks[]`。按 `design-document-inviolability.md` §1，设计的嵌套结构为权威，代码需在 Phase 1 更新：① `batch["id"]`→`batch["batch_id"]`；② 遍历 `batch["tasks"]` 为每个 task 建一个 developer Task（batch=一组 task，非单 task）；③ `batch.get("files")`→ 各 task 的 `file_targets`；④ 传播 `component` 到 Task 供 BatchState/验证层裁剪；⑤ 现码另消费 `verification`/`steps`（task_factory.py:70-71，v5.5 Phase 3 遗留）——本 schema **不含**这两字段，迁移决策：`steps` 折叠进 `description`（TDD 单元描述已含步骤意图），`verification` 由 `expected_output` + Gate 体系承担，二者从 Task schema **移除**（非保留）。归入 Phase 1 T1/T2/T3 验收。

### B6.6a Haiku verifier 误判兜底（DS-9，权威定义）

> **决策（2026-07-09 用户定案）**：component_verifier / system_verifier 用 Haiku（轻量、成本低），但 Haiku 映射能力弱于 Sonnet，可能误判 MISSING/DIVERGED（假阳 → 冤枉触发 plan_refine 重规划；假阴 → 漏报缺口）。兜底策略：**Haiku 负判定（MISSING/DIVERGED）在触发 architect（T9/T17）前，插入 Sonnet 窄范围复核**——只复核负判定条目（IMPLEMENTED 条目不复核，成本 O(负判定数) 而非全量）。填补 G9。

**为何只复核负判定、只升 Sonnet（不整体换 Sonnet）**：Haiku 对"已实现"的判定通常可靠（代码在、能读到）；易错在"判缺失/偏离"（Haiku 可能没定位到实现文件就报 MISSING）。故只对**负判定**做 Sonnet 二次确认，正判定信任 Haiku。整体换 Sonnet 则丢失 Haiku 的成本优势（verifier 是相对高频的组件/系统边界操作）。

**执行协议（Agent 侧，一 tick 内两模型，呼应 B6.7a 3-subagent 先例）**：verifier action 携带 `recheck` 指令：

```jsonc
"recheck": {"enabled": true, "model": "claude-sonnet-4-20250514", "trigger": "on_negative", "scope": "narrow"}
```

Agent 收到 verifier action 后：
1. Haiku 主 pass → 产出 `coverage_map`。
2. 过滤出 `status ∈ {MISSING, DIVERGED}` 的条目。**无负判定 → 跳过复核**（不 spawn Sonnet，零额外成本），直接写 result。
3. 有负判定 → spawn **1 个 Sonnet 子 Agent**，scope 收窄为 **仅这些负判定条目**（注入：各条目的 `design_item`/`design_section` 原文 + Haiku 猜测的实现位置 + 允许其重新 SearchCode 定位）。
4. Sonnet 对每条负判定回 `confirm`（维持 MISSING/DIVERGED）或 `overturn`（改判 IMPLEMENTED + 给出实现位置）。
5. Agent 用 Sonnet 结论覆盖对应条目，写回最终 `coverage_map` + 附 `recheck_log: [{design_item, haiku_status, sonnet_verdict, final_status}]`。

**Python 侧（after-handler）**：读**复核后**的 coverage_map 决定路由——`missing_count`/`diverged_count` 按 final_status 重算；全部被 overturn → 无负判定 → 走通过分支（T8/T11 或 T16）；仍有确认负判定 → 构造 `RefineRequest`（B6.10）→ T9/T17。`recheck_log` 存入 `action_history` 供审计。

**假阳/假阴兜底（双保险）**：
- **假阳**（Haiku 误报缺口，Sonnet 复核 overturn）→ 本机制直接消除，避免无谓 plan_refine。
- **Sonnet 也误判**（复核本身出错）→ 由下游 **system_deep_audit**（全量 Sonnet 6 维质量审计，exit gate 前必跑）兜底：真实缺口/偏离会在系统级质量审计暴露为 finding，重新触发 plan_refine（T19）。即：verifier 层假阴不是终局，system 层再筛一次。
- **成本/延迟自洽（DS-10）**：Sonnet 复核仅在有负判定时触发、scope 窄（单条目级），LLM 墙钟计入"验证 tick 延迟"独立观测项，**不计入** Python 编排 P95<2s 预算；正常收敛路径（verifier 全 IMPLEMENTED）零复核开销。

**代码缺口（Phase 4，非降级）**：现 B6.4/B6.6 verifier 无 recheck 指令、无 `recheck_log`。Phase 4 实现：verifier action 增 `recheck` 字段（`_build_action`）；Agent prompt（`component_verifier.md`/`system_verifier.md`）增负判定 Sonnet 复核协议；after-handler 按 final_status 重算 count。归入 Phase 4。

4. 阈值判定（供 T9/T19）：`p0_count > 0` 或 `p1_count > p1_threshold`（`p1_threshold` = 现码 `DeepAuditGate.__init__` 既有常量，**默认 6**，见 `deep_audit.py:59`）→ 触发 plan_refine 回 architect；否则通过。阈值可经 `DeepAuditGate(p1_threshold=N)` 配置，默认沿用 6。

> **`applies_to_stages` 纠正（T27）**：现码 `deep_audit.py:57` 的 `applies_to_stages=("critic",)` 是**骨架占位**，与本节"deep_audit 是 plate/system 验证层触发、不入 `run_gates` 并行集"叙述不符。DeepAuditGate **不注册进 `_build_default_gates` 的 7 道环内 Gate**（registry 未注册故当前无害）；Phase 6 T27 实现时改为由 `plate_deep_audit`/`system_deep_audit` 两 stage 的验证层直接调用（`applies_to_stages` 语义作废或改为这两个 stage 名）。

### B6.9 VerificationLayers — 验证层自动裁剪

```python
from enum import Enum

class VerificationLayers(str, Enum):
    LEAF = "leaf"    # 单组件 — 5 Agent
    PLATE = "plate"  # 单板块多组件 — 6 Agent
    FULL = "full"    # 多板块 — 7 Agent

def _determine_verification_layers(
    design_doc: DesignDoc | None,
    batch_plan: list[dict] | None
) -> VerificationLayers:
    """从设计层次自动判定验证深度。不依赖用户手动配置。"""
    if design_doc:
        total_components = sum(len(p.components) for p in design_doc.plates)
        total_plates = len(design_doc.plates)
    elif batch_plan:
        components = {b["component"] for b in batch_plan if "component" in b}
        total_components = len(components)
        total_plates = 1
    else:
        return VerificationLayers.LEAF

    if total_components == 1:
        return VerificationLayers.LEAF   # 跳过 plate_deep_audit + system_verifier
    elif total_plates == 1:
        return VerificationLayers.PLATE  # 跳过 system_verifier
    else:
        return VerificationLayers.FULL   # 全部 5 层验证
```

**LEAF 模式的合理性**：
- plate_deep_audit：只有 1 个 Component 时不存在跨组件契约，Agent 收到的 `cross_component_contracts` 为空——硬审无意义
- system_verifier：scope 与 component_verifier 完全相同（单组件系统的"系统"就是那个组件）——两次 Haiku 调用做同一件事

**PLATE 模式的合理性**：
- system_verifier：单板块内所有组件已被 component_verifier 逐项覆盖，plate_deep_audit 又审了跨组件交互。system_verifier 再遍历全量设计条目是重复工作——plate_deep_audit 通过后直接进入 system_deep_audit 做最终质量审计

### B6.10 plan_refine 输入契约（DS-7，权威定义）

> **填补 G7**：4 个回源（T9 component_verifier / T13 plate_deep_audit / T17 system_verifier / T19 system_deep_audit）都路由回 architect 重规划，但"验证 gap（coverage_map MISSING/DIVERGED）与审计 finding（P0/P1/P2）如何变成 architect 的输入"从未定义 → plan_refine 数据流断裂。本节定义**统一 refine 输入契约**（`RefineRequest`），两类源（覆盖缺口 / 审计发现）归一为同一结构，architect 在 PLAN-REFINE 模式消费。

**数据结构（跨 tick 持久化，序列化为 EngineState `refine_request_json` #35）：**

```python
@dataclass
class RefineGap:
    kind: str            # "MISSING" | "DIVERGED" | "AUDIT_FINDING"
    design_ref: str      # 设计定位: coverage 用 design_item/§X.Y; audit finding 用其关联章节(无则 "")
    detail: str          # coverage 的 note / audit 的 description
    suggested_action: str  # 建议动作(见下映射表)
    severity: str | None = None   # audit: "P0"|"P1"|"P2"; coverage gap = None(视为必修)
    location: str | None = None   # DIVERGED/finding = "file:line"; MISSING = None

@dataclass
class RefineRequest:
    source: str          # "component_verifier"|"plate_deep_audit"|"system_verifier"|"system_deep_audit"
    trigger_tick: int
    scope_plate: str | None       # 重规划范围定位(system 级为 None=全局)
    scope_component: str | None   # component_verifier 源=具体组件; 更高层=None

```

**源 → RefineGap 归一映射（确定性，Python `_build_refine_request()` 执行）：**

| 源字段 | kind | design_ref | severity | location | suggested_action |
|--------|------|-----------|----------|----------|------------------|
| coverage_map item `status=MISSING` | `MISSING` | `design_item` | `None` | `None` | `"补充实现：新增覆盖该设计条目的 batch/task"` |

| deep_audit finding（去重后 P0/P1） | `AUDIT_FINDING` | finding 关联 §（无则 `""`） | `finding.severity` | `finding.file:line` | `finding.suggested_fix`（无则 `"修复该 finding"`） |

> **过滤规则**：只有触发路由的 gap 进入 `gaps[]`——coverage 源取所有 MISSING+DIVERGED；audit 源取去重后 **P0 全部 + P1**（P2 不触发 plan_refine，仅记录，呼应 B6.7a 阈值 `p0>0 或 p1>p1_threshold`）。IMPLEMENTED 条目不进。

```jsonc

  "mode": "PLAN_REFINE",
  "refine_request": { /* RefineRequest 序列化 */ }

```

architect 在 PLAN-REFINE 模式的产出（受 B6.1a batch_plan schema 约束）：
- `MISSING` → 产出**新 batch**（或向现有组件追加 task）覆盖缺口，`design_section` 指向 `design_ref`。
- `DIVERGED` → 二选一并在 `plan` 说明：① 产出修正 task（改代码）；② 更新设计文档并标 `design_docs_updated`（走 C.11 同步，不产 task）。

**prompt 注入（走 B12 fragments，不硬编码）**：新增共享片段 `fragments/refine_input.md`，PromptRegistry 在 architect PLAN-REFINE 模式渲染 `RefineRequest` 为结构化列表注入。架构师 role `.md` frontmatter 的 `fragments` 增列 `refine_input`。B12.1 B 类"per-turn"行的 `audit_findings` 注入由本契约统一替代（不再散注 critic_feedback/audit_findings 两路）。

## B9. ProgressTree — 层次化进度看板

### B9.1 定位

ProgressTree 是**人视角的开发进度仪表盘**，与 BatchState（机器视角的路由状态）互补：

| 维度 | BatchState | ProgressTree |
|------|-----------|-------------|

### B9.2 ProgressNode

@dataclass
class ProgressNode:
    """层次进度节点 — 动态演进，非一次性快照"""

    id: str                          # "sys/B/component/B6/module/B6.1"

    total_tasks: int = 0
    done_tasks: int = 0
    current_task: str | None = None

    verifier_status: str = "pending" # "pending"|"pass"|"failed"|"skipped"
    verifier_missing: int = 0
    verifier_diverged: int = 0

    # 质量审计 (deep_audit)
    deep_audit_status: str = "pending"
    deep_audit_p0: int = 0
    deep_audit_p1: int = 0
    deep_audit_p2: int = 0

    test_coverage_pct: float | None = None

    # Gate
    gate_pass_count: int = 0
    gate_run_count: int = 0

    created_at: str = ""
    updated_at: str = ""

    @property
    def completion_pct(self) -> float:

            return 0.0
        return self.done_tasks / self.total_tasks * 100

### B9.5 父节点聚合规则

```python
def recalculate_parents(self, node_id: str):
    node = self.nodes[node_id]
    children = self.children(node_id)
    if not children:
        return

    # task 进度 = 子节点加权和
    node.total_tasks = sum(c.total_tasks for c in children)
    node.done_tasks = sum(c.done_tasks for c in children)

    # verifier: 任一 failed → failed; 全部 pass(/skipped) → pass; 其余 → pending
    statuses = {c.verifier_status for c in children}
    if "failed" in statuses:
        node.verifier_status = "failed"
    elif statuses <= {"pass", "skipped"}:
        node.verifier_status = "pass"

    # deep_audit: 子节点之和
    node.deep_audit_p0 = sum(c.deep_audit_p0 for c in children)
    node.deep_audit_p1 = sum(c.deep_audit_p1 for c in children)
    node.deep_audit_p2 = sum(c.deep_audit_p2 for c in children)

    # coverage: 子节点 task 加权平均
    covered = [c for c in children if c.test_coverage_pct is not None]
    if covered:
        total = sum(c.total_tasks for c in covered)
        if total > 0:
            node.test_coverage_pct = sum(
                c.test_coverage_pct * c.total_tasks for c in covered) / total

    if node.parent_id and node.parent_id in self.nodes:
        self.recalculate_parents(node.parent_id)
```

### B9.8 动态同步协议

**plan_refine 回路同步流程**：

```
1. Architect 收到 plan_refine feedback → 产出新 batch_plan / 更新设计文档
2. _after_architect() 调用:
   a. 设计文档更新 → ProgressTree.sync_from_design_doc(doc)
   b. 仅 batch_plan 更新 → ProgressTree.sync_from_batch_plan(batch_plan)
3. sync 方法（匹配键 = `design_section_ref` 归一化后**精确**匹配，非模糊字符串比对）：
   归一化 = strip 空白 + 统一 `§` 前缀（如 `"B6.1"`/`" §B6.1 "` → `"§B6.1"`）。逐节点决策：
   | 条件 | 动作 | SyncResult |
   |------|------|-----------|
   | ref 命中现有节点 | 更新 `total_tasks`（保留 `done_tasks`），`version += 1` | `modified` |
   | ref 未命中 且 parent_ref 已在树中 | 添加 `ProgressNode(version=1)` | `added` |
   | ref 未命中 且 parent_ref 也不在树中（层次悬空） | 不添加，标记 `[NEED MANUAL]` | `conflicts` |
   | 现有节点 ref 本轮不再出现 | `design_status="removed"`（不删除） | `removed` |
   末步：对每个 added/modified/removed 节点调 `recalculate_parents()` 向上聚合。
4. 展示同步摘要: "ProgressTree: +2 added, 1 modified, 0 removed, 0 conflicts"
```

**removed 节点**：不删除，保留审计轨迹。后续重新出现则恢复。`ae progress --show-removed` 查看。

**conflicts 处理（非阻塞）**：ProgressTree 是纯人视角看板，**从不参与路由**，故 `conflicts` **不 block 主循环**——sync 继续、下一 tick 照常推进。冲突节点在展示尾部以 `⚠ N conflicts (see: ae progress --show-conflicts)` 提示，供人工在设计文档中修正章节引用后由下一次 plan_refine sync 自动消解。路由所需的缺失/发散由 verifier/audit + plan_refine 回路（B6.6a/B6.10）独立保证，与本看板无耦合。

## B10. Pre-flight Design Gap Analysis — 设计文档模糊性预检

### B10.1 定位

Loop 启动时，设计文档往往**部分章节清晰、部分粗略**（用户强调："我给你设计文档一定所有地方都是清晰完整的吗？可能其中一部分是粗的"）。若直接进入主循环，粗略章节会拖到 component_verifier/plate_deep_audit 阶段才暴露为缺失，触发 plan_refine 回路——代价高（已开发的组件可能推倒重来）。

Pre-flight Gap Analysis 在 **T1 architect 之前**插入一个 Phase 0：扫描设计文档清晰度 → 分级 → 让用户在开发开始前决策（补充/研究/延后）。**只在 `--design-doc` 模式启用**；纯模糊需求模式（无设计文档）跳过，由 architect 单模式推理承担（D11）。

| 维度 | plan_refine 回路（已有） | Pre-flight（本节新增） |
|------|------------------------|----------------------|
| 触发时机 | 主循环中 verifier/audit 发现缺失 | 主循环**之前** |
| 代价 | 高（可能推倒已开发代码） | 低（尚未开发） |
| 决策者 | Python 自动回 architect | **用户显式介入** |
| 处理方式 | architect in-loop sub-design | 用户 Fill / Research / Defer |

两者互补：Pre-flight 消化"启动时可预见的模糊"，plan_refine 兜底"开发中才暴露的缺失"。Defer 的 gap 最终仍走 architect in-loop sub-design（D11）。

### B10.2 GapItem / GapReport

文件：`engine/gap_analysis.py`（新建）

```python
@dataclass
class GapItem:

    summary: str                  # 缺什么（人类可读）
    depends_on: list[str]         # 依赖的其他 gap id（architectural 常级联）
    resolution: str = "pending"   # "pending"|"fill"|"research"|"defer"|"defer_research"
    user_note: str | None = None

@dataclass
class GapReport:
    gaps: list[GapItem]
    scanned_sections: int
    has_blocking: bool            # 存在 architectural gap → 不允许全部 defer
```

### B10.3 分级标准

**grade（模糊的 scope，驱动阻塞约束与解决顺序）：**

| grade | 含义 | 影响范围 | 判定信号（gap_scan Agent 用） | 默认处理约束 |
|-------|------|---------|------------------------------|------------|
| **architectural** | 板块级/跨组件契约模糊（如"组件间通信协议未定义"） | 级联多个组件 | 模糊点被 ≥2 个 component 的设计引用 / 涉及跨组件数据流、接口契约、协议 / `parse_warnings` 报"无可识别层次" | **阻塞**——不允许全部 defer，须 Fill/Research 先解 |
| **component** | 单组件接口/职责模糊 | 单组件 | 模糊点局限于 1 个 component 的公共接口/职责边界 / Component 有 `design_items=[]` | 该组件开发前解，可 defer 到 architect sub-design |
| **module** | 模块内实现细节模糊 | 单模块 | 模糊点是组件内部实现细节（算法选型、数据结构），不影响对外契约 | 可 defer——architect in-loop 细化足够 |

分级由 gap_scan Agent 判定（需语义判断，Python 不承担）——上表"判定信号"列即其判级 rubric（进 `gap_scan.md` prompt）。architectural gap 因级联性必须优先解决——组件设计依赖板块契约。

**clarity（模糊的 kind，正交于 grade，驱动 gap_review 建议路径）：**

| clarity | 含义 | 典型来源 | gap_review 建议默认路径 |
|---------|------|---------|------------------------|
| **missing** | 章节完全缺失或空（`design_items=[]` / 无该 §） | parse 无内容 | **Fill**（从零补写，外部知识帮助有限）|
| **vague** | 有内容但太笼统，不足以实现（无具体 schema/算法） | 有 § 但无可验证断言 | **Research**（需外部知识/参考实现充实）|
| **partial** | 部分清晰、部分缺（如有字段无算法） | key_claims 稀疏 | **Fill 或 Defer**（缺口小，architect in-loop 可细化）|

> clarity 仅**呈现给用户**辅助 gap_review 选路（非强制），grade 才是阻塞约束的依据。二者组合示例：`architectural+missing` = 最高风险（跨组件契约完全缺失，必须 Fill/Research）。

### B10.4 DesignDoc + Supplement

pre-flight 与 plan_refine 的细化产出统一注入 `DesignDoc.supplements`，供 architect 消费：

```python
@dataclass
class Supplement:
    gap_id: str
    design_section_ref: str
    content: str                  # 细化后的设计内容
    source: str                   # "user" | "research_agent" | "architect_sub_design"
    source_tier: str | None       # research 来源层（见 B10.6），非 research 为 None
    confidence: str               # "high" | "medium" | "low"
    created_at: str

@dataclass
class DesignDoc:
    plates: list[Plate]
    supplements: dict[str, Supplement]      # gap_id → Supplement
    parse_warnings: list[str] = field(default_factory=list)  # parse() 结构不确定项（B10.4a 异常决策表产出，喂 gap_scan）
    # parse() 产出 plates/components/design_items（契约与层次识别见 B10.4a）
```

architect（T1 design-doc 模式）读取原始章节 + 对应 supplement；supplement 存在时以其为细化依据。Supplement 随 EngineState 持久化（跨 tick，见 C.10 #29）。

### B10.4a DesignDoc.parse() 契约与层次识别（DS-6，权威定义）

> **决策（2026-07-09，用户定案）**：层次识别用**混合策略**——① 标题层级启发（默认）② `<!-- ae:* -->` HTML 注释标记消歧（优先级高于启发）③ 解析不确定 → gap_scan 报 architectural gap 兜底。本节是 `plates/components/design_items` 识别的唯一权威定义（填补原 G6：C.12 T4 只有一行"解析层次结构"，无 how）。

**层级数据模型：**

```python
@dataclass
class DesignItem:
    item_id: str              # 稳定 id: f"{design_section}-{seq}", 如 "B2-1"
    design_section: str       # 章节 ref, 如 "§B2"
    title: str                # 条目标题
    key_claims: list[str]     # 可验证断言 (供 component_verifier 设计→代码映射)
    source_marker: str        # "explicit_marker" | "heading" | "table_row" | "list_item"

@dataclass
class Component:
    name: str                 # 组件名 (须等于 batch_plan[].component, 用于验证层裁剪/BatchState)
    design_section: str       # "§B6.1"
    design_items: list[DesignItem]
    source_marker: str

@dataclass
class Plate:
    name: str                 # 板块名
    design_section: str
    components: list[Component]
    cross_component_contracts_raw: list[str]  # 板块级跨组件契约声明 (供 plate_deep_audit action)
    def cross_component_contracts(self) -> list[str]: return self.cross_component_contracts_raw
```

**parse() 签名与契约：**

```python
@classmethod
def parse(cls, path: str | Path) -> "DesignDoc":
    """markdown 设计文档 → DesignDoc(plates=[...], supplements={})。
       纯结构识别 (确定性, Python), 不做语义充分性判定 (那是 gap_scan Agent 的职责)。"""
```

**① 标题层级启发（默认映射）**：Python markdown 解析（用成熟库 `markdown-it-py`，不自造正则）——

| markdown 元素 | 映射 | 说明 |
|---------------|------|------|
| `#` (H1) / `PART X` | 忽略（文档/分区标题，非层次单元） | — |
| `##` (H2) | **Plate** 候选 | 如 `## B6. Agent 规格` |
| `###` (H3) | **Component** 候选 | 如 `### B6.1 ArchitectAgent` |
| `####` (H4) / 表格行 / 列表项 | **DesignItem** 候选 | 具体断言 |
| `design_section` | 从标题抽取的编号（`B6.1`→`§B6.1`）| 无编号时用标题 slug |

**② 显式标记消歧（优先于启发，冲突时 marker 赢）**：文档作者可在标题下方插 HTML 注释显式声明层次，消除启发歧义——

```markdown
## Engine 层
<!-- ae:plate name="Engine层" contracts="TickOrchestrator→StageRouter.next() 参数匹配; ConvergenceJudge 读 round_history 格式" -->
### StageRouter
<!-- ae:component name="StageRouter" -->
```

标记语法：`<!-- ae:plate name="X" [contracts="a; b"] -->` / `<!-- ae:component name="X" -->` / `<!-- ae:design-item title="X" claims="c1; c2" -->`。存在标记 → 覆盖该节点的启发式归属与命名。

**③ key_claims 抽取规则**：一个 DesignItem 的 key_claims 来自其正文中的**可验证断言**——表格行、列表项、含 `必须/禁止/shall/≤/≥/==` 的句子、代码契约（`def`/dataclass 字段）。无标记时启发抽取；有 `claims="..."` 标记时以标记为准。

**异常结构处理决策表：**

| 情况 | 行为 |
|------|------|
| 文件不存在 | `raise FileNotFoundError`（`ae doctor` 应预检兜底，见 CLI doctor）|
| 无 H2/H3 结构且无 ae 标记 | `plates=[]` + `parse_warnings` 记录 → Phase 0 gap_scan 报 **architectural** gap「设计文档无可识别层次，请加 `<!-- ae:* -->` 标记或用 vague-requirement 模式」 |
| Component 无任何 DesignItem | 保留该 Component，`design_items=[]` → gap_scan 报 **component** gap |
| 标题跳级（H2 直接到 H4）| 按最近祖先 + 启发归属，记 `parse_warning` → 交 gap_scan review |
| explicit marker 与启发冲突 | **marker 优先**（作者显式意图明确）|

**与 B10 gap_scan 的复用关系（分工，不重叠）**：`parse()` 只做**结构识别**（层次拓扑，确定性 Python）；`gap_scan` Agent 做**语义充分性判定**（章节内容是否够细到可实现，需 LLM）。parse 产出的 `parse_warnings`（结构不确定项）作为 gap_scan 的输入之一。这也是原 [Q?] "Haiku 覆盖准确率" 的根因缓解：清晰的层次结构（标记加持）→ component_verifier 映射输入更确定。

**与 batch_plan 的一致性**：`Component.name` 必须与 `batch_plan[].component`（B6.1a）取值域一致——design-doc 模式下 architect 依 DesignDoc.plates 产 batch_plan，两者 component 命名同源。


### B10.5 处理流程（4 条用户路径）

用户对每个 gap 选择：

| 路径 | 动作 | 产出 | ProgressTree |
|------|------|------|-------------|
| **Fill** | 用户直接补充设计内容 | Supplement(source=user, confidence=high) | node → stable |
| **Research** | 触发 ResearchAgent 检索 | Supplement(source=research_agent) | node → stable |
| **Defer** | 留给 architect in-loop 细化 | 无（node 标记 fuzzy） | node → fuzzy |
| **Defer + Research** | 现在检索存档，细化留给 architect | ResearchFindings（作为 architect 输入，不直接成 Supplement） | node → fuzzy（带 research 附注） |

**约束**：`has_blocking==true`（存在 architectural gap）时，这些 gap 不允许选 Defer——否则组件设计无契约依据（由 gap_review 后的 Guardrail 强制，见 B3）。

### B10.6 ResearchAgent — 分层知识源

文件：`agents/prompts.py` + `agents/authz.py`（扩展）

| 维度 | 值 |
|------|-----|
| role | research |
| tools | ReadFile, SearchCode, ListDir, WebSearch, context7 |
| model | claude-sonnet-4-20250514（需综合推理） |
| scope | **单个 gap** |
| 触发 | gap_review 中用户选 Research / Defer+Research |
| output | ResearchFindings{gap_id, findings, sources[], source_tier, confidence, recommended_design} |

**四层知识源解析（核心）**——优先用被策展的领域相关源，外部盲搜为兜底：

```
Tier 0  项目规则声明（先读，拿"地图"）
  └─ 读 CLAUDE.md + .claude/rules/ 的 reference 声明表
  └─ 提取 {领域 → 源路径, 借鉴点, 读取约束}
  └─ 本项目实例：langgraph→pregel/_loop.py（tick 控制流）, autogen→AgentRuntime, crewai→guardrail.py

Tier 1  声明的参考代码（grep-scoped 三步法）
  └─ 匹配 gap 领域 → grep 定位 → 50-200 行 Read → 提炼 → 丢弃
  └─ ⚠️ 禁止批量/并行扫描，遵守 CLAUDE.md ⚠️ 硬禁令（96GB 内存事故根因）

Tier 2  项目历史文档 / KB
  └─ design/, docs/, CLAUDE.md 声明的 doc KB 路径

Tier 3  外部搜索（fallback）
  └─ WebSearch / context7 — 仅当 Tier 0 无相关声明，或 Tier 1-2 不足
```

**治理规则**：

| 情况 | 行为 |
|------|------|
| CLAUDE.md 声明了该 gap 领域的源 | **必须**先走 Tier 1-2，Web 仅补充 |
| CLAUDE.md 无相关声明 | 直接走 Tier 3，但记录"未找到项目源" → 提示用户可补声明 |
| Tier 1-2 命中但不完整 | Tier 1-2 结论 + Tier 3 补缺 |

**产出可信度标注**：每条 findings 标注来源 tier（"借鉴自 langgraph/_loop.py:L120" vs "web，未验证"），architect/用户据此判断可信度。**资产化边界（YAGNI）**：反向工程历史代码为知识库是**项目预置的前置条件**（用户在 CLAUDE.md 声明源路径 + 借鉴点），ResearchAgent 只消费声明、不在 loop 运行时建库。

### B10.7 依赖处理

- `gap.depends_on` 记录 gap 间依赖（architectural gap 常级联多个 component gap）
- 解决顺序：先 architectural → 再 component → 后 module
- architectural gap 未解 → 依赖它的 component gap 的 Research/Fill 应等待（上下文依赖板块契约）

### B10.8 ProgressTree 集成

- gap 对应 `node.design_status = "fuzzy"`（pre-flight 扫描时标记）
- Fill / Research 解决 → `design_status = "stable"`，supplement 记录来源
- Defer / Defer+Research → 保持 "fuzzy"，architect in-loop sub-design 后转 "stable"
- `ae progress` 展示 fuzzy 节点带 ⚠️ 标记，用户可见哪些设计尚未落定

## B11. Agent Prompt 行为塑形（借鉴 Superpowers）

### B11.0 定位与边界

Superpowers（`~/.claude/superpowers/`，9 SKILL.md + 零依赖）是一套成熟的**提示词行为塑形方法论**。我们借鉴其**提示词技术加固 Agent 行为层**，但**不借鉴其 Agent 自调节的执行模型**——我们的 Python 门控（Tick 协议）是结构约束，比提示词说服更强。

**互补关系**：Superpowers 教 Agent"应该怎么做"（说服）；Auto-Engineering 强制 Agent"必须怎么做"（门控）。两层叠加：
- **结构层（不可绕过）**：Python Guardrail + Gate + StageRouter + ConvergenceJudge —— 已有
- **行为层（可被合理化）**：Python 无法确定性检查的部分（TDD 是否真 RED 先行、critic 是否真审、architect 是否真细化）—— 本节用 Superpowers 技术加固

本节内容为**开发时写入 B12 中央提示词目录**（`prompts/roles/` 正文 + `prompts/fragments/` 共享片段；C 类命令 `.md` 由 `sync-prompts.py` 注入）的成品提示词块（对应实施计划 C.12 的 T13-T16g、T10-T11）。

### B11.1 CSO Description 纪律（description = 触发条件，非工作流摘要）

**Superpowers 实测发现**：description 中总结工作流会让 Claude 走捷径跳过正文。一段"code review between tasks"导致 Claude 只做一次审查（正文要求两次）；改为纯触发条件后才正确读流程。

**应用到我方文件**（关键——Tick 协议依赖 Agent 不跳阶段）：

| 文件 | ❌ 错误（摘要工作流，诱导捷径） | ✅ 正确（仅触发条件） |
|------|------------------------------|---------------------|
| `plugin.json` dev-loop command | "Runs architect→developer→critic→verifier→audit loop with TDD and gates" | "Use when the user wants to run an automated multi-stage development loop for a requirement" |
| `skills/SKILL.md` description | "Executes the 8-stage tick protocol with layered verification" | "Use when initiating or resuming an Auto-Engineering dev-loop in the current session" |

**规则**：description 只写"何时用"，绝不写"做什么/怎么做"。工作流细节在正文，靠 Agent 读正文而非读摘要。

### B11.2 Iron Law（单一不可商量的核心规则）

Superpowers 每个纪律技能压缩出一句铁律（"NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"）。我方两条铁律，置于对应文件顶部：

```
# SKILL.md / dev-loop.md 顶部（结构层铁律）
IRON LAW: PYTHON IS THE GATEKEEPER.
NO STAGE ADVANCEMENT WITHOUT `ae dev-loop --tick` VALIDATION.
You may NOT edit code before Python outputs {"action":"developer"}.
You may NOT declare done before Python outputs {"action":"done"}.
Violating the letter of this rule is violating the spirit of this rule.
```

```
# DeveloperAgent prompt 顶部（行为层铁律）
IRON LAW: NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST.
RED (write failing test, watch it fail) → GREEN (minimal code, watch it pass) → REFACTOR.
Wrote code before the test? Delete it. Start over. Delete means delete.
```

### B11.3 Red Flags 列表（Agent 自检停止信号）

置于 `SKILL.md` + `commands/dev-loop.md`，编码"静默降级禁令"（CLAUDE.md）为可自检信号：

```markdown
## Red Flags — STOP，不要继续，向用户报告
- 我正准备在 Python 输出 {"action":"developer"} 前编辑代码
- 我正准备在 Python 输出 {"action":"done"} 前宣布完成
- Bash 块失败了，我正准备静默切换到手工模式继续
- Agent tool spawn 失败了，我正准备自己手工模拟这个 stage
- 我正准备跳过 --tick 自己推进到下一个 stage
- critic 返回 MAJOR，我正准备忽略 findings 直接进收敛

以上任何一条都意味着：停止。向用户报告失败原因 + 状态 + 选项。禁止静默降级。
```

### B11.4 合理化破解表（Rationalization Tables，逐 Agent）

**方法**：Superpowers 从基线压力测试中收集 Agent 在时间/沉没成本/疲劳压力下的真实合理化话术，逐条写反驳。以下为我方各 Agent 的成品表，粘贴进对应 system prompt。

**DeveloperAgent（TDD 破解表 → B6.2 prompt）**：

| 合理化话术 | 反驳 |
|-----------|------|
| "这个 task 太简单不需要测试" | 简单代码也会坏。RED 只花 30 秒。 |
| "我先写实现再补测试更高效" | 测试后 = "这段代码做了什么"；测试先 = "这段代码应该做什么"。 |
| "我已经手工验证能跑" | 手工 ≠ 可复现。TestsPass Guardrail 需要真实测试。 |
| "先写失败测试只是形式主义" | 违反字面就是违反精神。先写 RED。 |
| "critic 反正会审，我快点提交" | critic 审代码质量，不补你漏掉的测试。Gate 会挡你。 |

**CriticAgent（反橡皮图章表 → B6.3 prompt）**：

| 合理化话术 | 反驳 |
|-----------|------|
| "diff 很小，APPROVE 快点推进" | 小 diff 照样发 P0。逐行审查改动。 |
| "测试通过了代码就没问题" | 测试通过 ≠ 代码好。按维度审。 |
| "我不想用 MAJOR 阻塞进度" | 橡皮图章才浪费进度——bug 会在 3 轮后回来。 |
| "看起来符合 plan 要求" | 对文件核实，不靠感觉。引用 file:line。 |
| "这是本轮 diff 之外的问题，不管" | 记录为 finding，标注超出本 diff 范围，不静默。 |

**ArchitectAgent（反偷懒设计 / YAGNI 表 → B6.1 prompt）**：

| 合理化话术 | 反驳 |
|-----------|------|
| "这段设计文档够清晰了，跳过 sub-design" | 若 component_verifier 会判 MISSING，现在细化更便宜（代码前）。 |
| "顺手加点灵活性/配置给未来用" | YAGNI。只为当前明确需求设计。 |
| "多塞几个文件进一个 batch 更快" | batch >5 文件会撑爆 critic 的 diff 审查。≤5。 |
| "需求模糊，我先猜一个" | 模糊 → gap_scan / sub-design，不是猜。 |

**Component/System VerifierAgent（反虚假覆盖表 → B6.4/B6.6 prompt）**：

| 合理化话术 | 反驳 |
|-----------|------|
| "文件存在，设计条目就算实现了" | 存在 ≠ 覆盖。每条设计声明映射到 file:line。 |
| "跟设计意图差不多" | DIVERGED 是 finding，不是 pass。报告它。 |
| "大部分实现了，剩下的忽略" | 遍历全部条目，缺一条报一条。 |

### B11.6 渐进披露（Progressive Disclosure，token 效率）

### B11.7 什么不借鉴（3 条根本架构不兼容，防后续误采）

### B12.4 PromptRegistry 组合机制

每个 `roles/*.md` 用 frontmatter 声明要 include 的片段：

```yaml
---
role: developer
model: claude-sonnet-4-20250514
fragments: [iron_law_tdd, rationalization_developer, letter_vs_spirit]
---
（developer system prompt 正文…）
```

```python
class PromptRegistry:
    """启动时一次性加载全部 prompt，组合 fragments，锁 content hash。"""
    def __init__(self, prompts_dir: Path):
        self._roles: dict[str, str] = {}       # role → 完整 system prompt
        self._hashes: dict[str, str] = {}      # role → sha256(content)
        self._load_all(prompts_dir)

    def get(self, role: str) -> str: ...        # 返回组合后的完整 system prompt
    def hash(self, role: str) -> str: ...       # 返回 content hash（写入 checkpoint）
    def _load_all(self, d: Path) -> None:
        # 读 roles/*.md → 解析 frontmatter.fragments → 拼接 fragments/*.md → 存 + hash
        ...
```

组合规则：`fragments` 声明的片段按顺序**追加到正文顶部**（Iron Law/Red Flags 需前置以最大化遵守）。

### B12.5 加载时机 + 版本锁（可复现性）

- **加载时机**：Engine `init` **一次性**加载全部 prompt，运行中不重载（保持 Python 门控确定性）
- **版本锁**：每个 prompt 组合后算 `sha256` → 写入 EngineState + checkpoint。resume 时校验 hash，prompt 变更导致 hash 不符 → 警告（同一 loop 不应换 prompt）
- **禁止**运行时可变 prompt——外部文件变化不影响进行中的 loop

### B12.6 C 类命令同步机制

`scripts/sync-prompts.py`：把中央 `fragments/*.md` 注入命令 `.md` 的标记区，保证一致不手抄。

```markdown
<!-- FRAGMENT:red_flags START -->
（由 sync-prompts.py 从 prompts/fragments/red_flags.md 注入，勿手改）
<!-- FRAGMENT:red_flags END -->
```

pre-commit 或 CI 校验标记区与中央片段一致（漂移即 fail）。

### B12.7 边界（YAGNI，明确不做）

| 不做 | 理由 |
|------|------|
| 模板引擎（Jinja 等） | 片段拼接用简单字符串组合即可，团队内部工具不需要 |
| 热重载 / 运行时可变 prompt | 破坏 Python 门控确定性；prompt 仅 init 加载 |
| A/B 测试框架 / prompt DB | 5-20 人内部工具，非 prompt-ops 平台 |
| 把命令 `.md` 移出 `.claude-plugin/` | Claude Code 发现机制的结构约束 |
| 压力测试评估基建 | 见 B11.7，属独立 eval 项目 |

**收益**：单一源消除漂移；B11 片段写一次 9 Agent 复用；prompt 可独立 diff/review/优化不动 Python；hash 锁保可复现。

## B13. Commit → PR Review → CI/CD Pipeline

> 本章统一设计 Commit → PR Review → CI/CD 端到端 Pipeline 各动作的**时机、颗粒度、自动化程度、工具选型**。补齐此前设计中散落在 PE.2/PE.3/B5/B6 各处的零散描述，整合为单一权威源。

### B13.0 核心心智模型

> **Loop 环内全自动到 `done`（commit + 本地 Gate + 5 层 AI review），环外由人触发把成果推成 PR 交人类 review + merge。**
> 颗粒度从细到粗层层收口：`task → batch → component → plate → system → PR`。

### B13.1 颗粒度金字塔

```
        system  ← 整个需求/设计文档（所有板块）        —— PR / 人类review / merge 挂这里
      ╱   plate  ← 1个板块（若干组件）                 —— plate_deep_audit 挂这里（天然PR切分点）
    ╱   component ← 1个组件（若干batch，如"EngineState")—— component_verifier 挂这里
  ╱     batch    ← ≤5文件（若干task）                  —— critic(diff) + 本地Gate 挂这里
╱        task    ← 1个TDD单元（1个失败测试→通过）       —— commit 挂这里（最细）
```

**自动裁剪**（D13 LEAF/PLATE/FULL）：输入需求的层次结构决定验证深度——单组件(LEAF)跳过 plate+system_verifier，单板块(PLATE)跳过 system_verifier，多板块(FULL)全量 5 层。**PR 粒度遵循同一原则：由输入需求层次决定，不引入手动模式切换。**

### B13.2 时间轴（一次 FULL dev-loop 完整生命周期）

```
T0  architect  需求 → 拆 板块/组件/batch/task            [自动·system级]
│
├─══ 板块 P1 ══════════════════════════════════════════
│   ├─ 组件 C1
│   │   ├─ batch b1:  task→task→[COMMIT]→[COMMIT]        ① COMMIT  [自动·task级]
│   │   │            → critic 审 diff                    ② REVIEW-L1[自动AI·batch级]
│   │   │            → 本地 Gate ×7 并行                  ③ CI-本地 [自动·stage级]
│   │   └─ batch b2:  …同上…
│   │   → component_verifier(C1 设计项全覆盖?)           ② REVIEW-L2[自动AI·component级]
│   ├─ 组件 C2 …
│   └─ plate_deep_audit(P1 跨组件交互质量)               ② REVIEW-L3[自动AI·plate级]
│
├─══ 板块 P2 ══ …同上…
│
├─══ 全量收口 ═════════════════════════════════════════
│   → system_verifier(全量设计覆盖)                      ② REVIEW-L4[自动AI·system级]
│   → system_deep_audit(全量代码质量6维)                 ② REVIEW-L5[自动AI·system级]
│   → ConvergenceJudge → {"action":"done", GOAL_ACHIEVED}
▼
╫════════ 环 界 线：以上全自动，Python 门控，无人介入 ════════
▼
   人触发 /ae:code-review                                ← 人工闸门（红线#4）
   → git push -u origin <branch>                         ④ PUSH  [半自动·branch级]
   → gh pr create + body(gate表/diff/清单)               ⑤ PR    [半自动·loop级]
   → on-pr.sh 追加 gate 结果
   → 【应补】远程 CI ci.yml (pull_request → make ci)      ③ CI-远程[自动·PR级] ← 当前缺
   → 人类 peer review（架构/意图/安全）                   ② REVIEW-human[人工·PR级]
   → merge to main                                      ⑥ MERGE [人工决策·PR级]
```

**环界线**：AI 负责把代码写对写好（环内），人负责判断"要不要进主干"（环外）。

### B13.3 动作主表（6 动作 × 5 维度）

| 动作 | 时间点 | 颗粒度 | 自动/人工 | 触发者 | 实现 |
|------|--------|--------|-----------|--------|------|
| ① **commit** | GREEN+REFACTOR 后、critic 前 | **task**（最细） | **自动** | developer Agent | GitCommitTool（⚠ 待收窄：`git add -A`→按 target_files 精确 stage） |
| ② **review-AI** | 5 个检查点递进 | batch→comp→plate→system | **自动** | 5 层验证 Agent | B6 规格 + StageRouter T1-T22 |
| ③ **CI-本地** | 每 stage 后（asyncio.gather）| batch/stage | **自动** | Python Gate（B5） | 环内增量快子集（lint 改文件 + type_check + 增量 test；skip coverage） |
| ③ **CI-远程** | PR/MR 打开时 | PR/MR | **自动**（**当前缺**）| GitHub Actions / GitLab CI（薄壳） | `make ci` = 全量 ruff + pytest(覆盖率≥90%) + build；**平台无关单一逻辑入口** |
| ④ **push** | PR 创建前 | **branch**（整轮累积） | 半自动（人触发命令） | `/ae:code-review` 命令 | Bash 块 `git push -u origin` |
| ⑤ **PR** | loop `done` 后 | **loop = 一个需求** | 半自动（工具生成 body） | `/ae:code-review` 命令 | `gh pr create` + body（gate 表+diff+清单） |
| ② **review-human** | PR 创建后 | PR | **人工** | reviewer | GitHub/GitLab PR/MR 界面 |
| ⑥ **merge** | review 通过后 | PR | **人工** | 维护者 | 红线#6/#7 人类决策，**绝不由 AI 触发** |

### B13.4 自动/人工边界

```
全自动（Python门控，Agent执行，可 checkpoint 恢复）    │  人工（不可自动化的判断）
──────────────────────────────────────────────────┼──────────────────────────
① commit (task)                                     │  触发 /ae:code-review
③ CI-本地 gate (stage)                              │  ⑤ PR 前的确认（红线#4）
② review-AI 5层 (batch→system)                     │  ② review-human (PR)
④ push body 生成 / ⑤ PR body 生成                    │  ⑥ merge (PR → main)
③ CI-远程 (PR)  ← 当前缺                             │
```

### B13.5 PR 颗粒度策略（输入端控制）

**核心原则**：PR 的「切分粒度」≠ PR 的「人工闸门时机」。人工闸门恒锚在 `done` 之后（环界线外），仅粒度由输入端决定。

| 方案 | PR 何时创建 | 中断自动化? | PR 粒度 | 新增复杂度 | 适用场景 |
|------|------------|:---------:|---------|-----------|---------|
| **A** 当前: 一 loop 一 PR | done 后 | 否 | loop（FULL 时大） | 无 | 快速迭代 |
| **B** done 后延迟切分 | done 后**批量**按 plate | 否 | plate（stacked PR） | 中 | 想要板块 PR 又不想停 |
| **C** 每板块停+PR | 每 plate 后（interrupted）| **是** | plate | 高（loop 挂起/resume） | 高风险/强监管，**显式开关** |
| **D** ★ 输入端控粒度 | done 后 | **否** | **= 输入需求粒度** | **零** | **推荐默认** |

**推荐 D**：与 D13（自动裁剪 LEAF/PLATE/FULL）一脉相承——不在 loop 里造切分机制，而是"你想要板块级 PR，就用板块级需求启动 loop"。选择权在输入端。

- 想要「一 PR 一板块」→ dev-loop 只喂**一个板块**的需求（走 LEAF/PLATE 裁剪）→ 天然一 loop = 一 PR = 一板块
- 明确要「多板块一起做」→ 喂 FULL 需求 → 接受一个大 PR（方案 A），或在 done 后按 plate 拆（方案 B）

**方案 C 仅适用**：强监管环境明确要求"每个板块必须人工签字确认后才能建下一个板块"——此时中断是**特性不是缺陷**，但必须作为**显式开关**（非默认行为）。

### B13.6 CI 双平台架构（单一逻辑入口 + 平台薄壳）

**原则**：CI 真实逻辑收敛到一个平台无关入口，GitHub Actions / GitLab CI 仅为"checkout → 装依赖 → 调入口"的薄壳。逻辑零重复（DRY）。

**为什么"单一逻辑入口"可行**：远程 CI 跑的是 Gate（质量门），不是 dev-loop（Agent 循环）。dev-loop 需要 LLM + 人在 Claude Code 里触发，不进 CI。CI 是纯静态检查（safety 正则 / lint / type_check / test / build），**不需要 `ANTHROPIC_API_KEY`**。

```
              ┌────────────────────────────────┐
              │   单一 CI 逻辑入口（平台无关）    │
              │   make ci  =  ruff lint + pytest│
              │   uv run ae gate-check --all    │
              │   ← 本地 / 远程 / 环内 Gate      │
              └────────────────────────────────┘
                     ▲                    ▲
          ┌──────────┘                    └──────────┐
   .github/workflows/ci.yml            .gitlab-ci.yml
   (GitHub Actions 薄壳)               (GitLab CI 薄壳)
   on: pull_request                    rules: merge_request
```

**GitHub Actions 薄壳**:
```yaml
name: CI
on:
  pull_request:
  push:
    branches: [main]
jobs:
  gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v5
        with: { python-version: "3.12" }
      - run: uv sync --all-extras --dev
      - run: make ci                     # ← 唯一逻辑入口
```

**GitLab CI 薄壳**:
```yaml
stages: [gate]
gate:
  stage: gate
  image: python:3.12-slim
  before_script:
    - pip install uv
    - uv sync --all-extras --dev
  script:
    - make ci                            # ← 同一个入口
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == "main"
```

**YAGNI 提醒**：只落实际使用的平台即可——薄壳设计的价值是"需要哪个加哪个，逻辑不重复"。金融企业内网若以 GitLab 为主、GitHub 仅做 plugin marketplace 分发，则 CI 落 GitLab、GitHub 保留 release 薄壳即可。

### B13.7 环内 Gate vs 远程 CI：增量快子集 vs 全量权威（共享标准，非共享运行时）

**核心纠正（用户反馈）**：环内 Gate 和远程 CI **永不互相调用**，是独立进程。共享的是**检查标准/配置**（`pyproject.toml`），不是运行时实现。

**"同一套 Gate 实现"不可取**：远程 CI 跑全量 + coverage，环内高频路径不应承担此成本。把远程的重活塞进环内 = 拖慢 loop → 违背"高频窄范围用轻量"原则（BEACON #40）。

| 维度 | 环内 Gate（B5） | 远程 CI（B13.6） |
|------|----------------|------------------|
| 频率 | **高**（每 stage，几十次/loop） | **低**（每 PR 一次） |
| 耗时要求 | **秒级**（不能拖慢 loop） | 分钟级 OK（后台异步） |
| 范围 | **增量快子集** | **全量权威** |
| 具体跑 | lint 改动文件 + type_check + **增量 test**（`-k 相关文件`）+ skip coverage | **全量 pytest + coverage(≥90%) + build + safety 全扫描** |
| 目的 | 快速反馈，挡明显错误 | 权威复核，团队可信 |
| 运行时 | Python subprocess（asyncio.gather 并行） | GitHub/GitLab runner |
| 标准来源 | `pyproject.toml`（ruff rules / pytest 配置 / 覆盖率阈值） | **同一份** `pyproject.toml` |

**"环内绿 ≈ 远程绿"的正确含义**：共享 `pyproject.toml` 同一标准 ⇒ 本地绿≈远程绿。**不是在环内跑全量来保证一致性。**

- 环内**红** ⇒ 远程**必红**（明显问题当场挡住，高频轻子集已暴露）
- 环内**绿** ≠ 远程一定绿（远程还跑全量测试 + coverage ≥90%，低频重子集更严）——但这恰恰是我们要的：**高频挡明显问题，低频兜全量底。**

**附带发现**：现设计 B5.1 环内 `test` gate 超时 180s（`subprocess.run([runner])`），若跑全量 pytest 则每个 developer stage 都会很慢。应改为增量测试（只跑改动文件相关），全量 + coverage 交给远程 CI 一次性跑。

## B14. 外部 Skill/Agent 依赖管控（Internalization Constraint）

> 本章定位为**设计约束**——不是功能设计，而是项目对所有外部依赖的准入规则。结论来自 2026-07-09 全项目外部依赖审计。

### B14.0 核心原则

**项目的 runtime 依赖面必须是"可移植的"**——任何人在任何 Claude Code 环境安装本 Plugin 后，不应因缺失某个外部 agent/skill/framework 而运行失败。

**判断标准**：
- ✅ **PyPI 库**（pip install 自动拉）——标准依赖渠道
- ✅ **系统二进制**（doctor 预检，明确告知用户安装）——可文档化

- ❌ **Claude Code 内置 agent spawn**（Plan/code-reviewer 等）——假设宿主环境存在，不可移植
- ❌ **外部框架专属 agent spawn**（gsd-* / superpowers-* 等）——跨框架耦合，高危

## B15. 借鉴 Superpowers 验证方法论加固审计与验证层

> 本章来自 2026-07-09 两组分析的合并：① Superpowers 三工具（test-driven-development / verification-before-completion / requesting-code-review）方法论借鉴；② 本项目 `/audit` 命令三层实现现状与缺口。两组结论去重后统一为本章借鉴清单。核心哲学承接 B11 / D16：**Superpowers 说服"应该"，我方门控强制"必须"**——本章把 Superpowers 的验证纪律转成 Guardrail / Gate 的**确定性检查**（非 Agent 自觉）。

### B15.0 核心定位

Superpowers 验证类 skill 的本质是**"验证证据必须真实且新鲜"的纪律**。本项目已有 critic/verifier/audit 层结构，但缺三个确定性检查：① RED 阶段是否**真见过失败**（Iron Law: NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST）；② Gate 证据是否**新鲜**（非 stale，NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE）；③ 回归测试是否**真能捕捉回归**（revert 后必须 FAIL）。这三项 Python 门控可确定性执行，不依赖 LLM 自觉。

### B15.1 `/audit` 命令三层现状

| 层 | 文件 | 现状 |
|----|------|------|
| 命令入口 | `.claude-plugin/commands/audit.md` | 委托通用 `/audit` Phase 1→2→3（**依赖 Superpowers 运行时**，违反 B14）|
| 确定性 Gate | `auto_engineering/gates/audit.py` (476 行) | 6 个正则 pattern（吞异常/密钥/TODO/注释代码/调试残留/超长行/大文件），支持增量。**只有语法检查，无语义** |
| LLM 审计 Gate | `auto_engineering/gates/deep_audit.py` (123 行) | **骨架** — 不 spawn agent，只从 contracts 读 findings 统计。真正的 LLM 审计从未运行（`deep_audit.py:47` 自述"当前 Phase 1 为骨架实现"）|

### B15.2 借鉴清单（按主题）

**主题 A — TDD RED 阶段强化（借鉴 test-driven-development Iron Law）**
- **REDGuard**（新 Guardrail）：developer 提交实现前，Python 校验"该 task 存在一个**先于实现 commit** 的测试 commit 且该测试当时 FAIL"。commit-time 对比实现——B11.2 已注入 prompt（说服），B15 加门控（强制）。

**主题 B — 验证新鲜度（借鉴 verification-before-completion Gate Function）**

- **RegressionGate**（**新 Guardrail**，DS-3 定案：非 Gate，见 B5.5——有状态改工作树，与并行 Gate 不兼容，依 #36 先例归 Guardrail，`applies_to_stages=(developer,)`）：对回归修复类 task 执行 Write→Pass→Revert→**MUST FAIL**→Restore→Pass 序列，确定性证明"测试真能捕捉这个回归"。**同一技术复用到审计规则自测**（audit.py 每个正则 pattern 附正例/反例断言）。

**主题 C — 审查上下文（借鉴 requesting-code-review）**
- **crafted context**：给 deep_audit / critic LLM 的是**针对性上下文**（改动 diff + 相关设计文档片段），非全量代码——降噪、提升信噪比。

# PART C: v5.6 Tick-Based Discrete Invocation Protocol

> 本章为 v5.6 核心新增内容——Tick-Based 生产执行路径的完整规格。
> PART B 中所有组件（StageRouter/Guardrail/Gate/ConvergenceJudge/Checkpoint）在本章中全部复用。

## C.1 问题定义

### C.1.1 原始设计约束（从未变更）

```
Python 控制流: StageRouter(22条转换) · ConvergenceJudge(4级) · Gate(7+1并行) ·
               Guardrail(9道3态) · Checkpoint(SQLite WAL) · 阈值计算 · MAJOR计数
LLM 能力:      Architect(设计+plan+sub-design) · Developer(TDD编码) ·
               Critic(diff审查) · ComponentVerifier(组件级需求验收) ·
               PlateDeepAudit(板块级跨组件审计) ·
               SystemVerifier(系统级需求覆盖) · SystemDeepAudit(系统级质量审计)
```

**D1**: "Plugin = 薄壳, Engine = 控制流"

**核心设计**: 验证分层——每一层的范围和频率不同：
```
critic (每batch, diff范围) → component-verifier (每组件, 单组件范围) 
  → plate-deep-audit (每板块, 跨组件范围) → system-verifier (系统完成时, 全量设计)
    → system-deep-audit (系统完成时, 全量代码)
```

### C.1.2 为什么 JSONL 协议不可行

JSONL 协议架构正确（Python while 循环 → stdout/stdin JSONL 与 Agent 通信），但实现机制有缺陷：

```
Python while loop → write_jsonl(stdout) → read_jsonl(stdin) → continue
                         ↑                    ↓
            Claude Code Bash 块: fire-and-forget，不支持子进程运行时交互 I/O
```

Claude Code 的 Bash 块启动命令后等待退出才拿输出，无法在子进程运行期间持续读写 stdin/stdout。

**但"实现机制有缺陷"≠"架构设计错误"。** 正确做法是换一个桥接机制（Tick-Based 文件桥接），而不是连 Python 调度层一起抛弃。

### C.1.3 v5.4/v5.5 偏离设计的具体表现

- `.claude-plugin/commands/dev-loop.md`: Agent Tool 纯 markdown — 依赖 AI 自觉，无强制力
- Python orchestrator 12 步主循环、StageRouter、ConvergenceJudge、Gate 并行、SQLite checkpoint **全部被绕过**
- BEACON 决策 33/34 被错误标记为 ❌ 已废弃——审计 AI 无声降级设计文档

### C.1.4 三个参考项目的共同模式

| 维度 | LangGraph | CrewAI | AutoGen | v5.6 |
|------|-----------|--------|---------|------|
| **控制流** | PregelLoop.tick() Python | _execute_tasks() Python for | _process_next() Python while | TickOrchestrator Python |
| **LLM 调用** | Node 函数内 SDK | Agent.execute_task() 不透明 | Agent.on_message() 消息响应 | Claude Code Agent spawn（文件桥接） |
| **状态持久化** | SQLite checkpoint | 无内置 | save/load_state | SQLite WAL checkpoint |
| **任务调度** | prepare_next_tasks() 版本比较 | 顺序 for 循环 | FIFO 队列 | StageRouter T1-T11 |
| **质量门禁** | interrupt_before/after | GuardrailResult post-filter | 干预中间件 | 7+1 Gates asyncio.gather |
| **收敛判定** | recursion_limit | 无（手动） | stop_when_idle() | ConvergenceJudge 4 级 |

**核心洞察**: 三个参考项目都采用**离散步骤 + 状态持久化**模型。没有哪个要求 Python 进程在 LLM 调用期间保持连续运行。

## C.2 Tick 架构总览

### C.2.1 核心思路

**将 Python orchestrator 的连续 while 循环改为离散的 CLI 调用序列。** 每次调用处理一个"tick"——从 SQLite 读取当前状态、验证 Agent 提交的结果、运行 Guardrail/Gate/ConvergenceJudge、保存 checkpoint、输出下一个 action JSON、退出。

循环由 Claude Code Agent 反复调用 `ae dev-loop --tick` 驱动，但**状态机控制权始终在 Python**。

### C.2.2 关键约束：Python 是 Gatekeeper

**Agent 不能自行推进状态。** 每次状态推进都必须经过 Python 验证：

| Python 职责（Gatekeeper） | Agent 职责（Worker） |
|--------------------------|---------------------|
| 决定当前应执行哪个 Stage | 按 Python 指令执行具体 LLM 任务 |
| 管理 batch 完成状态 + 组件依赖 | 不接触状态机 |
| 验证 Agent 提交的结果格式和内容 | 产出符合格式要求的 JSON 结果 |
| 运行 Guardrail（pre/post） | 不参与 Guardrail 判定 |
| 运行 Gate 并行检查 | 不参与 Gate 判定（只接收结果） |
| 执行收敛判定（4 级） | 不能自行宣布"完成" |

### C.2.6 Tick 延迟预算与降级（DS-10，权威定义）

> **决策（2026-07-09 用户定案）**：预算约束的是 **Python 编排开销**（确定性控制流），**目标 P95 < 2s**；LLM 墙钟与 Gate/Guardrail 子进程墙钟**不计入**该预算（各由既有 timeout 兜底、单独观测）。**超预算只告警不中断**——延迟是可观测性指标，不是正确性门控。填补 G10 + 关闭 BEACON Tick 延迟 [Q?]。

**为何切分"编排开销"而非"整 tick 墙钟"**：整 tick 墙钟被外部工具主导（pytest 可达 180s、LLM 数秒~数十秒），对这些设 P95 无意义（不由 Python 控制）。真正反映 Python Engine 健康度的是**去掉外部子进程/LLM 后的纯编排时间**——SQLite I/O、状态反/序列化、result schema 校验、纯 Python Guardrail 逻辑、StageRouter、ConvergenceJudge、写 action JSON、checkpoint 写入。这部分若 >2s，说明 Engine 自身有性能退化（如 state 膨胀、序列化低效），是可行动信号。

**测量点定义（墙钟切分）：**

| 符号 | 含义 | 计入编排预算 |
|------|------|:-----------:|
| `t_total` | tick 进程墙钟（`exit − start`）| — |
| `t_gate` | `run_gates` asyncio.gather 墙钟（lint/type_check/test/build 子进程）| ❌ 单独观测（各 Gate 有 30-180s timeout）|
| `t_guard_sub` | Guardrail 内子进程墙钟（G3/G5/G7/G9 的 git 命令、RegressionGate 的 `_run_test`）| ❌ 单独观测 |
| **`t_orchestration`** | **`t_total − t_gate − t_guard_sub`** = 纯 Python 确定性编排 | ✅ **目标 P95 < 2s** |

> LLM 墙钟天然不在 `t_total` 内——Agent 在**两个 Python tick 之间**执行 LLM 任务（Bash 循环里），Python tick 只读上一 result + 出下一 action。故编排预算无需显式减 LLM。3-agent 并行审计的 LLM 墙钟（B6.7a）+ verifier Sonnet 复核墙钟（B6.6a）纳入**"审计/验证 tick 的 LLM 墙钟"独立观测项**，与编排预算正交。

**测量实现**：TickOrchestrator 在关键段落打点（`time.perf_counter()`）：进程入口→出口 = `t_total`；`run_gates` 前后 = `t_gate`；每个 Guardrail 内 subprocess 前后累加 = `t_guard_sub`。三者写入 `action_history` 的每条 tick 记录（`{tick, t_total_ms, t_gate_ms, t_guard_sub_ms, t_orchestration_ms}`），供 Phase 5 分布分析。

**超预算行为（只告警不中断）：**

```python
    t_orch = t_total - t_gate - t_guard_sub
    if t_orch > ORCH_BUDGET_MS:   # ORCH_BUDGET_MS = 2000
        log.warning(f"[latency] tick {tick} 编排开销 {t_orch:.0f}ms 超预算 2000ms "
                    f"(total={t_total:.0f} gate={t_gate:.0f} guard_sub={t_guard_sub:.0f})")
    # 不中断: tick 正常输出 action, 无 retry / 无降级 / 不影响正确性
```

告警只写日志 + 打点（不 block、不改路由、不触发 retry）。P95 判定在 Phase 5 测试离线聚合（滚动窗口），不在运行时逐 tick 判 P95。

**Phase 5 验收（新增测试）**：跑 ≥30 tick 的代表性 run，收集 `t_orchestration_ms` 分布，断言 **P95 < 2000ms**；同时输出 `t_gate`/LLM 墙钟分布作参考（不设阈值，仅观测）。对应 C.12 Phase 5。

**代码缺口（Phase 2/5，非降级）**：现 TickOrchestrator 无打点、`action_history` 无延迟字段、无 Phase 5 延迟测试。Phase 2 加打点 + `action_history` 延迟字段；Phase 5 加分布断言测试。归入 Phase 2（打点）+ Phase 5（验收）。

### C.5.1 构造与状态管理

```python
class TickOrchestrator:
    """Discrete-tick orchestrator with layered verification."""

    def __init__(self, project_root: Path | None = None):
        self.project_root = project_root or Path.cwd()
        self._state: EngineState | None = None
        self._router: StageRouter | None = None
        self._guardrail: GuardrailChain | None = None
        self._judge: ConvergenceJudge | None = None
        self._gates: list[Gate] | None = None
        self._plan: Plan | None = None
        self._checkpoint_store: CheckpointStore | None = None
        self._init_manifest: dict | None = None
        self._design_doc: DesignDoc | None = None  # 解析后的设计文档层次
        self._batch_state: BatchState | None = None  # 当前组件/batch 进度
        self._progress_tree: ProgressTree | None = None  # 层次化进度看板
        self._verification_layers: VerificationLayers | None = None  # 验证层裁剪
```

### C.5.2 公共入口

```python
    def init(self, requirement: str, design_doc_path: str | None = None,
             max_rounds: int = 5) -> dict:
        """初始化 loop。有设计文档时解析层次结构。"""
        # 读 init-manifest
        manifest_path = self.project_root / ".ae-state" / "init-manifest.json"
        if manifest_path.exists():
            self._init_manifest = json.loads(manifest_path.read_text())

        # 读设计文档并解析层次结构
        if design_doc_path:
            self._design_doc = DesignDoc.parse(design_doc_path)
            # 产出: plates[]→components[]→design_items[](design_section, key_claims)。契约见 B10.4a

        # 创建 state
        self._state = EngineState(requirement=requirement, thread_id=str(uuid4()))
        self._router = StageRouter()
        self._guardrail = GuardrailChain.default()
        self._judge = ConvergenceJudge(ConvergenceConfig(
            max_iterations=max_rounds))   # DS-8: plan_refine 上限移至 StageRouter(分源2/全局4), 不在 Judge
        self._gates = DEFAULT_GATES
        self._checkpoint_store = CheckpointStore(
            str(self.project_root / ".ae-state" / "checkpoints.db"))

        # batch 状态: 延迟到 _after_architect (batch_plan 就绪后) 用 from_design_doc/from_batch_plan 统一构造 (见 B1.1a)
        # __init__ 不建 BatchState — batch_plan 此时尚不存在

        # T0.1/T1: 有设计文档 → gap_scan (Pre-flight 模糊性预检); 无 → architect
        if self._design_doc:
            self._state.stage = "gap_scan"
            self._state.expected_stage = "gap_scan"
        else:
            self._state.stage = "architect"
            self._state.expected_stage = "architect"
        self._state.tick = 0
        self._save_checkpoint()
        return self._build_action()

    def tick(self, result_file: Path) -> dict:
        """处理一个 tick: 验证 → Guardrail → Gate → 路由 → Checkpoint → action."""
        result = self._read_and_validate(result_file)
        if isinstance(result, ErrorResponse):
            return result.to_dict()

        self._apply_result_to_state(result)

        gr = self._guardrail.check("post", self._state.stage, self._state, self.project_root)
        if gr.action != "pass":
            return self._handle_guardrail_result(gr)

        if self._state.stage == "developer":
            gate_results = asyncio.run(
                run_gates(self._gates, self.project_root, "developer",
                          contracts=self._state.contracts))
            self._state.gate_results = {
                name: {"passed": v.passed, "message": v.message}
                for name, v in gate_results.items()}

        return self._after_tick(result)
```

### C.5.3 核心路由

```python
    def _after_tick(self, result: dict) -> dict:
        stage = self._state.stage
        handlers = {
            "gap_scan": lambda: self._after_gap_scan(result),
            "gap_review": lambda: self._after_gap_review(result),
            "research": lambda: self._after_research(result),
            "architect": lambda: self._after_architect(),
            "developer": lambda: self._after_developer(),
            "critic": lambda: self._after_critic(result),
            "component_verifier": lambda: self._after_component_verifier(result),
            "plate_deep_audit": lambda: self._after_plate_deep_audit(result),
            "system_verifier": lambda: self._after_system_verifier(result),
            "system_deep_audit": lambda: self._after_system_deep_audit(result),
        }
        handler = handlers.get(stage)
        if handler:
            return handler()
        return ActionError(error_code="UNKNOWN_STAGE",
                          message=f"Unknown stage: {stage}").to_dict()

    # ─── Phase 0: Pre-flight Gap Analysis (仅 --design-doc 模式) ───

    def _after_gap_scan(self, result: dict) -> dict:
        # T0.2/T0.3: gap_scan → gap_review (有 gap) / architect (无 gap)
        report = json.loads(self._state.gap_report_json)   # 已由 _apply_result_to_state 写入
        gaps = report["gaps"]
        # 标记对应 ProgressTree 节点为 fuzzy（若已构建）
        if self._progress_tree:
            for g in gaps:
                node = self._progress_tree.find_by_design_section(g["design_section_ref"])
                if node:
                    node.design_status = "fuzzy"
        if not gaps:
            self._advance_stage("architect")           # T0.3
        else:
            self._advance_stage("gap_review")           # T0.2
        return self._build_action()

    def _after_gap_review(self, result: dict) -> dict:
        # T0.4/T0.5: gap_review → research (有待研究) / architect (全 Fill/Defer)
        # G6 Guardrail 已确认无 architectural gap 被 defer
        decisions = self._state.pending_gap_decisions
        report = json.loads(self._state.gap_report_json)
        by_id = {g["id"]: g for g in report["gaps"]}
        pending_research = []
        for d in decisions:
            g = by_id.get(d["gap_id"])
            if not g:
                continue
            g["resolution"] = d["resolution"]
            g["user_note"] = d.get("user_note")
            if d["resolution"] == "fill":
                self._inject_supplement(g, d.get("fill_content", ""),
                                        source="user", source_tier=None, confidence="high")
            elif d["resolution"] in ("research", "defer_research"):
                pending_research.append(g["id"])
        self._state.gap_report_json = json.dumps(report)
        self._state.pending_research_ids = pending_research
        if pending_research:
            self._advance_stage("research")             # T0.4
        else:
            self._advance_stage("architect")            # T0.5
        return self._build_action()

    def _after_research(self, result: dict) -> dict:
        # T0.6/T0.7/T0.8: research → research / gap_review / architect
        report = json.loads(self._state.gap_report_json)
        by_id = {g["id"]: g for g in report["gaps"]}
        current_id = self._state.pending_research_ids.pop(0)
        g = by_id[current_id]
        if g["resolution"] == "research":
            # 直接落为 supplement
            self._inject_supplement(g, result.get("recommended_design", ""),
                                    source="research_agent",
                                    source_tier=result.get("source_tier"),
                                    confidence=result.get("confidence", "medium"))
        else:  # defer_research: findings 存档，细化留给 architect
            self._state.research_archive[current_id] = result
        self._state.gap_report_json = json.dumps(report)

        if self._state.pending_research_ids:
            self._advance_stage("research")             # T0.6
        elif self._state.research_archive:
            self._advance_stage("gap_review")           # T0.7 (defer_research 复审) — 简化可直接 architect
        else:
            self._advance_stage("architect")            # T0.8
        return self._build_action()

    def _inject_supplement(self, gap: dict, content: str, source: str,
                           source_tier: str | None, confidence: str) -> None:
        """将细化产出注入 DesignDoc.supplements + 标记 ProgressTree 节点 stable。"""
        if self._design_doc is not None:
            self._design_doc.supplements[gap["id"]] = Supplement(
                gap_id=gap["id"], design_section_ref=gap["design_section_ref"],
                content=content, source=source, source_tier=source_tier,
                confidence=confidence, created_at=_now_iso())
            self._state.design_supplements_json = json.dumps(
                {k: asdict(v) for k, v in self._design_doc.supplements.items()})
        if self._progress_tree:
            node = self._progress_tree.find_by_design_section(gap["design_section_ref"])
            if node:
                node.design_status = "stable"

    def _after_architect(self) -> dict:
        # T2: architect → developer
        if self._state.batch_plan:
            batches = self._state.batch_plan
            # BatchState 统一构造 (双模式, 首轮建; plan_refine 轮保留游标只更新 batch_plan) — 见 B1.1a
            if self._batch_state is None:
                self._batch_state = (
                    BatchState.from_design_doc(self._design_doc, batches)
                    if self._design_doc
                    else BatchState.from_batch_plan(batches)
                )
            else:
                self._batch_state.batch_plan = batches
                self._batch_state.total_batches = len(batches)
            self._plan = _tasks_from_batch_plan(
                batches, self._state.requirement, self._design_doc)

            # 自动判定验证层深度
            if self._verification_layers is None:
                self._verification_layers = _determine_verification_layers(
                    self._design_doc, batches)

            # ProgressTree: 初始构建或增量同步
            if self._progress_tree is None:
                if self._design_doc:
                    self._progress_tree = ProgressTree.from_design_doc(self._design_doc)
                else:
                    self._progress_tree = ProgressTree.from_batch_plan(
                        batches, self._state.requirement)
            elif self._state.plan_refine_count > 0:
                # plan_refine 回路：增量同步。设计文档可能更新 → 重新判定
                self._verification_layers = _determine_verification_layers(
                    self._design_doc, batches)
                if self._design_doc:
                    sync_result = self._progress_tree.sync_from_design_doc(self._design_doc)
                else:
                    sync_result = self._progress_tree.sync_from_batch_plan(batches)
                self._state.round_history[-1]["progress_sync"] = asdict(sync_result)

        self._advance_stage("developer")
        return self._build_action()

    def _after_developer(self) -> dict:
        # T3: developer → critic（同一组件内继续下一个 batch 或进入 critic）
        self._batch_state.current_batch_idx += 1

        # ProgressTree: 更新当前组件的 task 进度
        if self._progress_tree:
            comp = self._batch_state.current_component()
            node = self._progress_tree.find_by_design_section(comp.design_section)
            if node:
                node.done_tasks += len(self._batch_state.batches_for(comp)[self._batch_state.current_batch_idx - 1].get("tasks", []))
                node.current_task = None
                self._progress_tree.recalculate_parents(node.id)

        current_component = self._batch_state.current_component()
        batches_for_component = self._batch_state.batches_for(current_component)
        if self._batch_state.current_batch_idx < len(batches_for_component):
            # 还有 batch 未完成 → 继续 developer
            # ProgressTree: 设置下一个 task
            if self._progress_tree:
                node = self._progress_tree.find_by_design_section(current_component.design_section)
                if node:
                    next_batch = batches_for_component[self._batch_state.current_batch_idx]
                    node.current_task = next_batch["tasks"][0]["description"] if next_batch.get("tasks") else None
            return self._build_action()
        # 组件所有 batch 完成 → critic
        self._advance_stage("critic")
        return self._build_action()

    def _after_critic(self, result: dict) -> dict:
        verdict = result.get("verdict", "")
        update_majors_count(self._state, verdict)

        # ProgressTree: 更新当前组件的 Gate 计数
        if self._progress_tree:
            comp = self._batch_state.current_component()
            node = self._progress_tree.find_by_design_section(comp.design_section)
            if node:
                node.gate_run_count += 1
                if verdict == "APPROVE":
                    node.gate_pass_count += 1

        if verdict == "MAJOR":
            # T4/T5: MAJOR 回路
            decision = self._router.next(
                "critic", "MAJOR",
                self._state.majors_in_a_row, self._state.total_majors)
            if decision.should_stop:
                return ActionDone(verdict="HARD_LIMIT",
                                  reason=decision.stop_reason).to_dict()
            # T4: → developer (修复当前 batch)
            self._advance_stage("developer")
            return self._build_action(feedback=json.dumps(result.get("findings", [])))

        elif verdict == "APPROVE":
            # T6/T7: 判定组件是否所有 batch 完成
            current_component = self._batch_state.current_component()
            if self._batch_state.has_more_batches_for(current_component):
                # T6: 组件还有 batch 未做 → developer（下一个 batch）
                self._advance_stage("developer")
                return self._build_action()
            # T7: 组件所有 batch 完成 + 全部 APPROVE → component_verifier
            self._advance_stage("component_verifier")
            return self._build_action()

    def _after_component_verifier(self, result: dict) -> dict:
        missing = result.get("missing_count", 0)
        diverged = result.get("diverged_count", 0)

        # ProgressTree: 更新组件验证状态
        if self._progress_tree:
            comp = self._batch_state.current_component()
            node = self._progress_tree.find_by_design_section(comp.design_section)
            if node:
                node.verifier_status = "failed" if (missing > 0 or diverged > 0) else "pass"
                node.verifier_missing = missing
                node.verifier_diverged = diverged
                self._progress_tree.recalculate_parents(node.id)

        if missing > 0 or diverged > 0:
            # T9/T10: 设计覆盖有缺口（missing/diverged 为 B6.6a Sonnet 复核后的 final_status 计数）
            self._state.audit_findings = result.get("coverage_map", [])
            return self._handle_plan_refine("component_verifier")

        # T8/T11: 所有设计条目已实现
        self._batch_state.advance_component()
        if self._batch_state.has_more_components_in_plate():
            # T8: 板块内还有下一个组件 → developer
            self._advance_stage("developer")
            return self._build_action()
        # T11: 板块所有组件完成 → 根据验证层深度裁剪
        if self._verification_layers == VerificationLayers.LEAF:
            # LEAF: 单组件，跳过 plate_deep_audit → 直接 system_deep_audit
            self._advance_stage("system_deep_audit")
        else:
            # PLATE/FULL: → plate_deep_audit
            self._advance_stage("plate_deep_audit")
        return self._build_action()

    def _after_plate_deep_audit(self, result: dict) -> dict:
        p0 = result.get("p0_count", 0)
        p1 = result.get("p1_count", 0)
        p1_threshold = self._get_p1_threshold()

        # ProgressTree: 更新板块内所有组件的 deep_audit 状态
        if self._progress_tree:
            current_plate = self._batch_state.current_plate()
            for comp in current_plate.components:
                node = self._progress_tree.find_by_design_section(comp.design_section)
                if node:
                    node.deep_audit_status = "failed" if (p0 > 0 or p1 > p1_threshold) else "pass"
                    node.deep_audit_p0 = p0
                    node.deep_audit_p1 = p1
                    node.deep_audit_p2 = result.get("p2_count", 0)
            self._progress_tree.recalculate_parents(f"sys/{current_plate.id}")

        if p0 > 0 or p1 > p1_threshold:
            # T13/T14: 板块级质量问题
            self._state.audit_findings = result.get("findings", [])
            return self._handle_plan_refine("plate_deep_audit")

        # 通过
        self._batch_state.advance_plate()
        if self._batch_state.has_more_plates():
            # T15: 还有下一个板块 → developer
            self._advance_stage("developer")
            return self._build_action()

        # T12: 所有板块完成 → 根据验证层深度裁剪
        if self._verification_layers == VerificationLayers.PLATE:
            # PLATE: 单板块，跳过 system_verifier → 直接 system_deep_audit
            self._advance_stage("system_deep_audit")
        else:
            # FULL: → system_verifier
            self._advance_stage("system_verifier")
        action = self._build_action()
        self._display_progress()  # 进入系统级阶段前展示进度
        return action

    def _after_system_verifier(self, result: dict) -> dict:
        self._state.coverage_map = result.get("full_coverage_map", [])
        missing = result.get("missing_count", 0)
        diverged = result.get("diverged_count", 0)

        if missing > 0 or diverged > 0:
            # T17/T17b: 全量设计覆盖有缺口 → architect（复核后 final_status 计数）
            self._state.audit_findings = self._state.coverage_map
            return self._handle_plan_refine("system_verifier")

        # T16: 所有需求已覆盖 → system_deep_audit
        self._advance_stage("system_deep_audit")
        action = self._build_action()
        # 自动展示进度树（不阻塞）
        self._display_progress()
        return action

    def _after_system_deep_audit(self, result: dict) -> dict:
        p0 = result.get("p0_count", 0)
        p1 = result.get("p1_count", 0)
        p1_threshold = self._get_p1_threshold()

        # 设计文档同步
        if result.get("design_docs_stale"):
            self._state.critic_feedback = (
                (self._state.critic_feedback or "") + "\n"
                + "[Design Doc Sync] " + result.get("design_doc_suggestions", ""))

        if p0 > 0 or p1 > p1_threshold:
            # T19/T20: 系统级质量问题
            self._state.audit_findings = result.get("findings", [])
            return self._handle_plan_refine("system_deep_audit")

        # T18: 全部通过 → 收敛判定
        self._write_audit_history(p0, p1, result.get("p2_count", 0), False)

        # 自动展示进度树（不阻塞）
        self._display_progress()

        design_coverage_ok = (
            result.get("missing_count", 0) == 0
            and result.get("diverged_count", 0) == 0
        )
        return self._convergence_check(
            design_coverage_ok=design_coverage_ok,
            system_deep_audit_ok=True)
```

### C.5.4 Plan-Refine 回路（T9/T13/T17/T19，分源+全局计数 DS-8）

```python
    # source ∈ {"component_verifier","plate_deep_audit","system_verifier","system_deep_audit"}
    def _handle_plan_refine(self, source: str) -> dict:
        MAX_PER_SOURCE, MAX_GLOBAL = 2, 4
        src_count = self._state.plan_refine_by_source.get(source, 0)
        # 放行谓词(与 StageRouter.refine_allowed 一致)
        if src_count >= MAX_PER_SOURCE or self._state.plan_refine_count >= MAX_GLOBAL:
            self._save_checkpoint()
            reason = (f"REFINE_LIMIT: {source} 分源 {src_count}/{MAX_PER_SOURCE} 未解决"
                      if src_count >= MAX_PER_SOURCE
                      else f"REFINE_LIMIT: 全局 {self._state.plan_refine_count}/{MAX_GLOBAL}")
            return ActionDone(verdict="REFINE_LIMIT", reason=reason).to_dict()
        # 计数递增(分源 + 全局, 单调不重置)
        self._state.plan_refine_by_source[source] = src_count + 1
        self._state.plan_refine_count += 1
        # 归一构造 RefineRequest(见 B6.10) 并持久化 → 下一 architect tick 注入 feedback
        self._state.refine_request_json = self._build_refine_request(source).to_json()
        clear_stage_fields(self._state, self._state.stage)
        self._advance_stage("architect")
        return self._build_action()   # _build_action 反序列化 refine_request_json → feedback.PLAN_REFINE
```

> `_build_refine_request(source)`：按 B6.10 映射表把该源的 coverage_map(MISSING/DIVERGED) 或去重后 audit findings(P0全部+P1) 归一为 `RefineRequest`。`_build_action` 检测 `refine_request_json` 非空 → 注入 `feedback={"mode":"PLAN_REFINE","refine_request":...}`，architect 消费后 orchestrator 清空 `refine_request_json`。

### C.5.5 收敛判定

```python
    def _convergence_check(self, design_coverage_ok: bool = False,
                           system_deep_audit_ok: bool = False) -> dict:
        verdict = self._judge.evaluate(
            self._state.round_history,
            design_coverage_ok=design_coverage_ok,
            system_deep_audit_ok=system_deep_audit_ok)

        if verdict.should_stop:
            level_name = ["CONTINUE", "GOAL_ACHIEVED", "STAGNANT", "QUALITY", "HARD_LIMIT"][verdict.level]
            self._save_checkpoint()
            return ActionDone(
                verdict=level_name, reason=verdict.reason,
                verdict_level=verdict.level).to_dict()

        self._save_checkpoint()
        return ActionDone(
            verdict="UNEXPECTED",
            reason="ConvergenceJudge returned CONTINUE after full cycle").to_dict()

- MUST NOT skip stages (Python validates stage field)
- MUST NOT declare "done" without Python's {"action":"done"}
- Each stage result MUST include the "stage" field matching the current action
- critic ONLY reviews the diff — do NOT judge requirement satisfaction
- component_verifier checks ONE component's design coverage
- system_verifier checks ALL design items' coverage ONCE (at system exit gate)
```
