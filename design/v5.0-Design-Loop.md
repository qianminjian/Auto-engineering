## A.0a 背景

**本项目专注 Loop Engineering**——提供 Claude Code Plugin 形态的 Stage-sequenced Agent loop 调度脚手架。用户在 Claude Code 会话中输入 `/dev-loop "实现登录功能"`，插件调度 Python Loop Engine 在子进程中运行 architect → developer → critic 三阶段 Agent 循环，产出代码变更、测试和审查结论。

**Init Engineering 是独立项目，不在本项目范围内**。Init 项目负责项目脚手架初始化 (5 阶段流水线 + 多层模板组合)，loop 项目只消费 Init 产出的结构化产物。**两者的衔接通过 Init-Loop 接口契约定义** (见 IL.1-IL.3)，本项目负责定义契约的 Loop 侧接口，Init 项目按契约实现 Init 侧。

**不是 standalone CLI 工具**。CLI (`ae dev-loop`) 是 Engine 的调试入口。正式使用形态是 Plugin——安装在 Claude Code 中，通过 slash command 触发，Engine 作为 subprocess 运行。

| 维度 | 说明 |
|------|------|
| **安装方式** | `uv tool install` 全局安装 (一次, 所有 Plugin 可调), Plugin 目录放源码 (只 marker) |
| **目录布局** | 跟 Node 生态一致: Plugin 目录内无 .venv/, Engine 通过 `uv tool install` 装到 `~/.local/bin/ae` 全局调用 |
| **交互方式** | Claude Code slash commands (`/dev-loop`, `/status`, `/checkpoint list` 等) |
| **Engine 运行** | Python subprocess — 全局 `ae` (来自 `~/.local/bin/ae`, `uv tool install` 全局), Bash 委托, 控制流在 Python |
| **数据持久化** | SQLite (`.ae-state/checkpoints.db`)，跨 Claude Code 会话保留 |
| **进程模型** | 每次 command 启动一个 Engine 进程 → 执行 → 输出 JSON → 退出。长 loop 由 Engine 内部 asyncio 驱动，不由 Claude Code 轮询 |

**业界参照**: Claude Code plugin convention (plugin.json + commands/ + hooks/ + skills/) × VS Code extension manifest (activation + contribution points) × LangGraph PregelLoop (tick/after_tick 控制流在 Python，不在宿主)

## A.1 双层架构

```
┌─────────────────────────────────────────────────────────┐
│ Claude Code Agent (宿主进程)                              │
│                                                           │
│  /dev-loop "需求" ──→ commands/dev-loop.md                │
│                        Bash 块: ae dev-loop ...    │
│                              │                            │
│  /status ───────────→ commands/status.md                  │
│                        Bash 块: ae status          │
│                              │                            │
│  hooks/ ── 事件驱动 ──→ pre-tool (sandbox 审计)           │
│                        post-edit (变更感知)               │
│                        stop (checkpoint 保存)             │
│                        session-start (环境预检)           │
│                              │                            │
│  skills/auto-engineering/SKILL.md                         │
│    定义 Agent 如何使用 ae 命令                            │
└──────────────────────────┬──────────────────────────────┘
                           │ subprocess (stdout JSON)
┌──────────────────────────▼──────────────────────────────┐
│ Python Loop Engine (子进程, 全局 `ae` 来自 `~/.local/bin/`)     │
│                                                           │
│  cli/dev_loop.py ──→ Orchestrator.run()                  │
│                        │                                  │
│          ┌─────────────┼─────────────┐                   │
│          ▼             ▼             ▼                   │
│     StageRouter   GuardrailChain  ConvergenceJudge       │
│          │             │             │                   │
│          ▼             ▼             ▼                   │
│     architect ──→ developer ──→ critic                   │
│     (Agent+LLM)   (Agent+LLM)   (Agent+LLM)             │
│          │             │             │                   │
│          └─────────────┼─────────────┘                   │
│                        ▼                                  │
│              7 Gates (asyncio.gather)                     │
│                        │                                  │
│                        ▼                                  │
│              SQLite Checkpoint                            │
└─────────────────────────────────────────────────────────┘
```

**两层之间的契约**：Plugin 层做 Bash 委托 + 参数转发 + 输出展示 + **轻量事件响应**（hook 中做 sandbox 审计、环境预检；不做 loop 控制、不做状态管理）。**所有控制流和状态在 Python Engine 内**。Plugin 不维护跨命令状态。这是借鉴 LangGraph 的设计——PregelLoop 控制流在 Python，LangGraph Platform 只是宿主。

**Plugin ↔ Engine 通信协议**：
- **下行**（Plugin → Engine）：CLI args + stdin (Task YAML 内容)
- **上行**（Engine → Plugin）：stdout 一行 JSON + exit code
- **stderr**：实时进度（B13.3 终端格式），Plugin 直接透传到 Claude Code 会话

## A.2 核心概念

| 概念 | 定义 | 所在层 | 类比 |
|------|------|--------|------|
| **Plugin** | Claude Code 安装单元 — plugin.json + commands/ + hooks/ + skills/ | Plugin 层 | VS Code Extension |
| **Command** | 一个 slash command 的 Markdown 定义，含 Bash 委托块 | Plugin 层 | VS Code Command Contribution |
| **Hook** | Claude Code 事件触发器，调 ae 命令或执行 shell 检查 | Plugin 层 | VS Code Activation Event |
| **Stage** | 三阶段流水线的一步 (architect/developer/critic) | Engine 层 | LangGraph Node |
| **Round** | 一个 Stage 的完整周期 = Guardrail → Agent → Gate → Judge → Checkpoint | Engine 层 | LangGraph Superstep |
| **Task** | 工作单元, role/description/expected_output/depends_on/file_targets | Engine 层 | AutoGen Task |
| **Guardrail** | 进程内确定性校验, 4 态 (pass/block/drop/retry), 按 timing+stage 过滤 | Engine 层 | CrewAI GuardrailResult |
| **Gate** | subprocess 质量检查, pass/fail, 7 道并行 | Engine 层 | CI Pipeline Stage |
| **StageRouter** | 决定下一 Stage, 综合 stage+verdict+MAJOR 计数 | Engine 层 | LangGraph conditional edge |
| **Agent→LoopState 隔离** | Agent 不写 LoopState; Agent 返回 TaskOutcome, Orchestrator 提取写入 | Engine 层 | Actor model |

```

3. 中断恢复
   用户: /checkpoint list

   用户: /dev-loop --resume a1b2c3...
   Agent: → ae dev-loop --resume a1b2c3...
```

## A.4 关键设计决策

| # | 决策 | 理由 | 业界参照 |
|---|------|------|---------|
| D1 | Plugin = 薄壳, Engine = 控制流 | Plugin 只做 Bash 委托, 所有 loop 逻辑在 Python | LangGraph PregelLoop (控制流在 Python, 不在宿主) |
| D2 | 全局 `ae` 命令 (uv tool install) + stdout JSON Plugin-Engine 唯一上行契约 | 简单、可测试、所有 Plugin 共享同一 Engine 进程, 无重复依赖 | Claude Code plugin convention + uv tool install |
| D3 | Stage-sequenced, Agent→LoopState 隔离 | Agent 不写共享状态, Orchestrator 控制写入 | LangGraph apply_writes + Actor model |
| D4 | Task 来源双模式 (外部 YAML / ArchitectAgent 生成) | 灵活适配有/无预定义 Task 的场景 | Devin plan-act-observe |
| D5 | StageRouter + MAJOR 计数分离 | 回路控制独立于状态机 | LangGraph recursion_limit |
| D6 | Gate 全并行 | 7 Gate 互不依赖, 串行浪费 | CI 并行 job |
| D7 | Checkpoint 在 Stage 推进后保存 | 避免 resume 重复执行已完成 Stage | LangGraph after_tick |
| D8 | Plugin 支持用户级全局安装 + 项目级安装双模式 | 用户级：`claude plugin install . --scope user`，所有项目共享。项目级：文件复制，项目隔离。目录布局匹配 Claude Code 标准 plugin convention（commands/hooks/skills 在 plugin root 同级，非 .claude-plugin/ 内部） | Claude Code plugin convention + VS Code extension manifest |

### PE.1a 多 Skill 拆分建议 (借鉴 project-engineering-init)

**当前设计**: 单 Skill `auto-engineering` (PE.4 SKILL.md).

**业界参照**: `project-engineering-init` 用数组结构列 6 个 Skill (`init` / `tdd` / `worktree` / `agent-dev` / `agent-critic` / `agent-orchestrator`). 用户通过 `/skill tdd` 等激活特定 Skill.

**多 Skill 拆分方案** (v5.1 候选, v5.0 暂保留单 Skill):

| Skill 名 | 触发场景 | 关联 Agent |
|----------|---------|-----------|
| `auto-engineering` | 通用: `/dev-loop` 编排 | 全部 3 Stage |
| `ae-architect` | 仅分析需求 | ArchitectAgent |
| `ae-developer` | 仅 TDD 编码 | DeveloperAgent |
| `ae-critic` | 仅代码审查 | CriticAgent |
| `ae-orchestrator` | Loop 控制流 | Orchestrator |

**plugin.json 数组写法** (借鉴 project-engineering-init):
```json
"skills": [
  { "path": "skills/auto-engineering", "name": "auto-engineering" },
  { "path": "skills/ae-architect", "name": "ae-architect" },
  { "path": "skills/ae-developer", "name": "ae-developer" },
  { "path": "skills/ae-critic", "name": "ae-critic" }
]
```

**v5.0 决策**: **保留单 Skill** (`skills/auto-engineering/SKILL.md`). 多 Skill 拆分排期 v5.1, 避免 v5.0 范围爆炸. 当用户在单 Stage 卡点 (如想直接调 Critic) 时, 通过 `/project-agent critic <指令>` 满足, 不需多 Skill.

### PE.1b Hook 可执行权限约束

5 个 hook 文件 (`hooks/*.sh`) 创建后必须 `chmod +x`. v5.0 设计 B17.1 文件清单未提及, 现补充:

```bash

### PE.2b Command 示例: dev-loop.md 完整定义

```markdown
# /dev-loop — 启动 Loop Engineering 开发循环

## Description
输入自然语言需求, 自动完成 architect(分析)→developer(实现)→critic(审查) 三阶段循环。
支持 Task YAML 预定义、中断恢复、Gate 跳过等模式。

## Usage
/dev-loop "<需求描述>" [--tasks FILE] [--max-rounds N] [--no-gates] [--no-semantic] [--resume ID]

## Arguments
| 参数 | 类型 | 必需 | 默认 | 说明 |
|------|------|------|------|------|
| `requirement` | string | 是 | — | 自然语言需求描述, 1..4096 chars |

## Flags
| Flag | 类型 | 默认 | 说明 |
|------|------|------|------|
| `--tasks` | path | — | Task YAML 文件路径, 预定义 Task 列表 |
| `--max-rounds` | int | 10 | 最大轮次 (硬上限) |
| `--no-gates` | bool | false | 跳过所有 Gate, 降为 3 级收敛 |
| `--no-semantic` | bool | false | 禁用 LLM 语义评估 |
| `--resume` | string | — | 从指定 thread_id 的 checkpoint 恢复 |
| `--project-root` | path | cwd | 项目根目录 |

## Execution
```bash
# Claude Code 将用户输入解析后传入: $1=requirement, 后续为 --flag value 对
# 实现时用 while + case 解析所有参数 (处理 boolean flag 位置不定问题)
ARGS=("$1")
shift
while [[ $# -gt 0 ]]; do
  case "$1" in
    --tasks)        ARGS+=("--tasks" "$2"); shift ;;
    --max-rounds)   ARGS+=("--max-rounds" "$2"); shift ;;
    --no-gates)     ARGS+=("--no-gates") ;;
    --no-semantic)  ARGS+=("--no-semantic") ;;
    --resume)       ARGS+=("--resume" "$2"); shift ;;
    --project-root) ARGS+=("--project-root" "$2"); shift ;;
  esac
  shift
done
ae dev-loop "${ARGS[@]}"
```

### PE.3 Hook 规范 (hooks/*.sh)

Claude Code 在特定事件时执行对应 hook 脚本。脚本输出 JSON `{"decision":"block"|"allow"}` 控制流程。

| 文件名 | 触发事件 | 时机 | 用途 |
|--------|---------|------|------|
| `session-start.sh` | 会话启动 | Claude Code 启动后 | 环境预检 (uv/Python/git/ANTHROPIC_API_KEY) |
| `pre-tool.sh` | PreToolUse | 每次工具调用前 | sandbox 审计: 拦截危险 Bash/文件逃逸 |
| `post-edit.sh` | PostToolUse | Write/Edit 后 | 检测新增文件 → 自动跑 lint + type_check |
| `stop.sh` | 会话结束 | Claude Code 关闭前 | 检查 `.ae-state/checkpoints.db` 最新 checkpoint status; 若 status="running" → 调 `ae checkpoint save` 保存为 "interrupted" |
| `on-pr.sh` | PR 创建 | gh pr create 后 | 收集本轮 Gate 结果 → 附加到 PR body |

**Hook 通用模板**:
```bash
#!/bin/bash
# hooks/<name>.sh — <用途>
INPUT="$1"  # Claude Code 传入的 JSON
# 检查逻辑
if <条件>; then
  echo '{"decision":"block","reason":"<原因>"}'
  exit 0
fi
echo '{"decision":"allow"}'
```

**pre-tool.sh 核心逻辑**（两层拦截, v5.1 扩展）:

**Layer 1 — 13 条危险模式 denylist (复用 Engine B12.2):**
- 检测 Bash 命令是否命中 `rm -rf /`/`mkfs`/`dd if=`/fork bomb/`shutdown`/`reboot`/`halt`/`chmod -R 777 /`/`curl \| bash`/`wget \| bash`/系统文件覆写 等 13 条正则
- 检测文件写入路径是否在 project_root 内 (realpath 双侧归一化 + lexical fallback)
- 命中 → `{"decision":"block"}`, 否则 → `{"decision":"allow"}`

**Layer 2 — 7 条 Agent Working Agreements (v5.1 新增, 借鉴业界 pre-commit "No Escape" 哲学):**
- 拦截 `git commit --no-verify` / `git commit -n`
- 拦截 `HUSKY=0` (禁用 husky hooks)
- 拦截 `SKIP=pre-commit` (跳过 pre-commit)
- 拦截 `git push --force` / `git push -f`
- 拦截 `--no-gpg-sign`
- 命中 → `{"decision":"block","reason":"Agent Working Agreement violation (No Escape): ..."}`
- 三层防御架构: hook (本地毫秒级) → CI Gate (分钟级) → Review (人工) — 与业界 pre-commit 2024-2025 对齐

**post-edit.sh 核心逻辑**:
- 检测本次 Edit/Write 的新增/修改文件
- 若文件在 `src/` 或 `lib/` 下 → 调 `ae gate-check --quick` (仅 lint+type_check, 跳过 test+build)
- 结果写入 `_scratch/hook-post-edit.log`

## Init-Loop 接口契约（Loop 侧接口定义）

> **背景**：Init Engineering 是独立项目，负责项目脚手架初始化（5 阶段流水线 detect→prompt→render→tasks→finalize）。本节定义 **Loop 侧**接口契约——Loop 侧如何消费 Init 产出的结构化产物。Init 侧按本契约实现 Init 项目自身。

### IL.1 契约总览

| 方向 | 数据 | 位置 | 说明 |
|------|------|------|------|
| **Init → Loop** | `.ae-state/init-manifest.json` | 项目根 | Init 完成时写入, Loop 启动时读取 |
| **Loop → Init** | (无) | — | Loop 不反向调用 Init (单向依赖, Init 是上游) |
| **共享契约** | `.ae-state/checkpoints.db` | 项目根 | Loop 写入, Init 可读 (用于生成 Init-aware Task YAML) |

**依赖方向**：Init 完成后用户才能运行 Loop。Loop 不应在 Init 未完成的项目上运行 (可检测 `.ae-state/init-manifest.json` 是否存在, 不存在则提示先跑 Init)。

### IL.2 `.ae-state/init-manifest.json` Schema

Init 项目在脚手架完成后必须写入此文件。Loop 项目读取此文件获取项目上下文。

```json
{
  "schema_version": "1.0",
  "project_type": "app-service",
  "language": "python",
  "framework": "fastapi",
  "created_at": "2026-06-30T10:00:00Z",
  "init_version": "1.0.0",
  "structure": {
    "source_root": "src/",
    "test_root": "tests/",
    "config_files": ["pyproject.toml", ".env.example"],
    "entry_point": "src/main.py"
  },
  "conventions": {
    "package_manager": "uv",
    "linter": "ruff",
    "type_checker": "pyright",
    "test_runner": "pytest",
    "build_cmd": "uv build",
    "test_cmd": "pytest tests/ -v",
    "lint_cmd": "ruff check src/",
    "type_check_cmd": "pyright src/"
  },
  "templates_applied": [
    "app-service/python/_shared",
    "app-service/python/fastapi"
  ],
  "answers": {
    "project_name": "my-app",
    "python_version": "3.12",
    "include_docker": true,
    "include_ci": true
  }
}
```

**字段约束** (Loop 侧读取契约):

| 字段 | 类型 | 必需 | Loop 用途 |
|------|------|------|----------|
| `schema_version` | string | 是 | Loop 检测 Init 兼容性 (与 Loop 期望版本对齐) |
| `project_type` | enum | 是 | Loop 启动时类型校验 |
| `language` | enum | 是 | Loop 决定使用哪个 linter/type_checker/test_runner |
| `conventions.linter` | string | 是 | Loop 配置 lint Gate |
| `conventions.type_checker` | string | 是 | Loop 配置 type_check Gate |
| `conventions.test_runner` | string | 是 | Loop 配置 test Gate |
| `conventions.build_cmd` | string | 否 | Loop 配置 build Gate |
| `conventions.test_cmd` | string | 是 | Developer Agent 跑测试的命令 |
| `structure.source_root` | string | 是 | Loop 限定文件沙箱根 |
| `structure.test_root` | string | 是 | Loop 限定测试目录 |

**Loop 侧读取失败行为** (与 IL.4 一致):
- 文件不存在 → `ae doctor` 提示"未找到 init-manifest.json, 请先运行 Init"
- schema_version < Loop 最低支持版本 → `ae doctor` 报错退出, 提示重新 Init
- schema_version > Loop 最高支持版本 → WARN + 提示升级 Loop, 仍尝试运行 (forward-compat)
- 必需字段缺失 → `ae doctor` 报错退出, 列出缺失字段名

**enum 合法值** (Init 项目必须从下列值中选):
- `project_type`: `app-service` | `library` | `cli-tool` | `skill` | `hook` | `mcp-server` | `spec-doc` | `monorepo`
- `language`: `python` | `typescript` | `go` | `rust` | `bash`
- Loop 支持的 Gate 工具映射 (Init 写入 conventions 时可参考):
  - python: linter=ruff, type_checker=pyright, test_runner=pytest
  - typescript: linter=eslint, type_checker=tsc, test_runner=vitest
  - go: linter=golangci-lint, type_checker=go vet, test_runner=go test
  - rust: linter=clippy, type_checker=cargo check, test_runner=cargo test
  - bash: linter=shellcheck, type_checker=bash -n, test_runner=bats
- 若 Init 写入未在上述列表的 `language`/`project_type`, Loop 启动时报错 (IL.4)

### IL.3 Task YAML 共享契约 (Init 生成, Loop 消费)

Init 项目产出的 `tasks.yml` (Loop 的预定义 Task 输入) 必须遵循 B1.3 Task 数据模型约束, 并包含以下 Init-specific 字段约定:

```yaml
# tasks.yml 格式 (Init 项目生成, Loop 项目读取)
tasks:
  - id: "init-data-models"
    role: "developer"
    description: "Create SQLAlchemy data models for User, Order, Product"
    expected_output: "models.py with 3 classes"
    depends_on: []
    file_targets: ["src/models.py"]
    # Init 特有字段 (Loop 可读可忽略)
    init_metadata:
      template_source: "app-service/python/fastapi/models.py.jinja"
      generated_by: "init-engineering 1.0.0"
```

**Init 特有字段**:
- `init_metadata.template_source` (string, optional): 模板来源 (追溯用, Loop 忽略)
- `init_metadata.generated_by` (string, optional): 生成 Init 版本 (追溯用)

**约束**: Init 生成的 tasks.yml 字段名/结构必须符合 B1.3 Task 数据模型, `init_metadata` 作为扩展字段, Loop 侧解析时**静默忽略未知字段** (forward-compatibility)。

### IL.4 边界契约：Loop 不依赖 Init

| 场景 | Loop 行为 |
|------|----------|
| 用户在未 Init 的项目上跑 `/dev-loop` | `ae doctor` 报错, 提示运行 Init (但不自动调用 Init) |
| Init manifest 中 `language` 是 Loop 不支持的语言 | Loop 启动时报错, 列出支持的语言, 不静默 fallback |
| Init manifest schema_version > Loop 支持的最高版本 | WARN + 提示升级 Loop, 但仍尝试运行 (forward-compat) |
| Init manifest schema_version < Loop 支持的最低版本 | 拒绝运行, 提示重新 Init |

**核心原则**：Loop 是 Init 的消费者, 不是调用者。Loop 启动时检查 Init 产物存在性和兼容性, 不主动触发 Init, 不假设 Init 已完成。

### IL.5 未来 Init 项目必须实现的接口 (本项目视角)

为支持 Loop 正常运行, Init 项目必须提供:
1. **写** `.ae-state/init-manifest.json` (IL.2 schema), 脚手架完成后立即写入
2. **写** `tasks.yml` (IL.3 format), 当用户选择 "预定义 Task" 模式时
3. **创建** `.ae-state/` 目录 (Loop 写入 SQLite 所需)
4. **不修改** `.ae-state/checkpoints.db` (Loop 独占)

Init 项目的具体实现 (5 阶段流水线 + 模板加载 + Jinja2 渲染等) 在 Init Engineering 项目设计中, 本项目不约束。

### IL.6 验收条件 (Init-Loop 接口)

| ID | 验收条件 |
|----|---------|
| IL-AC-01 | When Loop 启动且 `.ae-state/init-manifest.json` 不存在, `ae doctor` shall 报错并提示运行 Init |
| IL-AC-02 | When Loop 启动且 init-manifest.json 存在, Loop shall 读取 conventions 并配置对应 Gate |
| IL-AC-03 | When Init 写入 tasks.yml 含 B1.3 Task 数据模型未定义的未知字段 (排除 init_metadata 已知字段), Loop shall 静默忽略 |
| IL-AC-04 | When Init manifest schema_version < Loop 支持的最低版本, Loop shall 拒绝运行 |
| IL-AC-05 | When Loop 完成, Loop shall 不修改 `.ae-state/init-manifest.json` |

### B2.12a run_round 内部流程

```
run_round(stage, tasks, executor, ctx, cancellation, round_id, gates, project_root, contracts=None) → RoundResult:

  ┌─ 1. 初始化 ───────────────────────────────────────────┐
  │ result = RoundResult(round_id=round_id, stage=stage)    │
  │ result.started_at = time.monotonic()                    │
  │ if not tasks: → 跳到步骤 5 (只跑 Gate)                  │
  └────────────────────────────────────────────────────────┘

  ┌─ 2. 任务调度 (DAG 拓扑排序) ─────────────────────────┐
  │ # 对 tasks 按 depends_on 做拓扑排序, 分层并行执行       │
  │ layers = _topological_layers(tasks)                    │
  │ # layers[0] = 无依赖的 task, layers[1] = 依赖 L0 的...  │
  └────────────────────────────────────────────────────────┘

  ┌─ 3. 逐层执行 ────────────────────────────────────────┐
  │ for layer in layers:                                   │
  │   # 同层内并行, 层间串行 (依赖保证)                     │
  │   coros = [_execute_one(t, ctx, executor, cancellation) │
  │            for t in layer]                              │
  │   outcomes = await asyncio.gather(*coros,               │
  │                return_exceptions=True)                  │
  │   for item in outcomes:                                │
  │     if isinstance(item, BaseException): continue        │
  │     result.outcomes.append(item)                        │
  │   # 提取当前层非异常的 outcomes                          │
  │   layer_outcomes = [o for o in outcomes                 │
  │       if isinstance(o, TaskOutcome)]                    │
  │   # 检查是否需要提前终止 (全部 failed)                   │
  │   if layer_outcomes and all(o.status == "failed"        │
  │       for o in layer_outcomes):                         │
  │     break  # 后续层不执行                               │
  └────────────────────────────────────────────────────────┘

  ┌─ 4. 单 Task 执行 (_execute_one) ─────────────────────┐
  │ # ctx 可为 None (B2.12 签名); None 时用空默认值          │
  │ # 为每个 task 构造独立 TaskContext (避免共享 ctx.task)   │
  │ per_task_ctx = TaskContext(                             │
  │   requirement=ctx.requirement if ctx else "",           │
  │   plan=ctx.plan if ctx else None,                       │
  │   critic_feedback=ctx.critic_feedback if ctx else None, │
  │   task=task,  # 当前 task (不是共享的)                   │
  │   project_root=ctx.project_root if ctx else Path.cwd()  │
  │ )                                                       │
  │ start = time.monotonic()                               │
  │ try:                                                    │
  │   cancellation?.check()                                 │
  │   outcome = await executor(task, per_task_ctx)          │
  │   outcome.duration = time.monotonic() - start          │
  │   return outcome                                        │
  │ except AEError as e:                                    │
  │   if e.code == ERR_TASK_CANCELLED:                      │
  │     return TaskOutcome(task_id=task.id,                 │
  │       task_role=task.role, status="cancelled")          │
  │   raise                                                 │
  │ except asyncio.CancelledError:                          │
  │   return TaskOutcome(task_id=task.id,                   │
  │     task_role=task.role, status="cancelled")            │
  │ except Exception as e:                                  │
  │   return TaskOutcome(task_id=task.id,                   │
  │     task_role=task.role, status="failed", error=str(e)) │
  └────────────────────────────────────────────────────────┘

  ┌─ 5. Gate 执行 (Agent 完成后, 按 stage 过滤) ─────────┐
  │ if gates and project_root is not None:                  │
  │   result.gate_results = await run_gates(                │
  │     gates, project_root, stage, contracts=contracts)     │
  │ # 只运行 applies_to_stages 包含当前 stage 的 Gate       │
  │ # architect: safety+lint+type_check                     │
  │ # developer: 全部 7 道                                  │
  │ # critic: safety+lint+type_check+contract+test          │
  └────────────────────────────────────────────────────────┘

  ┌─ 6. 构造 RoundHistory ───────────────────────────────┐
  │ # git_diff_numstat 返回 (added, removed, file_count)      │
  │ lines_added, lines_removed, file_count =                │
  │   await asyncio.to_thread(git_diff_numstat, project_root) │
  │ result.history_entry = RoundHistory(                    │
  │   round_id=round_id, stage=stage,                      │
  │   files_changed=file_count,  # 实际修改文件数 (非 task 数) │
  │   lines_added=lines_added,                             │
  │   lines_removed=lines_removed,                         │
  │   gate_results=dict(result.gate_results),               │
  │   task_ids=[t.id for t in tasks],                      │
  │   task_outcomes={o.task_id:o.status                    │
  │     for o in result.outcomes}                           │
  │ )                                                       │
  │ result.finished_at = time.monotonic()                   │

### B2.12b DAG 拓扑分层

```
_topological_layers(tasks) → list[list[Task]]:
    # 输入: Task 列表 (含 depends_on)
    # 输出: 分层列表, layers[0] 无依赖, layers[i] 仅依赖 layers[0..i-1]
    # 算法: Kahn 算法变体 — BFS 分层
    in_degree = {t.id: len(t.depends_on) for t in tasks}
    children = {t.id: [] for t in tasks}
    for t in tasks:
        for dep_id in t.depends_on:
            children[dep_id].append(t.id)

    queue = [t.id for t in tasks if in_degree[t.id] == 0]
    task_map = {t.id: t for t in tasks}
    layers = []
    while queue:
        layers.append([task_map[tid] for tid in queue])
        next_queue = []
        for tid in queue:
            for child_id in children[tid]:
                in_degree[child_id] -= 1
                if in_degree[child_id] == 0:
                    next_queue.append(child_id)
        queue = next_queue

    # 检查是否有环
    if sum(len(L) for L in layers) != len(tasks):
        raise ConflictError("DAG contains cycle", conflicts=[
            t.id for t in tasks
            if in_degree[t.id] > 0
        ])
    return layers
```

```
You are a Code Reviewer (Critic). Your role is to review code changes against
the original requirement and produce a verdict with detailed findings.

REVIEW DIMENSIONS (check all 5):
1. CORRECTNESS: Does the implementation fulfill the requirement? Are edge cases

2. TEST COVERAGE: Do tests cover the happy path AND edge cases? Are test

3. CODE QUALITY: Is the code clear, well-structured, and following project

4. INTERFACE CONTRACTS: If architect defined contracts, does the implementation

5. RUNTIME CORRECTNESS: Do all tests pass? Any security issues (SQL injection,

VERDICT RULES:
- APPROVE: 0 P0 findings AND <3 P1 findings
- MAJOR: ≥1 P0 finding OR ≥3 P1 findings

FINDING SEVERITY:
- P0 (BLOCKER): Security vulnerability, data loss, wrong behavior, contract
  violation, test failure. MUST be fixed before approval.
- P1 (IMPORTANT): Missing test coverage for key path, unclear error handling,
  code duplication, deviation from project conventions. SHOULD be fixed.
- P2 (NIT): Naming suggestions, minor style issues, optional improvements.

PRINCIPLES:
- FRESH CONTEXT: Review the code as-is. Do NOT assume the developer's intent.
  Judge only what the code does, not what it "was supposed to do."
- SPECIFIC: Every finding must include file path + line number (when possible)
  + concrete fix suggestion. No vague feedback like "improve error handling."
- HONEST: If unsure about severity, default to P2. Better to under-escalate
  than to cry wolf.
- ACTIONABLE: critic_feedback must be a structured summary that a developer
  can use as a checklist. Group by file, order by severity.

{
  "verdict": "APPROVE",

      "file": "src/service.py",

}

CONSTRAINTS:
- verdict must be exactly "APPROVE" or "MAJOR" (no other values)
- findings can be empty (if APPROVE with no issues found)
- Each finding must have file + issue + severity (line and suggestion are optional)
- critic_feedback must be non-empty when verdict is MAJOR
- critic_feedback should be structured as a checklist, not prose

### B4.4 Agent 工具调用循环

```
execute(task, ctx) → TaskOutcome:
  1. 构造 messages: [system_prompt, user_msg(task.description)]
  2. loop_count = 0
  3. while loop_count < max_tool_calls:
    3a. response = await llm.create_message(messages, tools, tool_choice="auto")
    3b. switch response.stop_reason:
        case "end_turn": break
        case "tool_use":
          for each tool_use in response.tool_use_blocks:
            tool = registry.get(tool_use.name)
            if tool is None:
              # 未知工具 → 返回 error tool_result (JSON: {"error":"unknown tool: <name>"})
              messages.append({"role":"user","content":[{"type":"tool_result","tool_use_id":tool_use.id,"content":'{"error":"unknown tool: '+tool_use.name+'"}'}]})
              continue
            if not authz_check(role, tool.name):
              # 未授权 → 返回 error tool_result (JSON: {"error":"tool <name> not authorized for <role>"})
              messages.append({"role":"user","content":[{"type":"tool_result","tool_use_id":tool_use.id,"content":'{"error":"tool '+tool.name+' not authorized for '+role+'"}'}]})
              continue
            try:
              result = tool.execute(tool_use.input)
              content = result.output or ""
            catch Exception as e:
              content = '{"error":"'+str(e)+'"}'
            messages.append({"role":"user","content":[{"type":"tool_result","tool_use_id":tool_use.id,"content":content}]})
        case "max_tokens": 标记 warning, break
    3c. loop_count += 1
  4. 双层解析 response.content:
     Layer1: Pydantic OutputModel.model_validate_json() → 成功返回
     Layer2: regex ```json fence 提取 → json.loads → validate → 成功返回
     都失败 → TaskOutcome(status="failed", error="parse_failed")
```

**LLM 重试** (BaseAgent 内部):
- 单次 `create_message` 超时/rate_limit/network_error → retry ≤3, exp backoff
- 参数: base_delay=1s, max_delay=30s, multiplier=2.0, jitter=±25%
- Orchestrator **不额外重试** LLM 调用

## B5. Guardrail 规格

### B5.1 5 个 Guardrail

| # | name | timing | applies_to_stages | 检查内容 | 失败动作 |
|---|------|--------|-------------------|---------|---------|
| G1 | **RequirementValid** | `"pre"` | `("architect",)` | state.requirement 非空, 1..4096, 非仅控制字符 | `block` |
| G2 | **PlanExists** | `"post"` | `("architect",)` | state.plan 非空 AND state.file_list 非空 AND len≥1 | `retry`(≤3) |
| G3 | **GitDiffExists** | `"post"` | `("developer",)` | `git diff HEAD~1..HEAD --numstat` 输出非空, 需 `project_root` 参数. 边界: 仓库无 HEAD~1 (首次 commit) → 改用 `git diff --cached --numstat` | `retry`(≤3) |
| G4 | **TestsPass** | `"post"` | `("developer",)` | state.test_results is not None AND state.test_results.failed==0 AND errors==0 AND (passed+failed+errors) > 0 | `retry`(≤3) |
| G5 | **GitClean** | `"post"` | `("developer",)` | `git status --porcelain` 输出为空, 需 `project_root` 参数 | `block` |

**`drop` 动作**: 4 态中保留给用户自定义 Guardrail。5 个内置 Guardrail 均不使用。扩展 API 允许注册自定义 Guardrail 返回 `drop` 来静默丢弃输出并重跑。

**GuardrailChain.default() 工厂方法 (v5.1 新增)**: `GuardrailChain.default()` 返回包含全部 5 个 Guardrail 的预配置链: `[RequirementValid(), PlanExists(), GitDiffExists(), TestsPass(), GitClean()]`。供 `_run_v2_orchestrator` 直接调用, 无需手动构造。

### B5.2 _handle_guardrail_result

```
_handle_guardrail_result(result, stage, state, retry_counters) → "continue"|"stop"|"retry":
    switch result.action:
        case "pass":  return "continue"
        case "block": return "stop"
        case "drop":
            if retry_counters.get(stage, 0) >= 3:
                return "stop"
            retry_counters[stage] = retry_counters.get(stage, 0) + 1
            _clear_stage_fields(state, stage)
            return "retry"
        case "retry":
            if retry_counters.get(stage, 0) >= 3:
                return "stop"  # 3 次重试耗尽 → stop (termination reason: guardrail exhaustion)
            retry_counters[stage] = retry_counters.get(stage, 0) + 1
            _clear_stage_fields(state, stage)  # 清除旧产出, 确保重跑干净
            return "retry"
```

### B6.1a ContractGate 验证算法

```
ContractGate.run(project_root, contracts=None) → GateVerdict:

  前提: Orchestrator 从 state.contracts 传入 contracts 参数
  若 contracts 为 None 或空 → return GateVerdict.passed("无契约定义, 跳过", self.name)

  对 contracts 中每条契约执行以下检查:

  1. 路由/端点存在性:
     若 contract.method + contract.path 已定义:
       在 project_root 下搜索对应路由注册代码
       搜索模式 (按语言):
         Python: @app.route("{path}", methods=["{method}"])
         TS/Express: app.{method}("{path}")
         Go/gin: router.{method}("{path}")
         Rust/actix: #[{method}("{path}")]
       未找到 → failed("端点 {method} {path} 未实现")

  2. 请求 Schema 匹配:
     若 contract.request 已定义:
       找到对应的请求处理函数 → 检查参数名是否匹配 contract.request 的 key
       至少 80% key 匹配 → pass; 否则 → failed("请求 schema 不匹配")

  3. 响应 Schema 匹配:
     若 contract.response 已定义:
       找到对应的响应构造代码 → 检查返回字段是否包含 contract.response 的 key
       至少 80% key 匹配 → pass; 否则 → failed("响应 schema 不匹配")

  4. 状态码实现:
     若 contract.status_codes 已定义:
       搜索对应处理函数中是否实现了这些状态码的返回
       检查 200/201 至少实现 1 个 → pass
       所有 status_codes 都未找到 → failed("状态码未实现")

  汇总: 任一条契约检查 failed → GateVerdict.failed(详情)
        全部 pass 或 skip → GateVerdict.passed(汇总)
```

**搜索范围**: `project_root/src/` (若存在), 否则 `project_root/`。仅搜索源码文件 (`.py/.ts/.js/.go/.rs`)。不搜索 `tests/` `node_modules/` `__pycache__/`。

**局限性 (明确不做)**: ContractGate 做静态文本匹配, 不做 AST 解析。运行时契约验证由集成测试覆盖。这是工程权衡: 静态匹配覆盖 80% 常见契约, AST 解析的复杂度不值得。

### B6.2 run_gates (全并行 + Stage 过滤)

```
run_gates(gates, project_root, stage, contracts=None) → dict[str, GateVerdict]:
    if not gates or project_root is None: return {}

    # 按 stage 过滤: 只运行 applies_to_stages 包含当前 stage 的 Gate
    applicable = [g for g in gates if stage in g.applies_to_stages]

    async def _run_one(g: Gate) → tuple[str, GateVerdict]:
        try:
            v = await asyncio.to_thread(g.run, project_root, contracts)
        except Exception as e:
            v = GateVerdict.failed(str(e), gate_name=g.name)
        return (g.name, v)

    coros = [_run_one(g) for g in applicable]
    gathered = await asyncio.gather(*coros, return_exceptions=True)
    results = {}
    for item in gathered:
        if isinstance(item, BaseException): continue
        name, verdict = item
        results[name] = verdict
    return results
    # 全并行, 按 stage 过滤. 如 architect 只跑 safety+lint+type_check
```

**Gate 基类字段** (与 B2.9 一致):
```
Gate:
    name: str (类属性)
    applies_to_stages: tuple[str, ...]  # 如 ("architect","developer","critic")
    run(project_root: Path, contracts: dict | None = None) → GateVerdict
```

### B6.3 Gate 失败恢复策略

Gate 失败时 **不触发 Stage 重试** (与 Guardrail retry 不同)。Gate 是质量检查点, 失败意味着代码质量不达标。

**重要约束**: **只有 critic stage 的 Gate 失败才会触发 ConvergenceJudge Level 3 QUALITY stop** (见 B2.4 Judge 算法, 仅检查 `history[-1].stage == "critic"`). developer/architect stage 的 Gate 失败只写入 `gate_results` 供后续 round 参考, Judge 不据此 stop — 由 Orchestrator 继续推进到下一 stage (developer → critic).

| Gate 失败场景 | 收敛判定 | 恢复路径 |
|-------------|---------|---------|
| safety/lint/type_check fail (architect) | Judge 跳过 (非 critic) | Orchestrator 继续 → developer, 由 developer stage 的 Gate 重新发现 |
| safety/lint/type_check fail (developer) | Judge 跳过 (非 critic) | Orchestrator 继续 → critic; critic Stage 内由 Gate + semantic 联合判定 |
| contract/test fail (developer) | Judge 跳过 (非 critic) | 同上 |
| **safety/lint/type_check fail (critic)** | **Judge Level 3 QUALITY stop** | 用户修复 → 重新 `/dev-loop` (因 verdict=APPROVE 失败, 无法 resume) |
| **contract/test fail (critic)** | **Judge Level 3 QUALITY stop** | 同上 |
| **build fail (developer)** | Judge 跳过 (非 critic) | Orchestrator 继续 → critic; 若 critic 也 build 失败 → QUALITY stop |
| Gate 工具缺失 | skip with WARN log | Gate 降级, 不减分 (见 B10.3 降级路径) |
| Gate 超时 | GateVerdict.failed | 增大 `AE_GATE_TIMEOUT` 或检查 CI 资源 |

**Gate 失败与 MAJOR 回路的关系**:
- Gate 失败在 critic stage → ConvergenceJudge Level 3 (QUALITY) 优先于 Level 1 (SEMANTIC)
- 若 critic 同时返回 MAJOR + Gate 失败 → StageRouter 先处理 MAJOR 超限检查 (返回 MAJOR 让 developer 修), 但 Judge 在下一轮 critic 时仍按 QUALITY 判定
- QUALITY stop 是终态, 不进入 developer 修复回路 (与 MAJOR→developer 不同)

**Gate 重试上限**: Gate 本身不自动重试。但 Orchestrator 在下一轮重新进入该 Stage 时会再次跑 Gate (因为每轮都跑 run_gates)。若连续 3 轮同一 Gate 失败 → 判定为 STAGNANT (由停滞检测捕获, 因 files_changed 无变化)。

### B6.4 Quality Gate 设计借鉴 (v5.1 新增, 2026-07-05)

CrewAI guardrail.py (参考源 `$AE_REFS_DIR/crewAI/`) 定义了:

_tasks_from_batch_plan(batch_plan, requirement) → Plan:
    # 设计假设: batch_plan 由 ArchitectAgent 生成, 仅含 developer 可执行的任务.
    # architect 的分析工作和 critic 的审查工作不由 batch_plan 描述.

    # 若未来需要支持非 developer batch, 扩展 batch_plan 的 role 字段.
    tasks = []
    for batch in batch_plan:
        tasks.append(Task(
            id=batch["id"],
            role="developer",          # 硬编码: batch_plan 仅描述 developer 任务
            description=batch["description"],
            file_targets=batch.get("files", []),
            depends_on=batch.get("depends_on", [])
        ))
    # 追加 critic task (审查所有 developer 产出)
    tasks.append(Task(
        id="critic-review",
        role="critic",
        description=f"审查所有 developer 产出. 需求: {requirement}",
        depends_on=[t.id for t in tasks]
    ))
    return Plan(tasks=tasks, requirement=requirement)
```

## B8. 收敛判定

### B8.1 停滞检测

**适用范围**: 仅检测**同 Stage 连续轮次**的产出变化。停滞检测主要捕获: Guardrail retry 导致的同 Stage 连续执行 (如 developer 连续 2 次无产出变化)。MAJOR 回路 (d→c→d→c) 中 critic 轮次始终重置计数器, 故停滞检测不适用于 MAJOR 回路 — 该场景由 StageRouter 的 MAJOR 上限控制。

```
detect_stagnation(history, threshold, diff_ratio_threshold) → bool:
    # 仅比较同 Stage 的相邻 entry (跨 Stage 自动重置计数器)
    if len(history) < threshold + 1: return False
    consecutive = 0
    for i in range(len(history)-1, 0, -1):
        # 跨 Stage → 重置 (不同 Stage 产出尺寸不可比)
        if history[i].stage != history[i-1].stage:
            consecutive = 0
            continue
        curr_size = history[i].files_changed + history[i].lines_added + history[i].lines_removed
        prev_size = history[i-1].files_changed + history[i-1].lines_added + history[i-1].lines_removed
        if curr_size == 0 and prev_size == 0:
            ratio = 0.0
        else:
            ratio = abs(curr_size - prev_size) / max(curr_size, prev_size)
        if ratio < diff_ratio_threshold:
            consecutive += 1
            if consecutive >= threshold: return True
        else:
            consecutive = 0
    return False
```

### B8.2 Gate 启用规则 (统一语义)

| config.gates 值 | 行为 | 触发方式 |
|-----------------|------|---------|
| `None` (默认) | 使用 DEFAULT_GATES (全部 7 道) | Orchestrator.__init__ 自动 |
| `[]` (空列表) | 跳过所有 Gate → gate_results={} → 降为 3 级收敛 | `--no-gates` CLI flag |
| `[Gate, ...]` | 使用指定 Gate 列表 | 编程接口 |

**`--no-gates` → `config.gates = []`** (不是 None)。`gate_results={}` → Judge Level 3 跳过质量门。系统降为 3 级收敛。这是设计意图。

## B9. 并发模型

### B9.1 三层架构

| 层 | 机制 | 内容 |
|----|------|------|
| L1 asyncio event loop | 主线程 | Orchestrator.run(), run_round() |
| L2 ThreadPoolExecutor | asyncio.to_thread | LLM sync 调用, Gate subprocess, git diff |
| L3 Subprocess | subprocess.run | Linter, type checker, test runner, build |

### B9.2 资源约束

| 资源 | 上限 | 机制 |
|------|------|------|
| 并发 Agent | 同 Stage 内 ≤20 task | Plan.get_tasks_by_stage() |
| 并发 Gate | 7 (全并行) | asyncio.gather |
| SQLite 连接 | 1 写 (WAL) | busy_timeout=5000ms |
| RoundHistory 内存 | 50 条 | deque(maxlen=50) |
| Checkpoint WAL | <10MB | 定期 VACUUM |
| LLM 并发请求 | 1 per Agent 实例 | Anthropic SDK 管理 |

### B9.3 并行 Task 写入安全

同 Stage 内多个 task 并行:
1. 每个 Agent 返回独立 TaskOutcome (含 output=dict)
2. run_round() 收集所有 outcomes → RoundResult
3. Orchestrator 在 run_round() 返回后, 串行遍历 outcomes → _apply_outcome_to_state()
4. Agent 不持有 LoopState → 无竞态条件
5. 多开发者写入 files_changed: 后来的覆盖 (Last-Write-Wins)。未来 _merge_outcomes() 做追加合并

### B9.4 Task 取消

```
CancellationToken.check() → raise AEError(TASK_CANCELLED) | None
安全取消点:
  - Round 开始前 (2a)
  - Stage 推进后 (2i 之后)
  不在 Agent LLM 调用中插入取消 (等当前调用返回)
  不在 Gate subprocess 中插入取消 (等 timeout)
```

## B10. 错误分类与恢复

### B10.1 错误树 (A-F)

| 类别 | 子类 | 动作 | 重试 |
|------|------|------|------|
| **A. 配置** | API_KEY 缺失 / project state 缺失 / uv 缺失 / project_root 不存在 | 提示 + exit 1 (project_root → 降级) | — |
| **B. Guardrail** | requirement 空 → block; plan 空 → retry≤3; git diff 空 → retry≤3; tests 失败 → retry≤3; git 不干净 → block | 见 B5 | ≤3 |
| **C. Gate** | safety/lint/type/test/build fail → failed; 工具缺失 → skip (不失败, 见 B10.3 降级); timeout → failed | Verdict 写入 gate_results | — |
| **D. LLM** | timeout/rate_limit/network → retry≤3 exp backoff; auth_error → abort; max_tokens → 降级; tool 异常 → error 回传 LLM | 见 B4.4 | ≤3 |
| **E. 中断** | SIGINT → checkpoint 保存 + exit 130; CancellationToken.check() → raise AEError(ERR_TASK_CANCELLED); asyncio.CancelledError → 捕获 → TaskOutcome(status="cancelled") | — | — |
| **F. 数据** | checkpoint 损坏 → dict fallback; schema mismatch → migrate; disk full → abort; SQLite busy → retry≤3 | 见 B7.5 | ≤3 |

### B10.1a 完整错误码枚举 (AEError.code)

所有 `AEError` 子类使用以下错误码。调用方按 code 做程序化处理 (不解析 message)。

| 错误码 | 类别 | HTTP 类比 | 含义 | 触发条件 |

- 调用方按 code 匹配 (如 `if e.code == ERR_LLM_RATE_LIMIT: backoff()`), 不解析 message

### B10.2 LLM 重试参数

| 参数 | 值 |
|------|-----|
| base_delay | 1.0s |
| max_delay | 30.0s |
| multiplier | 2.0 |
| jitter | ±25% |
| max_retries | 3 |

`delay = min(base * multiplier^(retry-1) * random(0.75, 1.25), max_delay)`

### B10.3 降级路径

**Engine 层降级**:

| 场景 | 降级 | 影响 |
|------|------|------|
| SafetyGate 工具缺失 | skip | 无 secret 检测 (pre-tool hook 独立 sandbox 兜底) |
| CoverageGate | 永远 skip | CI 独立检查覆盖率 |

| 场景 | 降级 | 影响 |

### B15.1 借鉴 LangGraph PregelLoop — 采纳/适配/跳过清单

研究来源: `langgraph/pregel/_loop.py:592-715` (PregelLoop.tick/after_tick), `_algo.py:87-110` (apply_writes/prepare_next_tasks)

| LangGraph 模式 | v5.0 处理 | 理由 |
|---------------|----------|------|
| **`tick() → bool`** — 单次迭代返回是否继续 | **适配** → B7.1 while 循环 | tick 封装单步逻辑, 但我们有 Stage 状态机 + MAJOR 回路, 单一 bool 不够表达; 改用 while + 多退出路径 |
| **`after_tick()` — apply_writes → clear → checkpoint 原子三步** | **采纳** → B7.1 2e(apply outcomes) → 2g(history) → 2i(checkpoint) | 写入-清理-持久化的顺序保证与 LangGraph 一致; 差异: 我们按 Stage 而非 Superstep 组织 |
| **`prepare_next_tasks` — channel version diff 找触发任务** | **适配** → B7.1 2c `get_tasks_by_stage` | LangGraph 用 channel_versions 增量触发; 我们按 Stage 粒度触发全部 task (Stage 内 task 少, 增量收益不大) |
| **`_checkpointer_put_after_previous` — 前一个 checkpoint 写入 durable 后才写下一个** | **跳过** → B7.4 同步 save | LangGraph 支持异步 checkpoint, 需要顺序保证; 我们用 SQLite 同步写入, WAL 模式提供原子性, 无需此机制 |
| **`apply_writes` — 收集所有 task writes → 批量 apply 到 channels** | **采纳** → B7.2 `_apply_outcome_to_state` | 同样: 所有 task 完成后统一写入, 避免 Agent 直接写共享状态; 差异: 我们用 dict key 匹配而非 channel version |
| **`interrupt_before/after` — 人工审批点** | **跳过 (v6.0)** | B15.2 已知差距; v5.0 是全自动 loop, 人工介入通过 SIGINT/CancellationToken |
| **`retry_policy` per task** | **跳过 (v5.1)** | v5.0 在 Guardrail 层做 retry (stage 粒度), 不做 per-task retry; per-task retry 增加复杂度但收益有限 |
| **`CachePolicy` — 缓存 task 输出供 replay** | **跳过** | v5.0 每次重新执行 task (LLM 调用天然非确定性, cache 命中率低) |
| **`PregelTaskWrites` — 每个 task 的 writes 结构化存储** | **采纳** → `TaskOutcome.output: dict` | LangGraph 用 `(channel, value)` tuple 列表; 我们用 `{field: value}` dict, 语义等价, 实现更简单 |

# PART C: Agent-Engine JSONL 通信协议 (v5.1, 2026-07-04 新增)

## C.1 问题

v5.0 Python 子进程通过 Anthropic SDK 直接调用 LLM (architect/developer/critic
三阶段 Agent), 在 Claude Code agent 内**完全失效**: 子进程无法获取 agent 的
ANTHROPIC_AUTH_TOKEN, architect/critic LLM 调用永远失败 (生产使用报告:
prismscan/_scratch/reports/2026-07-04-dev-loop-execution-analysis.md).

参考业界框架 (LangGraph PregelLoop.tick/after_tick、CrewAI kickoff/_execute_tasks、
AutoGen SingleThreadedAgentRuntime) 的共同模式: **Python 控制流 + LLM 数据流**。
Python 负责 while 循环 / 任务选择 / 收敛判定 / Gate / 持久化; LLM 在 Python
控制流内执行 architect/critic 分析任务.

v5.1 保留 Python 控制流不变, 将 architect/critic 两个 LLM 调用点从 SDK
改为 JSONL 流式协议——Claude Code agent 接收 JSON 请求后执行 Plan/code-reviewer
任务, 结果通过 stdin/stdout 回传给 Python orchestrator.

## C.2 架构

```
┌─────────────────────────────────────────────────────────────────┐
│ Claude Code Agent (父进程)                                       │
│                                                                   │
│   1. 读到 dev-loop.md Command spec                               │
│   2. 启动 Python 子进程 | ae dev-loop "<requirement>"           │
│   3. 轮询 stdin JSONL:                                           │
│      {"request":"architect","requirement":"...","context":{}}     │
│      → spawn Plan agent → 输出:                                  │
│      {"plan":"...","batch_plan":[],"file_list":[],"contracts":{}}│
│                                                                   │
│      {"request":"developer","tasks":[...],"context":{}}           │
│      → Claude Code agent 自己执行 TDD 循环 → 输出:               │
│      {"files_changed":[],"commit_hash":"","test_results":{}}      │
│                                                                   │
│      {"request":"critic","files_changed":[],"test_results":{}}    │
│      → spawn code-reviewer agent → 输出:                          │
│      {"verdict":"APPROVE|MAJOR","findings":[],"suggested_fix":""}│
│                                                                   │
│      {"request":"gate","tasks":[...]}                              │
│      → Python orchestrator 跑 gate 并行                          │
└──────────────────┬──────────────────────────────────────────────┘
                   │ JSONL (stdin←→stdout)
┌──────────────────▼──────────────────────────────────────────────┐
│ Python Orchestrator (子进程, ae dev-loop)                       │
│                                                                   │
│  while round < max_rounds:                                      │
│    _request_architect()     → write_jsonl(request) → read_jsonl(response)│
│    _request_developer()     → write_jsonl(developer_task) → read_jsonl(result)│
│    _request_critic()        → write_jsonl(critic_task) → read_jsonl(verdict)│
│    if verdict == "MAJOR": continue                              │
│    gate 并行 → 持久化 → break (收敛)                            │
└─────────────────────────────────────────────────────────────────┘
```

## C.3 JSONL 协议 (4 个请求类型)

### C.3a: architect 请求

```jsonl
{"request":"architect","requirement":"<user requirement>","context":{"project_root":"<path>"}}
```

Claude Code agent 响应:
```jsonl
{"plan":"markdown","batch_plan":[{"id":"T1","title":"","description":"","expected_output":"","file_targets":[],"depends_on":[]}],"file_list":[],"contracts":{}}
```

### C.3b: developer 请求

```jsonl
{"request":"developer","tasks":[{"id":"T1","description":"","expected_output":"","file_targets":[]}],"context":{"project_root":"<path>"}}
```

Claude Code agent 响应:
```jsonl
{"files_changed":[],"commit_hash":"","test_results":{"passed":N,"failed":M,"errors":E}}
```

### C.3c: critic 请求

```jsonl
{"request":"critic","files_changed":[],"commit_hash":"","test_results":{"passed":N},"gate_results":{}}
```

Claude Code agent 响应:
```jsonl
{"verdict":"APPROVE|MAJOR","findings":[{"file":"","line":N,"severity":"P0|P1|P2","issue":"","suggested_fix":"diff patch"}],"critic_feedback":""}
```

### C.3d: gate 请求 (Python 自己执行, 不需要 agent)

```jsonl
{"request":"gate","status":"running","gate_results":{"safety":{"passed":true},"lint":{"passed":false,"message":"..."},"type_check":{"passed":true},"test":{"passed":true,"message":"N passed"},"build":{"passed":true}}}
```

## C.4 错误处理与协议边界

| 场景 | Python 行为 |
|------|------------|
| agent 返回 malformed JSON | 重试 2 次 (CRITIC_RETRIES=2), 仍失败 → HARD_LIMIT |
| agent 超时 (5 min 无响应) | Python write_jsonl({"request":"heartbeat"}) → 无响应 → HARD_LIMIT |
| agent crash (stdin EOF) | Python 写 checkpoint → exit 130 |
| gate 失败 (safety/lint/test) | Python 在 verdict APPROVE 前必须检查 gate_results.all_passed() (且 gate fail 时降级 MAJOR) |

## C.5 与 v5.0 子进程模式对比

| 维度 | v5.0 子进程模式 | v5.1 JSONL 协议 |
|------|---------------|----------------|
| 控制流 | Python orchestrator | Python orchestrator (不变) |
| LLM 调用 | Python SDK (需要独立 ANTHROPIC_API_KEY) | Claude Code agent (复用 agent 的 LLM 连接, **零配置**) |
| architect 角色 | BaseAgent + AnthropicProvider (SDK 调) | Plan agent spawn (agent 内执行) |
| critic 角色 | BaseAgent + AnthropicProvider (SDK 调) | code-reviewer agent spawn (agent 内执行) |

## C.6 实施详情 (2026-07-05 完成)

### C.6a AE_JSONL_MODE 条件开关

JSONL 路径仅在 `AE_JSONL_MODE=1` 时启用 (Plugin agent context)。非 agent 模式 (CLI/测试) 回退到 `run_round` — 保证 1337 测试全通过, 向后兼容。

```python
# orchestrator.py _step_2e_run_agent:
in_jsonl = os.environ.get("AE_JSONL_MODE") == "1"
if in_jsonl and current_stage == "architect":
    arch_response = self._request_architect(...)  # JSONL
elif in_jsonl and current_stage == "critic":
    critic_response = self._request_critic(...)  # JSONL
else:
    round_result = await run_round(...)  # fallback: CLI/测试/developer
```

### C.6b 合成架构师 Task (空 tasks 入口)

当 Orchestrator `tasks=[]` 且 stage=architect 时, `_step_2c_select_tasks` 原逻辑会推进到下一 stage (跳过架构师)。修复: 在 JSONL 模式或 agent_runtime 可用时, 创建合成 architect task 触发 JSONL 规划路径。

```python
# orchestrator.py step 2c:
if not round_tasks and current_stage == "architect" and advance:
    in_jsonl = os.environ.get("AE_JSONL_MODE") == "1"
    if in_jsonl or self.config.agent_runtime is not None:
        round_tasks = [Task(id="architect-synthetic", role="architect", ...)]
```

### C.6c batch_plan → developer Tasks 链路

`_tasks_from_batch_plan()` (v5.0 §B7.3) 之前定义但从未调用。修复: 架构师 JSONL 响应中的 `batch_plan` 经 `_tasks_from_batch_plan` 转换为 developer Task 对象, 追加到 `self.plan.tasks`。
