## 六、风险评估

### 高风险
1. **LLM 真实调用的成本**：每次 dev-loop 可能消耗数千 token，必须 budget 限制(CLI 已有 `--max-tokens` 但 dead code)
2. **Agent 输出解析稳定性**：LLM 输出格式变化，parser 双层防御可能仍失败 → Phase 3 必须真测试
3. **Tools 真实执行的副作用**：WriteFile / Bash 可能破坏真实文件系统 → 必须路径白名单 + 沙箱
4. **v2.0 Channel 改造破坏 v1.0**：Channel 是基础抽象，必须向后兼容(渐进迁移)

### 中风险
1. **ruff B904 修复影响异常处理**：`raise X from e` 改变异常链，可能影响调试 → 逐个判断
2. **CancellationToken + LLM API 中断**：LLM 调用阻塞时 Ctrl-C 不能立即停止 → 需 asyncio.shield + 短超时
3. **多 Agent 并发文件冲突**：即使 check_file_isolation，边界情况可能漏

### 低风险
1. **CLI 命令扩展**：标准模式，按 click 文档
2. **tools registry**：字符串 → 实例解析简单
