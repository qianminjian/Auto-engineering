# 讨论记录 — v7.0 单引擎 + 双驱动架构

> 记录日期：2026-07-12 | 类型：讨论过程（"为什么这么想"）
> 决策产物：BEACON 决策 #54 | 规格产物：`../v7.0-Plan-DualDriver.md`
> 触发：用户就 T10d（v5.5 orchestrator 是否退役）追问"是否值得保留"，进而提出远期思路

---

## 一、讨论起点

T10d 上轮定案"保留 v5.5/v6 共存"（BEACON #53）后，用户追问 orchestrator.py(v5.5) 当前逻辑 + 是否值得保留。深入审计暴露一个不舒服的事实：

- v5.5 是**活代码**（`ae dev-loop "需求"` 裸参数路径可达），但
- 在**主场景**（Claude Code Plugin）下**完全失效**（2026-07-04 生产反馈：子进程拿不到 agent 的 AUTH_TOKEN），只在"独立 CLI + 自带 key"这个次要场景活着；
- 且它的流水线（三阶段 + deep_audit）**落后** v6（+ gap 分析 + 5 层验证 + plan_refine）；
- 维护上有**双引擎税**（orchestrator.py:580-609 的 T9 `10**9` 旁路 shim 就是证据）。

所以"保留共存"程序正确（红线保护），但没暴露"活路径主场景已死、只服务存疑次要场景"。

## 二、用户提出的思路

> 本项目方案开发完毕后，再基于当前版本增加一种调用方法：除 v5.6 的 agent 下运行外，也支持独立 CLI + 自带 key 的入口，**但内部循环的所有机制设计完全一致**。

关键洞察：用户**不是**要养第二个 fork，而是要 **一套引擎、两个入口**。

## 三、为什么这个思路成立（可行性论证）

### 3.1 v5.6 已经把"谁执行 LLM"抽成了契约边界

- `TickOrchestrator._after_tick(result: dict) -> action: dict` 已经是**纯函数核**（result 进、action 出），只有外层 `tick(result_file)` 绑了文件。
- 也就是说，"谁填 result"这件事**已经是可插拔的**：现在是 Claude Code Agent 填，将来可以是进程内 runtime 填。

### 3.2 独立跑需要的执行代码"已经存在且测过"

- Driver B 要做的 `action → 调 LLM → 产出 result` = v5.5 `_step_2e_run_agent`（run_round→AgentRuntime→anthropic）**已实现**。
- 核实：执行栈（AgentRuntime/run_round/BaseAgent/sandbox）**不只 v5.5 用**，`ae agent architect "需求"`（cli/agent.py 单 Agent 调用）也在用 → 即使删 v5.5 循环，执行层**不会成孤儿**。

### 3.3 tick 模型为什么不能直接独立跑（这正是需要 Driver B 的原因）

tick 模型必须有**外部 Agent** 在 tick 之间填 LLM 结果。脱离 Claude Code 就没人填 → 所以"独立/headless 全自动"**只能**靠一个"进程内自己填"的驱动。这就是 Driver B 的必要性，也是 v5.5 当前唯一护城河的本质。

## 四、为什么它优于"保留 v5.5 fork"

| | 保留 v5.5 fork（现状） | 单引擎 + 双驱动（远期） |
|--|--|--|
| 流水线真相源 | 两套分叉 | 一套（v5.6 全量） |
| 加新功能 | 两处同步（T9 shim 税） | 加一次，两驱动共享 |
| headless 能力 | 有，但落后三阶段 | 跑完整 v5.6 流水线 |
| v5.5 归宿 | 永久 fork | 被 Driver B subsume → 可干净退役 |

结论：**双驱动才是干掉 v5.5 尴尬的正确长期形态**——用薄驱动替代 fork，而不是留 fork。它也给 T10d 一个明确的远期出口（#53 保留成立，但不再是"永久保留"，而是"保留到 Driver B 就位再退役"）。

## 五、争议点与定调

### 5.1 "Python 永不调 LLM"原则冲突

Driver B 进程内调 LLM，直接顶撞 BEACON #39/#40。定调：这是**原则精确化（扩展）非翻转**——
> 循环**引擎**永不调 LLM；**驱动**可以。

引擎保持纯 Python 可测试（原则真正要保护的东西），Driver B 显式 opt-in 打破约束。#39/#40 status 不变。

### 5.2 "完全一致"能到什么程度

诚实边界：**编排层完全一致**（stages/验证/gates/收敛/plan_refine 同一份代码），但**执行保真度必然有差**（进程内 runtime 是 Claude Code agent 的能力子集）。预期设为"循环逻辑一致、执行后端不同"，不承诺输出质量对齐。

### 5.3 价值取决于一个业务问题

"团队是否真的用/将用 headless 独立跑"决定 Driver B 的价值。但**无论答案如何**，两项 P0 预留（action/result schema 固化 + 不误删执行栈）对当前代码质量都是净收益 → 所以当前阶段先只做这两项，Driver B 本体放 v7.0。

## 六、当前落地边界（YAGNI）

- 当前**只做** Phase 10 两项 P0 预留（保接缝），**不实现** StandaloneDriver。
- 不做多驱动插件框架（两个够了）。
- 不动 v5.6 现有流水线逻辑。
