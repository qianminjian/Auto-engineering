## 1. 系统架构概览

### 1.1 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│ CLI / Plugin Layer (slash commands + hooks)                   │
│  /dev-loop  /status  /checkpoint  /project:*                  │
├──────────────────────────────────────────────────────────────┤
│ Orchestrator Layer (主循环)                                    │
│  Orchestrator.run() → Round → Gate → Judge → Continue/Stop    │
├──────────────────────────────────────────────────────────────┤
│ Round Layer (单轮调度)                                         │
│  run_round() → asyncio.gather [Agent.execute()] → RoundResult │
├──────────────────────────────────────────────────────────────┤
│ Agent Layer (LLM 交互)                                        │
│  BaseAgent.execute() → tool_use loop → parse_output           │
│  ArchitectAgent / DeveloperAgent / CriticAgent                │
├──────────────────────────────────────────────────────────────┤
│ Infrastructure Layer                                           │
│  Checkpoint (SQLite WAL) / Channel System / Tools / Gates     │
│  Guardrail Chain / CancellationToken / SemanticEvaluator      │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 核心组件交互矩阵

| 组件 | Orchestrator | Round | Agent | Gate | Guardrail | Checkpoint | Judge |
|------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| Orchestrator | — | 调用 run_round | 间接(通过 Round) | 委托 Round | 不直接 | 调用 save | 调用 evaluate |
| Round | 被调用 | — | 调用 executor | 调用 _run_gates | 不调用 | 不调用 | 不调用 |
| Agent | 被 Round 调度 | 通过 executor | — | 不调用 | 被 GuardrailChain 检查 | 不调用 | 不调用 |
| Gate | 配置来源 | 在 run_round 内跑 | 不调用 | — | 不调用 | 不调用 | Judge 读取结果 |
| Guardrail | Orchestrator 预检 | 不调用 | 检查 Agent 输入/输出 | 不调用 | — | 不调用 | 不调用 |
| Checkpoint | Orchestrator after_tick | 不调用 | 不调用 | 不调用 | 不调用 | — | Judge 读 history |
| Judge | Orchestrator 每轮调 | 不调用 | 不调用 | 读 gate_results | 不调用 | 读 history | — |

## 2. 数据模型（完整 Schema）

### 2.1 LoopState (运行时状态容器 — engine.state.EngineState)

| 字段 | 类型 | 约束 | 默认值 | 语义 |
|------|------|------|--------|------|
| requirement | `str` | 1..4096 chars, NOT NULL | "" | 原始需求文本 |
| round | `int` | 0..N, NOT NULL | 0 | 当前轮次编号 |
| stage | `Literal["","architect","developer","critic"]` | 枚举 | "" | 当前阶段 |
| plan | `str \| None` | 0..65535 chars | None | architect 产出的计划文本 |
| file_list | `list[str] \| None` | 0..200 元素 | None | architect 指定要创建/修改的文件 |
| files_changed | `list[str] \| None` | 0..200 元素 | None | developer 实际修改的文件 |
| commit_hash | `str \| None` | pattern `[0-9a-f]{40}` | None | developer 提交的 commit |
| test_results | `dict \| None` | `{passed:int, failed:int, errors:int}` | None | developer 测试结果 |
| verdict | `Literal["APPROVE","MAJOR",""]` | 枚举 | "" | critic 判定 |
| findings | `list[dict] \| None` | 0..50 元素 | None | critic 发现的问题列表 |
| critic_feedback | `str \| None` | 0..65535 chars | None | critic 给 developer 的反馈 |

### 2.2 CheckpointEnvelope (checkpoint 持久化容器 — loop.state)

| 字段 | 类型 | 约束 | 语义 |
|------|------|------|------|
| thread_id | `str` (UUID v4) | PK, NOT NULL | 会话唯一标识 |
| round | `int` | PK, ≥0 | 当前轮次 |
| step | `int` | PK, ≥0 | 当前步骤 (Round 内子步骤) |
| status | `Literal["running","completed","interrupted","failed"]` | NOT NULL | 运行状态 |
| channels | `dict[str, BaseChannel]` | NOT NULL | Channel 实例 dict (key → BaseChannel) |
| channel_versions | `dict[str, int]` | NOT NULL | Channel 版本号 (key → version, LangGraph 借鉴) |
| created_at | `datetime` (UTC) | NOT NULL, auto | 创建时间戳 |

**Channel 序列化三件套 (LangGraph BaseChannel 协议)**:
- `channel.copy() → BaseChannel` — 深拷贝当前 channel
- `channel.checkpoint() → Any` — 序列化为可持久化值
- `channel.from_checkpoint(value) → BaseChannel` — 从持久化值重建

### 2.3 RoundHistory (单轮历史记录)

| 字段 | 类型 | 约束 | 语义 |
|------|------|------|------|
| round_id | `int` | ≥1, NOT NULL | 轮次编号 (1-indexed) |
| files_changed | `int` | 0..9999, default=0 | 修改文件数 |
| lines_added | `int` | 0..99999, default=0 | 新增行数 |
| lines_removed | `int` | 0..99999, default=0 | 删除行数 |
| gate_results | `dict[str, Verdict]` | 0..7 keys | Gate 名称 → Verdict 对象 |
| semantic_satisfied | `bool \| None` | default=None | LLM 语义评估结果 (None=未评估) |
| tasks_run | `list[str]` | 0..20 元素 | 本轮执行的 task ID 列表 |
| task_outcomes | `dict[str, str]` | key=task_id, value="completed"\|"failed"\|"cancelled" | 每个 task 的最终状态 |

### 2.4 Task (Plan 中的任务单元)

| 字段 | 类型 | 约束 | 语义 |
|------|------|------|------|
| id | `str` | NOT NULL, unique | 任务唯一标识 |
| role | `Literal["architect","developer","critic"]` | NOT NULL | 执行此任务的 Agent 角色 |
| description | `str` | 1..4096 chars, NOT NULL | 任务描述 (给 Agent 的指令) |
| expected_output | `str \| None` | 0..4096 chars | 期望输出格式 |
| depends_on | `list[str]` | 0..20 元素 | 依赖的 task ID 列表 |
| file_targets | `list[str] \| None` | 0..50 元素 | 此任务涉及的文件路径 |

### 2.5 TaskOutcome (单 task 执行结果)

| 字段 | 类型 | 语义 |
|------|------|------|
| task_id | `str` | 对应 Task.id |
| status | `Literal["completed","failed","cancelled"]` | 执行状态 |
| output | `Any \| None` | 成功时的输出 |
| error | `str \| None` | 失败时的错误信息 |
| duration | `float` | 耗时 (秒) |

### 2.6 RoundResult (单轮汇总)

| 字段 | 类型 | 语义 |
|------|------|------|
| round_id | `int` | 轮次编号 |
| outcomes | `list[TaskOutcome]` | 每个 task 的结果 |
| gate_results | `dict[str, Verdict]` | Gate 运行结果 (name → Verdict) |
| history | `list[RoundHistory]` | 本轮的 RoundHistory (通常 1 元素) |

| finished_at | `float` | 完成时间戳 |

**派生属性**: `duration`, `completed_count`, `failed_count`, `all_succeeded`, `all_gates_passed`

### 2.7 Verdict (两类)

**Gate Verdict** (gates/base.py):
| 字段 | 类型 | 语义 |
|------|------|------|
| gate_name | `str` | Gate 名称 |
| passed | `bool` | 是否通过 |
| message | `str` | 详细信息 |

**Convergence Verdict** (convergence.py):
| 字段 | 类型 | 语义 |
|------|------|------|
| should_stop | `bool` | 是否应停止循环 |
| level | `int` (0-4) | 触发级别 (0=继续, 1=语义, 2=停滞, 3=质量, 4=硬上限) |
| reason | `str` | 触发原因 |

### 2.8 Plan (任务 DAG)

| 字段 | 类型 | 语义 |
|------|------|------|
| tasks | `list[Task]` | 所有任务 |
| requirement | `str` | 原始需求 |
| dag | 内部有向图 | task 依赖关系 (depends_on 解析) |

**校验方法**: `validate()` → 检查 DAG 无环 + 文件隔离 + role 合法

### 2.9 Checkpoint (SQLite 持久化行)

```sql
CREATE TABLE checkpoints (
    thread_id   TEXT NOT NULL,
    round       INTEGER NOT NULL,
    step        INTEGER NOT NULL,
    state_json  TEXT NOT NULL,       -- CheckpointEnvelope JSON
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    PRIMARY KEY (thread_id, round, step)
) WITHOUT ROWID;
```

```sql
CREATE TABLE gate_results (
    thread_id   TEXT NOT NULL,
    round       INTEGER NOT NULL,
    gate_name   TEXT NOT NULL,
    passed      INTEGER NOT NULL,    -- 0/1
    message     TEXT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    PRIMARY KEY (thread_id, round, gate_name)
);
```

**配置**: `PRAGMA journal_mode=WAL; PRAGMA busy_timeout=5000;`

## 3. Stage 状态机（完整）

```
                    ┌──────────────────────────────────┐
                    │           [ idle ]                │
                    │  stage=""  round=0                │
                    └────┬──────────────────────────────┘
                         │ CMD-01 /dev-loop
                         ▼
                    ┌──────────────────────────────────┐
                    │        [ architect ]              │
                    │  agent=ArchitectAgent             │
                    │  tools: read_file, search_code,   │
                    │         list_dir                  │
                    │  output: plan, file_list,          │
                    │          files_needed              │
                    └────┬──────────────────────────────┘
                         │ Guardrail: PlanExists ✓
                         │ Guardrail: RequirementValid ✓
                         ▼
                    ┌──────────────────────────────────┐
                    │        [ developer ]              │
                    │  agent=DeveloperAgent             │
                    │  tools: read_file, write_file,    │
                    │         edit_file, run_bash,      │
                    │         git_commit, run_tests     │
                    │  output: files_changed,            │
                    │          commit_hash, test_results │
                    │  TDD: RED → GREEN → REFACTOR      │
                    └────┬──────────────────────────────┘
                         │ Guardrail: GitDiffExists ✓
                         │ Guardrail: TestsPass ✓
                         │ Guardrail: GitClean ✓ (no dirty)
                         ▼
                    ┌──────────────────────────────────┐
                    │         [ critic ]                │
                    │  agent=CriticAgent                │
                    │  tools: read_file, git_diff,      │
                    │         run_tests                 │
                    │  output: verdict, findings,        │
                    │          critic_feedback           │
                    └────┬──────────────────────────────┘
                         │
                    ┌────┴────┐
                    │         │
               APPROVE     MAJOR
                    │         │
                    ▼         ▼
              [ idle ]   [ developer ]  (最多 3 轮 MAJOR, 连续 2 次 → HARD_STOP)
```

**状态转换表**:

| 从 | 到 | 条件 | 动作 |
|----|-----|------|------|
| idle | architect | CMD-01 触发 | next_stage(state) → "architect" |
| architect | developer | Guardrail pass | next_stage(state) → "developer" |
| developer | critic | Guardrail pass | next_stage(state) → "critic" |
| critic | idle | verdict=APPROVE | ConvergenceJudge → stop |
| critic | developer | verdict=MAJOR AND retry_count < 3 AND 非连续2次 | state.critic_feedback 传递 |
| critic | idle | verdict=MAJOR AND (retry_count ≥ 3 OR 连续2次MAJOR) | HARD_STOP, verdict.reason="连续 MAJOR 超限" |
| any | idle | round ≥ max_iterations | HARD_STOP |
| any | idle | SIGINT / CancellationToken | 保存当前 checkpoint → idle |

## 4. Agent 规范

### 4.1 ArchitectAgent

| 维度 | 规格 |
|------|------|
| **角色** | 技术架构师 — 分析需求，产出可执行计划 |
| **输入** | `requirement: str`, `TaskContext { state, requirement }` |
| **允许工具** | `read_file`, `search_code`, `list_dir` (只读) |
| **禁止工具** | 任何写工具 (write_file, edit_file, run_bash, git_commit, run_tests) |
| **输出 Schema** | `{ files_needed: list[str], files_to_create: list[str], files_to_modify: list[str], plan: str, file_list: list[str], batch_plan?: list[dict], contracts?: dict }` |
| **解析策略** | 双层: 1) JSON Schema 校验 (Pydantic) → 2) regex 从 markdown fence 提取 JSON → 都失败则本轮 failed |
| **max_tokens** | 4096 output |
| **temperature** | 0.3 (需要确定性输出) |
| **thinking** | disabled (规划任务不需要深度思考) |

**System Prompt 核心段落** (完整 prompt 在 `agents/prompts.py:ARCHITECT_SYSTEM_PROMPT`):
1. 角色定义 + 职责边界
2. 文件集预检协议 (广撒网/基于现状/不预判改动)
3. 输出格式规范 (JSON schema + 字段说明)
4. 设计原则 (最小化变更/可独立验证/明确边界/文件粒度)
5. 工具使用约束

**预检规则**: 必须先输出 `files_needed` (含创建+修改+引用), 再进入 plan 分析。宁多勿漏。

### 4.2 DeveloperAgent

| 维度 | 规格 |
|------|------|
| **角色** | 开发者 — 按 architect plan 实施变更，严格 TDD |
| **输入** | `Task { description, expected_output }`, `TaskContext { state, plan?, critic_feedback? }` |
| **允许工具** | `read_file`, `write_file`, `edit_file`, `run_bash`, `git_commit`, `run_tests` |
| **禁止工具** | 无 (全工具可用, 但受 bash sandbox 限制) |
| **输出 Schema** | `{ files_changed: list[str], commit_hash: str, test_results: {passed:int, failed:int, errors:int} }` |
| **解析策略** | 双层: 1) JSON Schema → 2) regex → 失败则本轮 failed |
| **max_tokens** | 8192 output |
| **temperature** | 0.2 |
| **thinking** | disabled |
| **TDD 纪律** | RED (写失败测试 → run → 确认 FAIL) → GREEN (最少实现 → run → 确认 PASS) → REFACTOR (保持绿) |
| **行为约束** | 不偏离 plan.file_list; 单个 commit 一个 batch; commit 前测试全绿; 失败不 skip/注释 |

**System Prompt 核心段落** (完整 prompt 在 `agents/prompts.py:DEVELOPER_SYSTEM_PROMPT`):
1. TDD 三步循环
2. 输出格式
3. 工具列表 + 使用方式
4. 行为约束 (不偏离 plan, 小步提交, 测试必跑, 失败不绕过)

**Critic Feedback 注入**: 若 `state.critic_feedback` 非空, 在 system prompt 末尾追加: "上一轮 Critic 反馈:\n{critic_feedback}\n请针对性修复以上问题。"

### 4.3 CriticAgent

| 维度 | 规格 |
|------|------|
| **角色** | 代码审查者 — 审查 developer 产物，判定 APPROVE/MAJOR |
| **输入** | `Task { description }`, `TaskContext { state }` (含 commit_hash, files_changed, test_results) |
| **允许工具** | `read_file`, `git_diff`, `run_tests` (只读+验证) |
| **禁止工具** | `write_file`, `edit_file`, `run_bash`, `git_commit` |
| **审查维度** | 正确性 / 测试覆盖 / 代码质量 / 接口契约 / 运行时正确性 |
| **输出 Schema** | `{ verdict: "APPROVE"\|"MAJOR", findings: [{file, issue, severity:"P0"\|"P1"\|"P2"}], critic_feedback: str }` |
| **解析策略** | 双层: 1) JSON Schema → 2) regex → 失败则本轮 failed |
| **max_tokens** | 4096 output |
| **temperature** | 0.1 (审查需要一致性) |
| **thinking** | disabled |
| **判定规则** | APPROVE: 所有维度通过; MAJOR: ≥1 P0 或 ≥3 P1 |
| **审查原则** | Fresh Context (不看 developer 推理, 只看产物); 具体问题 (file:line + 修复建议); 诚实 (不确定标 P2) |

### 4.4 Agent 工具调用循环 (BaseAgent 协议)

```
execute(task, ctx) → TaskOutcome:
  1. 构造 messages: [system_prompt, user_message(task.description)]
  2. while True:  # tool_use loop (最多 10 轮, 借鉴 Anthropic 推荐 ≤25)
    2a. response = llm.create_message(messages, tools, tool_choice="auto")
    2b. if response.stop_reason == "end_turn":
          break  # LLM 完成, 去解析输出
    2c. if response.stop_reason == "tool_use":
          for tool_use in response.tool_use_blocks:
            tool = registry.get(tool_use.name)
            if tool is None: → 返回错误 (未知工具)
            if not authz.can_use(agent_role, tool_use.name): → 返回错误 (越权)
            result = tool.execute(tool_use.input)  # 沙箱隔离
            messages.append({"role":"user","content":[{"type":"tool_result","tool_use_id":tool_use.id,"content":result}]})
    2d. if response.stop_reason == "max_tokens": → 截断, 标记 warning
    2e. if loop_count > 10: → 强制终止, 返回 failed
  3. parse_output(response.content) → 双层解析 → dict
  4. validate_output(dict) → 合法 → TaskOutcome(status="completed"); 不合法 → failed
```

**借鉴来源**:
- Anthropic 官方 agent loop 模式: `stop_reason == "tool_use"` → 执行工具 → 追加 `tool_result` → 续调
- AutoGen `SingleThreadedAgentRuntime`: 消息循环 + 工具执行 + 结果回传

## 5. Guardrail 链规范

### 5.1 Guardrail 基类接口

```
GuardrailResult { action: "pass" | "block" | "drop" | "retry", message: str }
Guardrail.check(stage: Stage, state: LoopState, context: dict) → GuardrailResult
```

**4 态语义 (CrewAI GuardrailResult 借鉴)**:
- `pass` — 通过, 继续下一阶段
- `block` — 硬阻断, 本轮失败 (不可恢复)
- `drop` — 静默丢弃本轮产出, 重跑当前 stage
- `retry` — 重试当前 stage (最多 3 次, 连续 3 次 → block)

### 5.2 5 个 Guardrail

| # | Guardrail | 检查时机 | 检查内容 | 失败动作 |
|---|-----------|---------|---------|---------|
| G1 | **RequirementValid** | pre-architect | requirement 非空, 长度 1..4096, 不含仅控制字符 | block |
| G2 | **PlanExists** | post-architect | state.plan 非空 AND state.file_list 非空 | retry (≤3) |
| G3 | **GitDiffExists** | post-developer | git diff HEAD~1..HEAD 非空 (确实有变更) | retry (≤3) |
| G4 | **TestsPass** | post-developer | state.test_results.failed == 0 AND errors == 0 | retry (≤3) |
| G5 | **GitClean** | post-developer | git status --porcelain 为空 (无未提交变更) | block |

### 5.3 GuardrailChain

```
GuardrailChain.check(stage, state) → GuardrailResult:
  for guardrail in chain:  # 按注册顺序
    result = guardrail.check(stage, state)
    if result.action != "pass":
      return result  # 第一个失败就停止 (fail-fast)
  return GuardrailResult.pass()
```

**注册顺序**: RequirementValid → PlanExists → GitDiffExists → TestsPass → GitClean

## 6. Gate 体系规范

### 6.1 Gate 基类接口

```
Gate { name: str }
Gate.run(project_root: Path) → Verdict  # 同步方法, 调用方用 asyncio.to_thread 包装
```

### 6.2 7 道 Gate

| # | Gate | 实现 | 输入 | 判定逻辑 | 耗时估算 | 依赖 |
|---|------|------|------|---------|---------|------|
| G1 | **SafetyGate** | 正则扫描秘密/密钥 | 所有源码文件 | 匹配 API key / token / 密码模式 → fail | <5s | 无 |
| G2 | **LintGate** | subprocess 调 linter | project_root | 基于语言调 ruff/eslint/golangci-lint/clippy; exit≠0 → fail | 10-30s | 无 |
| G3 | **TypeCheckGate** | subprocess 调 type checker | project_root | pyright/tsc/go vet/cargo check; exit≠0 → fail | 10-30s | 无 |
| G4 | **ContractGate** | 检查跨模块接口 | architect.contracts + 实际代码 | 多 Agent 时检查 contracts 是否被遵守; 单 Agent 返回 skip | <5s | G2 |
| G5 | **TestGate** | subprocess 调 test runner | project_root | pytest/vitest/go test/cargo test; exit≠0 → fail | 30-120s | 无 |
| G6 | **CoverageGate** | subprocess 调 coverage tool | project_root | 覆盖率 < 80% → fail; **冻结, 永远返回 skip** (16G 内存约束) | 0s | G5 |
| G7 | **BuildGate** | subprocess 调 build | project_root | 基于语言调 build 命令; exit≠0 → fail | 10-60s | G2, G3 |

### 6.3 Gate 并发模型

```
_run_gates(gates, project_root) → dict[name, Verdict]:
  coros = [asyncio.to_thread(gate.run, project_root) for gate in gates]
  results = await asyncio.gather(*coros, return_exceptions=True)
  # 并行跑 — 总耗时 ≈ max(单个 gate) 而非 sum
  # Gate 异常 → Verdict(passed=False, message=str(exc))
  # LangGraph 借鉴: after_tick 中 apply_writes 也是批量并行
```

**Gate 依赖链**: ContractGate 依赖 LintGate 通过 (lint 失败则 contract 检查无意义), 但并行模型下 ContractGate 自身判断是否执行。

### 6.4 Gate 异常处理

- 单个 Gate 抛异常 → 捕获, 返回 `Verdict.failed(f"Gate {name} 异常: {exc}")`
- 所有 Gate 异常 → gate_results 含全部 failed, Judge 触发 QUALITY_FAIL
- Gate subprocess 超时 → `subprocess.run(timeout=...)` → TimeoutExpired → failed

## 7. Channel 系统

### 7.1 设计定位

Channel 系统是 checkpoint 持久化专用的状态存储抽象。v2.0 实现借鉴 LangGraph `BaseChannel` 协议 (generic `Value/Update/Checkpoint` 类型参数 + `copy/checkpoint/from_checkpoint` 三件套)。运行时状态走 `engine.state.LoopState` dataclass, Channel 仅供 persistent envelope (`CheckpointEnvelope.channels`)。

### 7.2 Channel 类型

| Channel | 用途 | Update 语义 | Get 语义 | 实际文件 |
|---------|------|------------|---------|---------|
| **LastValue** | 单值覆盖 | 最后写入的值覆盖 | 返回最后值 | `loop/state/last_value.py` |
| **BarrierChannel** | N 个 task 都完成后才可读 | 收集所有 task 的值, 计数 | 计数 < N → EmptyChannelError; = N → 所有值列表 | `loop/state/barrier.py` |
| **AccumulatingChannel** | 累加所有更新 | 追加到列表 | 返回完整列表 | (预留, 未来) |
| **Topic** | 发布/订阅 | 按 topic 分组 | 返回指定 topic 的更新列表 | (预留, 未来 Send API) |

### 7.3 Channel 协议 (LangGraph BaseChannel 对齐)

```
BaseChannel[Value, Update, Checkpoint]:
  # 序列化三件套
  copy() → Self                          # 深拷贝
  checkpoint() → Checkpoint              # 序列化当前值
  from_checkpoint(c: Checkpoint) → Self  # 从序列化值重建

  # 读写
  get() → Value                          # 读当前值 (空时 raise EmptyChannelError)
  update(values: Sequence[Update]) → bool  # 批量更新 (True=有变化, False=无变化)
  is_available() → bool                  # 是否有值可读
```

### 7.4 Channel 版本追踪 (LangGraph `channel_versions` 借鉴)

每个 channel 维护一个单调递增的 version 整数。每次 `update()` 返回 True 时 version++。version 用于:
1. Checkpoint 差异检测 (`_get_new_channel_versions`)
2. Orchestrator 增量 task 选择 (只有关联 channel 被修改的 task 才触发执行)
3. 停滞检测辅助信号 (哪个 channel 真正有变化)

## 8. Orchestrator 主循环（完整时序）

### 8.1 run() 生命周期

```
Orchestrator.run(cancellation?) → list[RoundHistory]:
  ┌─ 初始化 ─────────────────────────────────────────────┐
  │ 1. Plan.validate() → DAG 无环 + 文件隔离检查          │
  │ 2. judge = ConvergenceJudge(config)                     │
  │ 3. max_rounds = judge.config.max_iterations (单一来源)  │
  │    (LangGraph Pregel.recursion_limit 借鉴 — 单一字段    │
  │     多处引用, 不重复存储)                               │
  └──────────────────────────────────────────────────────┘
  │
  ┌─ 主循环 (for round_id in 1..max_rounds) ────────────┐
  │ 4a. cancellation?.check() → 中断则 break               │
  │ 4b. round_tasks = _select_round_tasks(round_id,        │
  │       history)                                         │
  │     - Round 1: 所有 task                               │
  │     - Round 2+: 仅 failed + new task                   │
  │     (LangGraph _prepare_next_tasks 借鉴 —              │
  │      channel version diff 决定触发哪些 task)            │
  │                                                        │
  │ 4c. pre_guardrail_chain.check(stage, state)             │
  │     → pass: 继续; block/drop/retry: 处理               │
  │                                                        │
  │ 4d. round_result = await run_round(                    │
  │       tasks=round_tasks,                               │
  │       executor=agent_executor,                         │
  │       ctx=state,                                       │
  │       cancellation=cancellation,                       │
  │       round_id=round_id,                               │
  │       gates=config.gates,                              │
  │       project_root=config.project_root)                │
  │     → RoundResult { outcomes, gate_results, history }  │
  │                                                        │
  │ 4e. post_guardrail_chain.check(stage, state)           │
  │                                                        │
  │ 4f. semantic_satisfied = await _evaluate_semantic(     │
  │       round_result)                                    │
  │     → 写回 round_result.history[0].semantic_satisfied  │
  │     (LangGraph Pregel.tick() Packet 模式借鉴:         │
  │      数据在生产方打包, 调用方不二次构造)                │
  │                                                        │
  │ 4g. round_results.append(result);                      │
  │     history.extend(result.history)                     │
  │                                                        │
  │ 4h. verdict = judge.evaluate(history=history)          │
  │     → should_stop? return history                      │
  │                                                        │
  │ 4i. checkpoint.save(state, round, history)             │
  │     (LangGraph after_tick 借鉴: 每轮后保存              │
  │      checkpoint, 不丢失进度)                            │
  └──────────────────────────────────────────────────────┘
  │
  ┌─ 退出 ─────────────────────────────────────────────┐
  │ 5. verdict = Verdict.stop(level=4,                    │
  │      reason="达到最大轮次")                            │
  │ 6. return history                                     │
  └──────────────────────────────────────────────────────┘
```

| 阶段 | 最短 | 典型 | 最长 | 超时处理 |
|------|------|------|------|---------|

## 9. 收敛判定（4 级, 从硬到软）

### 9.1 判定优先级

```
ConvergenceJudge.evaluate(history) → Verdict:
  1. 硬上限 (level=4): len(history) >= max_iterations → stop
  2. 质量门 (level=3): 最新一轮所有 Gate 全 PASS → stop
     任一 Gate fail → stop (质量门失败, 不应继续)
  3. 停滞检测 (level=2): 连续 N 轮 diff_ratio < threshold → stop
  4. 语义收敛 (level=1): 最新一轮 semantic_satisfied == True → stop
  5. 默认: continue
```

### 9.2 停滞检测算法

```
detect_stagnation(history, threshold=2, diff_ratio=0.05) → bool:
  if len(history) < threshold + 1: return False
  # 从最新一轮往前回溯
  consecutive = 0
  for i in range(len(history)-1, 0, -1):
    ratio = diff_ratio(history[i], history[i-1])
    if ratio < diff_ratio_threshold:
      consecutive += 1
      if consecutive >= threshold: return True
    else:
      consecutive = 0  # 有变化, 重置
  return False

diff_ratio(current, previous) → float [0.0, 1.0]:
  # 4 维度: files_changed + lines_added + lines_removed
  # 公式: |current - previous| / max(current, previous)
```

### 9.3 语义评估 (ClaudeSemanticEvaluator)

```
ClaudeSemanticEvaluator.evaluate(round_result) → bool | None:
  if not ANTHROPIC_API_KEY or CLAUDE_CODE is set: return None  # 降级
  构造 prompt: "需求是 [{requirement}], 本轮产出: {outcomes summary}.
                产出是否满足需求? 回答 YES/NO."
  调 Claude API (轻量, max_tokens=50) → 解析 YES/NO
  YES → True, NO → False
  异常 / timeout 5s → None (safe degradation)
```

**借鉴**: LangGraph ConditionalEdge (LLM 评估路由, 开箱即用)

## 10. Checkpoint 系统

### 10.1 持久化接口

```
SQLiteCheckpointStore:
  save(envelope: CheckpointEnvelope) → checkpoint_id (UUID)
  load(checkpoint_id) → CheckpointEnvelope (含重建的 Channel 实例)
  list(thread_id) → [{thread_id, round, step, created_at}]
  delete(checkpoint_id) → bool
```

### 10.2 序列化协议

```
save 流程:
  1. envelope.model_dump(mode="json") → dict
     - 每个 channel: channel.checkpoint() → 可序列化值
  2. json.dumps(dict) → state_json
  3. INSERT INTO checkpoints (thread_id, round, step, state_json)

load 流程:
  1. SELECT state_json FROM checkpoints WHERE thread_id=? AND round=? AND step=?
  2. json.loads → dict
  3. 遍历 dict["channels"]: channel_type.from_checkpoint(value) → 重建 BaseChannel
  4. CheckpointEnvelope(**dict) → 返回
```

### 10.3 Crash 恢复

```
场景 A: Round K 中途 crash
  1. checkpoint 已有 K-1 的保存 (after_tick 每轮保存)
  2. Round K 的部分写入在 SQLite tx 内, 未 COMMIT → 自动回滚
  3. CMD-03 resume <id> → load K-1 → 重跑 Round K

场景 B: checkpoint 文件损坏
  1. 尝试加载 → json.JSONDecodeError
  2. 返回 raw dict fallback (__raw__ 字段)
  3. 提示用户: "checkpoint 损坏, 建议重新初始化"

场景 C: schema mismatch (v1 → v2)
  1. CheckpointEnvelope 字段不匹配 → 降级尝试
  2. _migrate_checkpoint 钩子 (LangGraph 借鉴) — 版本升级时字段映射
  3. 迁移失败 → raise 明确错误 (不静默丢弃)
```

## 11. 并发与线程模型

### 11.1 模型层次

```
Layer 1: asyncio event loop (主线程)
  - Orchestrator.run() 是 async 协程
  - run_round() 用 asyncio.gather 并行调度多个 Agent
  - 所有 Agent LLM 调用通过 asyncio.to_thread 移到 thread pool

Layer 2: ThreadPoolExecutor (LLM 调用)
  - Anthropic SDK sync 调用 (create_message) → asyncio.to_thread 包装
  - 避免阻塞 event loop (D-P0-1 修复)
  - 默认线程池大小 = min(32, os.cpu_count() + 4)

Layer 3: Subprocess (Gate 执行)
  - 每个 Gate 是独立 subprocess (linter, type checker, tests)
  - asyncio.to_thread(gate.run) → 线程内 subprocess.run
  - timeout 控制: subprocess.run(timeout=120)
```

### 11.2 资源约束

| 资源 | 上限 | 超限行为 |
|------|------|---------|
| 并发 Agent 数 | 3 (architect/developer/critic 各 1) | 无 (plan 阶段保证) |
| 并发 Gate 数 | 7 (全部并行) | 无 (Gate 只读, 无共享写) |
| LLM 并发请求 | 1 per Agent (Anthropic SDK 限制) | SDK 层 rate limit → retry |
| SQLite 并发连接 | 1 (WAL mode, 单写者) | busy_timeout=5000ms |
| 内存 (RoundHistory) | 50 轮 (deque maxlen) | 旧记录自动丢弃 |
| 内存 (Checkpoint) | WAL 文件 < 10MB | 定期 VACUUM |

### 11.3 Task 取消

```
CancellationToken:
  is_cancelled() → bool
  check() → None | raise AEError(TASK_CANCELLED)

取消边界:
  - Round 开始前: cancellation.check() (安全取消点)
  - Agent LLM 调用中: Anthropic SDK 不原生支持取消 → 等当前调用返回后 check
  - Gate subprocess: process.terminate() → SIGTERM → 5s → SIGKILL
  - 不在 task 执行中间插入取消 (避免数据半写)
```

## 12. 错误分类与恢复策略

### 12.1 错误树 (6 类 A-F)

| 类别 | 子类 | 检测时机 | 动作 | 重试 | 传播 |
|------|------|---------|------|------|------|
| **A. 配置** | ANTHROPIC_API_KEY 缺失 | CMD-01 预检 | 提示设置 env | 否 | exit 1 |
| | project state 缺失 | CMD-01 预检 | 提示 /init | 否 | exit 1 |
| | uv 缺失 | Hook 入口 | 提示安装 + CLI fallback | 否 | exit 1 |
| | project_root 不存在 | Gate 入口 | 跳过所有 Gate | 否 | 继续(降级) |
| **B. Guardrail** | requirement 空 | pre-architect | block → exit 1 | 否 | exit 1 |
| | plan 空 | post-architect | retry ≤3 → block | 是(3) | round failed |
| | git diff 空 | post-developer | retry ≤3 → block | 是(3) | round failed |
| | tests 失败 | post-developer | retry ≤3 → block | 是(3) | round failed |
| | git 不干净 | post-developer | block | 否 | round failed |
| **C. Gate** | safety 命中 | STEP5 (parallel) | Verdict.failed | 否 | Judge 读 gate_results |
| | lint/type/test/build fail | STEP5 | Verdict.failed | 否 | Judge 触发 QUALITY_FAIL |
| | gate 工具缺失 | STEP5 | Verdict.failed("missing: tool") | 否 | 同上 |
| | gate subprocess timeout | STEP5 | Verdict.failed("timeout: 120s") | 否 | 同上 |
| **D. LLM** | timeout (>120s) | STEP3 | retry ≤3, exponential backoff | 是(3) | TaskOutcome failed |
| | rate_limit (429) | STEP3 | retry ≤3, exponential backoff + jitter | 是(3) | TaskOutcome failed |
| | auth_error (401) | STEP3 | abort (不重试, 密钥错) | 否 | exit 1 |
| | network_error | STEP3 | retry ≤3 | 是(3) | TaskOutcome failed |
| | max_tokens | STEP3 | output 截断标记 warning | 否 | 继续(降级) |
| **E. 中断** | SIGINT | 任意时刻 | 当前 round 中止, checkpoint 保留 | — | exit 130 |
| | CancellationToken | Round 边界 | 保存 checkpoint → return | — | 正常退出 |
| | asyncio.CancelledError | Agent 内 | 捕获 → TaskOutcome cancelled | 否 | round 继续 |
| **F. 数据** | checkpoint 损坏 | load 时 | JSONDecodeError → dict fallback | 否 | 提示用户 |
| | schema 反序列化失败 | load 时 | _migrate_checkpoint → raise | 否 | exit 1 |
| | disk full | save 时 | OSError → abort, 提示 | 否 | exit 1 |
| | SQLite busy | save/load | busy_timeout=5000ms → 超时 raise | 是(3) | exit 1 |

### 12.2 降级路径完整矩阵

| 场景 | 触发条件 | 降级动作 | 影响 |
|------|---------|---------|------|
| SafetyGate 工具缺失 | subprocess 找不到 safety scanner | skip, 提示 "safety scan unavailable" | 无 secret 检测 |
| CoverageGate | 永远 | 永远返回 skip (16G 约束) | 覆盖率由 CI 独立检查 |
| SemanticEvaluator 不可用 | 无 API key 或在 LLM agent 中 | 返回 None | 仅用停滞检测 (level 2) 替代 level 1 |
| Gate subprocess timeout | 单个 gate > 120s | Verdict.failed("timeout") | 该 gate 判定为失败 |
| checkpoint.save 失败 | disk full / SQLite IO error | abort, 提示用户释放空间 | 本轮进度丢失, 可从上一 checkpoint resume |
| post-edit hook 异常 | hook 脚本 crash | catch + exit 0 | 无 secret 检测 (不影响主流程) |
| pre-tool hook 超时 | >5s | 放行 (安全优先于速度) | 安全检查失效 |
| SemanticEvaluator LLM 异常 | API error / network | catch + return None | 降级到停滞检测 |

### 16.3 关键测试场景

|------|---------|

- **Plugin 首创**: Claude Code 原生集成 — 所有对标项目无此形态

## 20. 验收标准 (EARS 格式)

| ID | 验收标准 |
|----|---------|
| AC-01 | When 用户在 Claude Code 中输入 `/dev-loop "实现用户登录功能"`, the system shall 完成 architect→developer→critic 至少 1 轮并输出 verdict 和 gate_results |
| AC-02 | When 用户按 Ctrl-C 中断 dev-loop, the system shall 保存当前 checkpoint 并 exit 130 |
| AC-03 | When 用户输入 `/checkpoint resume <id>`, the system shall 从 checkpoint 恢复并继续未完成的 round |
| AC-04 | When post-edit hook 检测到代码变更, the system shall 在 30s 内完成 SafetyGate 扫描并输出结果 |
| AC-05 | When pre-tool hook 检测到危险 Bash 命令 (rm -rf /), the system shall 拒绝执行并 exit 2 |
| AC-06 | When developer 提交的代码有 P0 问题, the critic shall 判定 MAJOR 并给出具体修复建议 |
| AC-07 | When 连续 2 轮 diff_ratio < 5%, the ConvergenceJudge shall 触发 STAGNANT 并停止循环 |
| AC-08 | When lint Gate 失败, the Judge shall 触发 QUALITY_FAIL 并输出具体 lint 错误 |
| AC-09 | When ANTHROPIC_API_KEY 未设置, the system shall 在 CMD-01 预检阶段 exit 1 并提示设置 |
| AC-10 | When 用户输入 `/status`, the system shall 返回当前 thread_id, round, stage, verdict, 近 5 轮 history |
